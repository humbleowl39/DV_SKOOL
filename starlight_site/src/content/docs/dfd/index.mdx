## 왜 DFD를 배우는가

칩이 테이프아웃되어 실리콘으로 돌아오면, 그 안에서 무슨 일이 벌어지는지 들여다볼 방법이 필요합니다. RTL 시뮬레이션에서는 모든 신호를 파형으로 볼 수 있었지만, 실물 칩 안의 코어가 어느 주소에서 멈췄는지, 트레이스 버퍼에 무엇이 쌓였는지는 외부에서 직접 보이지 않습니다. DFD(Design For Debug)는 바로 이 "보이지 않는 칩 내부를 외부에서 관찰하고 제어할 수 있게" 만드는 설계 인프라입니다.

Arm 기반 SoC에서 이 인프라는 세 층으로 쌓여 있습니다. **JTAG**는 칩 밖에서 안으로 들어가는 물리적 통로이고, **ADI(Arm Debug Interface)가 정의하는 DAP**는 그 통로를 메모리-맵 기반 디버그 버스로 바꾸는 다리이며, **CoreSight**는 그 다리 너머에서 실제로 코어를 멈추고 트레이스를 뽑아내는 IP 생태계입니다. 이 스택을 모르면 디버거가 왜 칩에 붙지 못하는지, secure debug가 fuse 상태에 따라 왜 막히는지, 트레이스가 왜 off-chip으로 나오지 않는지 진단할 수 없습니다.

검증 엔지니어에게 DFD는 두 가지 의미가 있습니다. 첫째, 디버그 서브시스템 자체가 검증 대상입니다(DAP가 ROM table을 올바로 노출하는가, 인증 신호가 올바른 디버그 권한을 게이팅하는가). 둘째, post-silicon bring-up에서 칩이 죽었을 때 살아 있는 유일한 통로가 바로 이 JTAG/DAP 경로입니다. 이 코스는 그 통로를 신호 레벨부터 생태계 레벨까지 따라갑니다.

## 학습 목표

코스를 마치면 다음을 할 수 있습니다:

- **Explain (이해)** 외부 디버거 → JTAG/SWD 핀 → DAP(ADI) → CoreSight로 이어지는 4단계 디버그 스택의 각 층이 무엇을 책임지는지 설명
- **Trace (분석)** TAP controller 16-state FSM이 TMS 입력으로 IR/DR scan을 시퀀싱하는 과정을 단계별로 추적
- **Differentiate (분석)** JTAG-DP / SW-DP / SWJ-DP, MEM-AP / JTAG-AP, ADIv5 / ADIv6의 차이를 구분
- **Apply (적용)** SELECT → CSW → TAR → DRW 레지스터 시퀀스로 MEM-AP가 시스템 버스 트랜잭션을 일으키는 흐름을 구성
- **Evaluate (평가)** DBGEN/NIDEN/SPIDEN/SPNIDEN 인증 신호와 lifecycle/fuse 상태를 근거로 어떤 디버그가 허용/차단되는지 판단
- **Design (생성)** SoC 통합 시 ROM table 구성, 클럭/전원 도메인 격리, 트레이스 대역 계획을 설계

## 사전 지식

다음을 알고 있으면 본문이 매끄럽게 읽힙니다:

- **디지털 회로 기본**: 동기 FSM, 클럭/리셋 도메인, shift register
- **메모리-맵 IO**: 주소 디코딩, 레지스터 read/write 트랜잭션
- **버스 프로토콜 한 가지**: AHB / AXI / APB 트랜잭션의 기본 구조 ([AMBA 코스](../amba_protocols/) 참고)
- **SoC 구성 감각**: 멀티코어, 전원 게이팅, secure/non-secure 구분

부족한 부분이 있다면 [AMBA Protocols 코스](../amba_protocols/)를 먼저 보세요. DAP가 결국 CoreSight 레지스터를 APB/AHB/AXI 트랜잭션으로 두드리기 때문입니다.

## 학습 팁

:::tip[효율적 학습]
- **스택을 위에서 아래로 그려보기**: 디버거 → JTAG 핀 → DAP → CoreSight를 한 장에 그릴 수 있어야 함
- **신호와 레지스터를 분리해서 외우지 말 것**: JTAG의 TMS가 FSM을 움직이는 과정, DAP의 SELECT/CSW/TAR/DRW가 버스 트랜잭션을 만드는 과정을 _흐름_으로 이해
- **보안은 항상 같이**: 디버그 인프라는 secure debug 게이팅(인증 신호 + fuse)과 떼어 생각할 수 없음
- **AMBA와 연결**: MEM-AP가 내는 AHB/AXI/APB 트랜잭션이 곧 AMBA 코스의 그 트랜잭션
:::

:::caution[흔한 오해]
- **JTAG = boundary scan 전용**이라는 오해 — 현대 SoC에서는 DAP 접근 통로로 _재사용_됩니다.
- **디버그는 항상 가능**하다는 오해 — DBGEN/SPIDEN 등 인증 신호가 막으면 코어를 멈출 수도 없습니다.
- **디버그 로직은 코어 전원에 종속**된다는 오해 — bring-up을 위해 보통 always-on/독립 클럭으로 격리합니다.
:::
