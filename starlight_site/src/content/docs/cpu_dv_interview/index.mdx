CPU 코어를 검증하는 자리의 면접은 일반 DV 면접과 결이 다르다. 메모리 컨트롤러나 주변 IP는 "프로토콜대로 동작하는가"가 핵심이지만, CPU 코어는 **모든 소프트웨어가 의존하는 기계**여서 면접관은 "이 명령이 파이프라인 어디서 깨질 수 있는가", "out-of-order 실행이 왜 정확한 예외를 보장하는가", "weakly-ordered 메모리에서 왜 barrier가 필요한가" 같은 *마이크로아키텍처의 인과*를 묻는다. 외운 정의로는 두 번째 꼬리질문에서 막힌다.

이 코스는 ARM 기반 CPU 코어 검증 포지션(예: Google custom silicon, Apple/Qualcomm/Samsung SoC CPU 팀)을 목표로, **UVM/SystemVerilog 실무 경험을 CPU 코어 검증 문맥으로 재포지셔닝**하고 ARM 아키텍처·마이크로아키텍처·CPU 특유의 검증 방법론(ISS step-and-compare, 랜덤 명령 생성, formal)까지 한 줄 정의가 아니라 *왜 그렇게 하는가*로 답할 수 있게 설계되었다. 더해서 DV 면접의 변별 구간인 **즉석 작성 문제**(constraint·coverage·sequence)와 **코딩 테스트·영어 모의면접**까지 실전 코드와 스크립트로 다룬다.

## 🎯 학습 목표

이 코스를 마치면 다음을 할 수 있습니다:

- **Analyze (분석)** CPU DV 채용 공고의 요구 역량을 자신의 경력과 대조해 갭을 식별하고 재포지셔닝 전략을 세운다.
- **Explain (이해)** 파이프라인 해저드·out-of-order 실행·ROB·분기 예측·캐시·메모리 일관성을 *검증자 관점의 corner*와 함께 자기 언어로 설명한다.
- **Recall (암기)** ARM Exception Level·PSTATE·메모리 barrier(DMB/DSB/ISB)·MMU 변환의 핵심을 떠올린다.
- **Apply (적용)** ISS step-and-compare 기반 CPU 검증 환경을 계층(unit/core/subsystem)으로 설계해 설명한다.
- **Create (작성)** 주어진 요구에 맞는 SystemVerilog constraint·covergroup·virtual sequence를 즉석에서 작성한다.
- **Evaluate (평가)** 코딩 문제를 복잡도·edge case와 함께 풀고, 행동 면접 답변을 STAR로 구조화한다.

## 📋 사전 지식

본 코스는 **UVM/SystemVerilog 실무 경험 1년 이상**(또는 동등 학습)을 전제합니다.

- **UVM 기본**: component 계층, phase, sequence/sequencer/driver — 부족하면 [DV SKOOL — UVM](../uvm/)
- **컴퓨터 구조 기본**: 5단 파이프라인, 캐시 계층 — 부족하면 [DV SKOOL — Computer Architecture](../computer_architecture/)
- **선택**: ARM 아키텍처 — [DV SKOOL — ARM CPU (AArch64)](../cpu_arm/), CPU DV 방법론 — [DV SKOOL — CPU/Core Verification](../cpu_dv/)

```
[01 역할·전략·갭 분석]
```

- **01**: 무엇을 왜 공부하는지 — 면접 구조와 자신의 갭을 먼저 정렬.
- **02→05**: 마이크로아키텍처 corner를 알아야 검증 환경의 coverage·체크가 설계된다.
- **03·04**: 검증 대상이 ARM 코어 — ISA와 일관성은 자극·체크의 전제.

## 💡 학습 팁

:::tip[면접 답변 4단 구조]
1. **한 줄 정의** — "X는 ~다." (15초)
2. **인과** — "왜 그렇게 되는가" (그림/예시, 1분)
3. **트레이드오프** — "장점/단점, 언제 쓰고 언제 안 쓰는가" (30초)
4. **검증 corner** — "DV 관점에서 어디가 깨질 수 있는가" (CPU DV 면접의 가산점)
:::
:::caution[흔한 실수]
- 정의만 나열하고 *왜*를 못 말함 — 두 번째 꼬리질문에서 막힘.
- "OoO니까 예외도 순서 없이 처리된다" 같은 *틀린* 단순화 (ROB가 순서를 복원함).
- DMB/DSB/ISB를 "다 비슷한 barrier"로 뭉뚱그림.
- coverage 질문에 "테스트 수를 늘린다"고 답함 (정답은 방향성 자극).
:::
