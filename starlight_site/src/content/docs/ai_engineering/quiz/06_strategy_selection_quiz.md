## Q1. (Remember)

Prompt / RAG / Fine-tune 가 각각 무엇을 변경하는가?

<details>
<summary>정답 / 해설</summary>

세 전략의 차이는 "시스템의 어느 레이어를 변경하는가"로 명확히 구분된다.

**Prompt**는 모델도, 지식 저장소도 건드리지 않고 입력 텍스트만 바꾼다. 가장 빠르고 비용이 없지만, 모델이 이미 알고 있는 것 이상은 끌어낼 수 없다. **RAG**는 모델 자체는 그대로 두되, 요청마다 외부 지식을 동적으로 검색해 prompt에 주입한다. 모델이 모르는 도메인 지식을 실시간으로 제공할 수 있고, 지식 갱신은 인덱스 교체만으로 가능하다. **Fine-tune**은 모델의 가중치 자체를 도메인 데이터로 추가 학습시킨다. 특정 출력 형식, 스타일, 라벨 분포를 모델 내부에 내재화하는 데 적합하지만, 가장 비용이 높고 지식이 가중치에 고정되므로 갱신이 어렵다.

</details>
## Q2. (Understand)

같은 task 에 fine-tune 보다 RAG 가 유리한 두 시나리오는?

<details>
<summary>정답 / 해설</summary>

RAG가 fine-tune보다 유리한 두 시나리오는, 역으로 fine-tune의 구조적 약점이 드러나는 상황이기도 하다.

첫째, **자주 갱신되는 지식**이다. Fine-tune은 지식이 모델 가중치에 내재화되어 있어 지식이 바뀌면 재학습 전 과정을 반복해야 한다. 매주 바뀌는 정책이나 spec에 재학습 비용을 감당하는 것은 비현실적이다. RAG는 vector DB 인덱스만 교체하면 새로운 지식이 즉시 반영된다. 둘째, **출처 인용 의무**다. 법률, 규제, 안전 critical 도메인에서는 "이 답이 어디서 나왔는지"를 명확히 제시해야 한다. RAG는 검색된 문서를 답변과 함께 출처로 제시할 수 있지만, fine-tune 모델은 파라미터에 녹아든 지식이 어떤 학습 데이터에서 왔는지 역추적이 불가능하다.

</details>
## Q3. (Apply)

"사내 SystemVerilog spec 200 페이지 → 질문에 답하는 챗봇" 에 가장 적합한 조합은?

<details>
<summary>정답 / 해설</summary>

이 시나리오에서 세 전략을 비교하면 RAG가 유일한 현실적 선택이라는 결론이 나온다.

**Prompt만 사용**: 200페이지 spec 전체를 매 요청마다 prompt에 넣는 것은 context window 한계와 토큰 비용 때문에 불가능하다. 단독으로는 부적합하다. **Fine-tune**: 200페이지는 fine-tune에 필요한 데이터 규모(수천~수만 예시)에 훨씬 못 미친다. 게다가 spec이 갱신될 때마다 재학습해야 하는 운영 부담이 크다. 형식·스타일을 모델에 내재화하는 용도가 아니라면 이 케이스에는 맞지 않는다. **RAG**: spec을 chunk → embed → FAISS 인덱스 → top-k 검색 → prompt 주입하는 파이프라인이 가장 합리적이다. 답변 형식과 어조는 system prompt로 통제하고, 도메인 지식은 RAG가 제공한다. 결론적으로 **RAG + Prompt** 조합이 정답이며, fine-tune은 스타일 통일이 추가로 필요한 경우에만 보조적으로 더한다.

</details>
## Q4. (Analyze)

같은 task 의 4 조합 (Prompt / RAG / FT / FT+RAG) 비용 trade-off 를 분석하라.

<details>
<summary>정답 / 해설</summary>

네 조합은 비용 구조가 서로 달라서, task 요구사항과 조직 역량에 따라 최적 선택이 달라진다.

| 조합 | 초기 비용 | 운영 비용 | 갱신 비용 | 정확도 잠재력 |
|------|----------|----------|----------|--------------|
| Prompt | 매우 낮음 | 낮음 (토큰만) | 0 | 보통 |
| RAG | 중간 (인덱스 구축) | 중간 (검색+토큰) | 낮음 (re-index) | 높음 |
| FT | 높음 (학습+평가) | 낮음 (소형 모델 가능) | 매우 높음 (재학습) | 높음 (도메인 형식) |
| FT+RAG | 매우 높음 | 중간 | 중간 | 가장 높음 |

표에서 주목할 점은 Fine-tune의 **갱신 비용이 매우 높다**는 것이다. 초기 구축 비용과 운영 비용이 낮아 보여도, 지식이 자주 바뀌는 도메인에서는 갱신 사이클이 돌아올 때마다 전체 비용이 재발생한다. Prompt는 갱신 비용이 0이지만 정확도 한계가 명확하다. 실무 표준이 **Prompt → RAG → 필요 시 FT** 순서로 점진적 도입인 이유는, 앞 단계에서 실제 ROI를 확인하고 부족한 부분을 다음 단계로 채우는 것이 가장 낮은 리스크로 높은 효과를 내는 경로이기 때문이다.

</details>
## Q5. (Evaluate)

ROI 를 계산할 때 잊기 쉬운 비용 항목 3가지는?

<details>
<summary>정답 / 해설</summary>

ROI 계산에서 자주 누락되는 세 항목은 PoC와 실제 운영 사이의 가장 큰 함정이다.

첫째, **평가 셋 유지 비용**이다. gold set은 처음 만드는 데도 비용이 들지만, 더 중요한 것은 지속적인 갱신과 drift 모니터링이다. 도메인이 바뀌면 평가 기준도 바뀌어야 하고, 이를 방치하면 평가 점수가 높아도 실제 사용자 만족과 괴리가 생긴다. 둘째, **운영 신뢰성 비용**이다. LLM이 실패했을 때 fallback 로직, 요청·비용·실패를 추적하는 observability, 일시적 오류에 대한 retry 로직은 PoC에는 없지만 운영에는 필수다. 이를 나중에 추가하면 초기 설계를 뒤엎어야 하는 경우도 생긴다. 셋째, **인적 검증 비용**이다. LLM-as-judge로 자동 평가를 구축해도, 안전·법률·IP 등 critical 도메인에서는 인간 reviewer가 상시 개입해야 한다. 이 인건비가 ROI 계산에서 빠지면 현장 투입 후 예산 초과가 발생한다.

</details>
