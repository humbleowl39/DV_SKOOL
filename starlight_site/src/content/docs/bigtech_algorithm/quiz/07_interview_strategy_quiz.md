## Q1. (Remember)

45분 표준 타임라인 5단계는?

<details>
<summary>정답 / 해설</summary>

1. 이해 / Clarify (5분)
2. 예시 / Edge case (5분)
3. 접근 + 복잡도 합의 (5분)
4. 코딩 (20분)
5. 검증 / 리뷰 (10분)

각 단계가 존재하는 이유는 명확하다. Clarify 없이 코딩을 시작하면 다른 문제를 풀 수 있고, 예시 확인 없이는 edge case 를 코딩 중에 발견해 20분을 낭비한다. 복잡도 합의 없이 구현하면 면접관이 더 좋은 풀이를 기대했는데 코드를 다 짠 후에야 방향이 어긋났음을 알게 된다. 검증 단계 10분은 면접관에게 "자기 코드를 검증할 수 있다"는 신호를 보여주는 시간으로, 통과 여부에 직결된다.

</details>
## Q2. (Understand)

면접관이 평가하는 4축은?

<details>
<summary>정답 / 해설</summary>

1. **Correctness** — 정상/엣지 입력에 동작하는가.
2. **Complexity** — 시간/공간 복잡도가 합리적인가.
3. **Code Quality** — 변수명, 모듈화, 테스트 가능성.
4. **Communication** — 사고 흐름을 소리내어 공유, 면접관 신호에 반응.

네 가지를 균형 있게 보여줘야 통과.

한 가지만 뛰어나면 합격이 어려운 이유는 빅테크 면접이 업무 시뮬레이션이기 때문이다. 정확한 코드를 짜도 복잡도를 모르면 실제 업무에서 성능 문제를 낼 것이라는 신호를 준다. 완벽한 코드라도 침묵 속에 혼자 작성하면 팀원과 협업하기 어렵다는 우려를 낳는다. Communication 이 특히 중요한 이유는 면접관이 힌트를 줄 때 그것을 알아채고 방향을 전환할 수 있는지를 보는 항목이기도 하기 때문이다.

</details>
## Q3. (Apply)

문제를 받자마자 5분 동안 무엇을 해야 하는가?

<details>
<summary>정답 / 해설</summary>

1. 입력 형식 / 제약 (N 의 범위, 값 범위) 확인.
2. 출력 형식 + 1~2 개 예시 손으로 trace.
3. 엣지케이스 후보 적기 (빈 입력, 한 원소, 중복, overflow).
4. 후보 패턴 2~3개 + 목표 복잡도 면접관과 합의.

이 5분이 끝나야 코드 시작.

코딩을 시작하기 전 5분은 낭비가 아니라 투자다. N 의 범위를 모르면 O(N²) 이 통과 가능한지 알 수 없고, 예시를 직접 trace 하지 않으면 출력 형식을 잘못 이해하는 실수가 코딩 중반에야 발견된다. 엣지케이스를 미리 목록으로 적으면 코딩 후 검증 단계에서 체계적으로 확인할 수 있다. 패턴 합의가 가장 중요한 이유는 면접관이 "더 좋은 방법이 있을 텐데?"라고 생각하는 동안 잘못된 방향으로 20분을 쓰는 최악의 상황을 막기 때문이다.

</details>
## Q4. (Analyze)

자기 모의 면접 녹음을 4축으로 분석할 때 주의 깊게 볼 신호는?

<details>
<summary>정답 / 해설</summary>

- **Correctness** : 엣지 케이스를 직접 trace 했는가? 아니면 "테스트 통과하면 된다" 식으로 넘어갔는가?
- **Complexity** : 코딩 전에 복잡도를 명시했는가? 합의 없이 코딩 시작은 큰 감점.
- **Code Quality** : 변수명이 의미 있나? 큰 함수가 한 덩어리로 쓰여 있지 않나?
- **Communication** : 30초 이상 침묵이 있는가? 면접관 힌트에 무반응이 있는가?

자기 녹음 분석이 효과적인 이유는 본인이 면접 중 인지하지 못한 패턴을 외부 시각으로 발견할 수 있기 때문이다. Correctness 항목에서 "그냥 맞을 것 같다"는 말이 자주 나오면 추론 습관 훈련이 필요하다. Complexity 에서 코드 완성 후에야 복잡도를 언급한다면 설계 우선 사고가 부족한 것이다. Communication 에서 30초 침묵은 면접관이 개입하기 어색한 시간이므로 "지금 X 를 생각하고 있습니다"라는 한 마디로 채우는 연습이 필요하다.

</details>
## Q5. (Evaluate)

"다 풀어도 떨어지는 답안" 의 전형적 특징 3가지는?

<details>
<summary>정답 / 해설</summary>

1. **소통 부재** — 한 번에 코드 작성 후 "이렇게 풀었습니다" → 평가 어려움.
2. **복잡도 합의 생략** — 면접관이 더 좋은 풀이를 기대하고 있었는데 그 합의가 없음.
3. **엣지케이스 무시** — 정상 입력만 동작, NULL/빈/한 원소에서 실패.

반대로 **다 풀지 못해도 패스 가능** : (1) 명확한 접근 + 복잡도 합의, (2) 부분 코드 + 잘 정리된 pseudo code, (3) 시간 부족을 솔직히 인지하고 마무리.

"다 풀었는데 탈락"과 "다 못 풀었는데 합격"은 모두 실제로 일어난다. 전자의 핵심 원인은 면접관이 사고 과정을 평가할 수 없었던 것이다. 코드를 완성해도 왜 그렇게 접근했는지, 복잡도는 얼마인지, 다른 방법은 알고 있는지를 말하지 않으면 채점자 입장에서 점수를 줄 근거가 없다. 후자가 가능한 이유는 pseudo code + 복잡도 분석 + trade-off 설명만으로도 설계 능력과 커뮤니케이션 능력을 충분히 보여줄 수 있기 때문이다.

</details>
