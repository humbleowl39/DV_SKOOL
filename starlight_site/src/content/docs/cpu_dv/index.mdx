## 왜 CPU DV를 배우는가

프로세서는 검증 대상 중에서도 가장 잔인한 부류에 속합니다. AXI 슬레이브나 DMA 엔진은 "입력을 넣으면 정해진 출력이 나온다"는 비교적 좁은 계약을 가지지만, CPU 는 ISA 라는 거대한 계약 전체를 구현해야 합니다. 명령 하나하나가 architectural state — program counter, 범용 레지스터 파일, CSR(control/status register), privilege mode, 메모리 — 를 정해진 방식으로 바꿔야 하고, 그 상태 공간은 사실상 무한합니다. 게다가 현대 코어는 이 명령들을 파이프라인으로 겹쳐 실행하고, 분기를 추측 실행하며, 예외가 터지면 추측한 것을 모두 되돌려 정확한 상태로 복구해야 합니다. 이 모든 것을 "기능적으로 올바르게" 검증하는 것이 CPU DV 입니다.

이 지식이 없으면 다음 순간에 막힙니다. 단순히 명령 몇 개를 directed test(특정 시나리오를 노려 손으로 짠 결정적 테스트) 로 돌려 PASS 를 봤다고 코어가 맞다고 착각하고, 두 register write 의 순서가 어긋난 미묘한 hazard(앞 명령의 결과를 뒤 명령이 너무 일찍 쓰려 해 생기는 파이프라인 충돌) 버그를 scoreboard(DUT 출력과 정답 기대값을 비교하는 검증 컴포넌트) 가 못 잡으며, CSR side-effect 나 privilege transition 버그가 깊은 실행 경로에서만 드러나는데도 그 경로에 도달하는 자극을 만들지 못합니다. 이 코스는 프로세서 검증의 표준 방법론 — reference model 과의 step-and-compare lockstep, retirement 시점의 commit monitor, ISA coverage — 을 인과의 사슬로 엮어 설명합니다. 구체적인 러닝 예제는 오픈소스 도구가 풍부한 RISC-V 를 쓰지만, 여기서 다루는 방법론은 ARM, MIPS, x86 등 어떤 ISA 의 코어에도 그대로 적용됩니다.

## 학습 목표

코스를 마치면 다음을 할 수 있습니다:

- **Explain (이해)** 프로세서 검증이 왜 어려운지를 상태 공간·파이프라인·예외·메모리 오더링의 네 축으로 설명하고, simulation-based DV 의 5 단계를 구분
- **Analyze (분석)** reference model(ISS) 과 RTL 을 retirement 시점에 비교하는 step-and-compare lockstep 이 첫 divergent instruction 을 어떻게 조기에 flag 하는지 추적
- **Differentiate (분석)** RVFI(검증용 retire 신호 인터페이스) 와 RVVI(DV 서브시스템 통합 인터페이스) 가 각각 무엇을 노출하고 무엇을 묶는지 구분
- **Implement (적용)** retire/commit monitor → scoreboard → golden predictor(DPI-C) 로 이어지는 UVM 코어 검증 환경의 데이터 흐름을 구성
- **Apply (적용)** riscv-dv 같은 제약 랜덤 instruction generator 로 코너 케이스를 노리는 자극을 생성하는 전략을 적용
- **Evaluate (평가)** riscv-formal 의 형식 검증과 시뮬레이션 기반 검증을 보완 관계로 평가하고, ISA coverage 와 특수 영역(privilege, 예외, memory ordering)의 closure 전략을 판단

## 사전 지식

다음을 알고 있으면 본문이 매끄럽게 읽힙니다:

- **컴퓨터 구조 기본**: ISA, 5-stage 파이프라인과 hazard, OoO 실행과 retirement, 예외/precise interrupt — [Computer Architecture 코스](../computer_architecture/)의 M01–M03
- **UVM 방법론**: agent/monitor/scoreboard, TLM analysis port, coverage — [UVM 코스](../uvm/)의 M01–M05
- **형식 검증 감각**: assertion, bounded model checking 의 개념 — [Formal Verification 코스](../formal_verification/)
- **C/DPI-C**: SystemVerilog 와 C 모델을 잇는 인터페이스 (golden predictor 연동에 사용)

## 학습 팁

:::tip[효율적 학습]
- **"무한 상태 공간"을 항상 떠올려라**: 프로세서 DV 의 모든 기법은 "directed test 로는 가망이 없는" 상태 공간을 어떻게 효율적으로 훑느냐의 답이다 — 새 기법을 만나면 "이게 상태 공간의 어느 부분을 채우나"를 먼저 물어라.
- **reference model 이 채점관**: scoreboard 가 무엇을 expected 로 쓰는지 항상 의식하라. 직접 짠 기대값이 아니라 ISS(golden predictor) 가 그 역할을 한다.
- **retirement 시점을 손으로**: 명령이 fetch → execute → retire 로 흐를 때 architectural state 가 "언제" 바뀌는지(retire 시점) 타이밍을 직접 그려 본다.
- **RISC-V 예제 ↔ 일반 방법론 분리**: 본문의 Spike/RVFI/riscv-dv 는 도구 사례일 뿐이다. 방법론(reference-model lockstep, retire monitor, ISA coverage)은 ARM 등 다른 코어에도 동일하게 옮겨진다.
:::

:::caution[흔한 함정]
프로세서 검증에서 가장 자주 생기는 오개념은 **"directed test 가 다 PASS 면 코어가 맞다"**(상태 공간의 극히 일부만 본 것), **"execution 순서대로 비교하면 된다"**(architectural state 는 retirement 시점에 in-order 로 확정된다), **"형식 검증이 시뮬레이션을 대체한다"**(둘은 보완 관계)입니다. M01·M02·M06 에서 각각 바로잡습니다.
:::
