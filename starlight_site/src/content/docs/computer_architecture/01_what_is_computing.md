## 1. Why care? — 모든 것은 결국 0 과 1 위에 서 있다

검증이든 설계든, 우리가 다루는 RTL·신호·레지스터 값은 결국 **0 과 1 의 모음**입니다. 파형에서 `8'hA3` 같은 값을 보거나, 레지스터가 "왜 이 비트가 1이지?"를 따질 때, 그 바닥에는 "전기 스위치가 켜졌나 꺼졌나"라는 단 하나의 사실이 있습니다. 이 가장 밑바닥 — 비트가 무엇이고, 그것이 어떻게 모여 덧셈이 되고, 무엇이 그 동작을 박자 맞춰 진행시키는가 — 를 모르면, 이후 모듈의 파이프라인·캐시·명령이 _무엇 위에서 도는지_ 가 흐릿해집니다.

이 모듈은 딱 그 바닥을 다집니다. 어려운 수식은 없습니다. "컴퓨터는 엄청나게 많은 켜짐/꺼짐 스위치를, 시계 박자에 맞춰, 정해진 규칙으로 바꾸는 기계"라는 한 문장을 몸으로 이해하는 것이 목표입니다.

## 2. Intuition — 전등 스위치의 바다

:::tip[💡 한 줄 비유]
**컴퓨터** ≈ **엄청나게 많은 전등 스위치 + 1초에 수십억 번 울리는 메트로놈**.<br>
스위치 하나가 비트 하나(켜짐=1, 꺼짐=0). 스위치 여러 개를 묶으면 숫자가 되고, 스위치끼리 "둘 다 켜지면 켜라" 같은 규칙으로 연결하면 계산이 된다. 메트로놈(클럭)이 "딱!" 할 때마다 모든 스위치가 다음 상태로 _동시에_ 넘어간다.
:::

### 비트와 바이트 — 숫자를 켜짐/꺼짐으로 적기

**비트(bit)** 는 0 또는 1 한 자리입니다. 전기로는 "전압 높음/낮음", 물리적으로는 "스위치 켜짐/꺼짐"입니다. 비트 하나로는 2가지(0,1)밖에 못 적지만, 여러 개를 묶으면 큰 수도 적을 수 있습니다 — 이것이 **이진수(binary)** 입니다. 우리가 쓰는 십진수가 0~9 열 개 기호로 자릿수를 올리듯, 이진수는 0,1 두 기호로 자릿수를 올립니다.

```
이진수 1011  =  1×8 + 0×4 + 1×2 + 1×1  =  11 (십진수)
        │ │ │ │
        8 4 2 1   ← 각 자리의 무게(2의 거듭제곱)
```

**바이트(byte)** 는 비트 8개를 묶은 단위입니다(2⁸ = 256가지 표현). 메모리 용량(KB, MB, GB)이나 데이터 크기를 셀 때 기준이 됩니다. 검증에서 자주 보는 16진수(hex) `A3` 은 이진수 `1010 0011` 을 4비트씩 묶어 짧게 적은 것일 뿐입니다 — `A`=1010, `3`=0011.

### 한 장 그림 — 스위치에서 계산까지의 층위

```d2
direction: down

T: "**트랜지스터**\n전기로 켜고 끄는 아주 작은 스위치"
G: "**논리 게이트**\n스위치 몇 개로 만든 규칙\nAND / OR / NOT"
C: "**조합 논리**\n게이트를 엮어 덧셈·비교 등\n'계산'을 만든다"
R: "**레지스터**\n계산 결과를 잠시 붙잡아 두는\n비트들의 저장칸"

T -> G: "모으면"
G -> C: "엮으면"
C -> R: "결과를 담으면"
```

## 3. 작은 예 — 스위치 두 개로 "둘 다 켜짐"을 판단하기

가장 단순한 계산. "두 스위치가 _모두_ 켜졌을 때만 불을 켜라"는 규칙을 생각해 봅니다. 이것이 바로 **AND 게이트** 입니다.

| 입력 A | 입력 B | 출력 (A AND B) |
|---|---|---|
| 0 (꺼짐) | 0 | 0 |
| 0 | 1 | 0 |
| 1 | 0 | 0 |
| 1 (켜짐) | 1 (켜짐) | **1 (켜짐)** |

이런 규칙(게이트)이 몇 가지 더 있습니다 — **OR**(하나라도 켜지면 1), **NOT**(켜짐을 꺼짐으로 뒤집음). 놀라운 사실은, 이 단순한 게이트 몇 종류만 충분히 많이 엮으면 _덧셈·곱셈·비교·그리고 결국 모든 계산_ 을 만들 수 있다는 것입니다.

```d2
direction: right

A: "A = 1"
B: "B = 1"
AND: "**AND 게이트**\n둘 다 1일 때만 1"
OUT: "출력 = 1"

A -> AND
B -> AND
AND -> OUT
```

:::note[여기서 잡아야 할 두 가지]
**(1) 계산은 마법이 아니라 스위치 규칙의 누적이다.** 덧셈기도 결국 AND/OR/NOT 게이트를 정해진 모양으로 엮은 것이다.<br>
**(2) 게이트의 출력은 입력이 들어오면 _잠시 뒤_ 안정된다.** 전기가 게이트를 통과하는 데 시간이 걸리기 때문 — 이 "잠시"가 다음 절의 클럭이 필요한 이유다.
:::

## 4. 일반화 — 레지스터와 클럭, 그리고 "박자"의 필요

### 4.1 레지스터 — 결과를 붙잡아 두는 칸

게이트로 만든 계산 결과는 그대로 두면 입력이 바뀌는 순간 사라집니다. 그래서 결과를 _붙잡아 두는_ 저장칸이 필요한데, 그것이 **레지스터(register)** 입니다. 레지스터는 비트 몇 개(예: 32비트)를 담아 두고, "지금 값을 저장해!"라는 신호가 올 때만 새 값으로 갱신합니다. 그 신호의 타이밍을 정하는 것이 클럭입니다.

### 4.2 클럭과 사이클 — 모두가 같은 박자에 움직인다

**클럭(clock)** 은 1초에 수억~수십억 번 "딱, 딱, 딱" 규칙적으로 오르내리는 신호입니다. 한 번의 "딱"에서 다음 "딱"까지의 한 박자가 **사이클(cycle)** 입니다. 회로의 모든 레지스터는 _이 박자에 맞춰_ 동시에 값을 갱신합니다.

왜 굳이 박자가 필요할까요? 게이트를 통과하는 전기에는 _시간_ 이 걸려서, 계산 결과가 안정되기 _전_ 에 다음 단계가 값을 읽으면 엉뚱한 값을 읽습니다. 그래서 "한 박자 동안 모든 계산이 안정될 시간을 준 뒤, 박자가 올 때 일제히 결과를 레지스터에 저장"하는 규칙을 둡니다. 이것이 **동기(synchronous) 회로** 입니다.

```d2
direction: right

CLK: "**Clock**\n딱! 딱! 딱!\n(규칙적 박자)"
CALC: "**한 사이클 동안**\n게이트들이 계산하고\n결과가 안정됨"
REG: "**박자가 오면**\n레지스터가 일제히\n새 값 저장"

CLK -> CALC: "사이클 시작"
CALC -> REG: "사이클 끝(박자)"
REG -> CALC: "다음 사이클" { style.stroke-dash: 4 }
```

여기서 중요한 직관 하나: **클럭 주파수(frequency)** 가 높을수록(박자가 빠를수록) 1초에 더 많은 사이클을 돌지만, 한 사이클이 너무 짧으면 가장 느린 계산이 _안정될 시간_ 을 못 받아 틀립니다. 그래서 "한 사이클 안에 끝나야 하는 가장 긴 계산 경로"가 클럭 속도의 상한을 정합니다. 이 생각은 이후 [M04](../04_measuring_speed/)의 성능 이야기와 파이프라인([M06](../06_pipeline_hazard/))으로 곧장 이어집니다.

## 5. 흔한 오해

:::danger[❓ 오해 — '컴퓨터는 십진수로 계산하고 0/1 은 그냥 표현 방식이다']
**실제**: 하드웨어는 _물리적으로_ 0 과 1(전압 낮음/높음) 두 상태만 다룹니다. 십진수 12 도 내부에서는 이진수 `1100` 으로 저장·계산됩니다. 우리가 보는 십진수·16진수는 사람이 읽기 쉽게 _번역해 보여주는_ 것일 뿐, 회로 안에서 도는 것은 언제나 비트입니다.<br>
**왜 헷갈리는가**: 화면에 십진수가 보이니 내부도 그럴 것이라 짐작해서.
:::

## 6. 핵심 정리 (Key Takeaways)

- **비트 = 0/1 한 자리**(스위치 켜짐/꺼짐), **바이트 = 8비트**. 컴퓨터의 모든 값은 비트의 모음이다.
- **이진수**는 0,1 두 기호로 자릿수를 올려 숫자를 적는 방식이고, 16진수는 그것을 짧게 줄여 쓴 것.
- **트랜지스터(스위치) → 논리 게이트(규칙) → 조합 논리(계산) → 레지스터(저장)** 로 층이 쌓인다.
- **클럭**은 회로의 박자, **사이클**은 한 박자. 모든 레지스터가 박자에 맞춰 동시에 갱신(동기 회로).
- 한 사이클은 "가장 긴 계산이 안정될 시간"보다 길어야 한다 — 이것이 클럭 속도의 상한.

### 6.1 자가 점검

:::tip[🤔 Q1 — 이진수 (Bloom: Apply)]
이진수 `1101` 은 십진수로 얼마인가?
<details>
<summary>정답</summary>

각 자리의 무게는 오른쪽부터 1, 2, 4, 8 입니다. `1101` = 1×8 + 1×4 + 0×2 + 1×1 = 8 + 4 + 0 + 1 = **13**. 비트가 1인 자리의 무게만 더하면 됩니다.

</details>
:::
:::tip[🤔 Q2 — 클럭의 필요성 (Bloom: Understand)]
게이트로 계산이 다 되는데 왜 굳이 클럭(박자)이 필요한가?
<details>
<summary>정답</summary>

게이트를 통과하는 전기에는 _시간_ 이 걸려서, 계산 결과가 안정되기 전에 다음 단계가 값을 읽으면 엉뚱한(아직 변하는 중인) 값을 읽습니다. 클럭은 "한 박자 동안 모든 계산이 안정될 시간을 준 뒤, 박자가 올 때 일제히 결과를 저장"하게 만들어 이 문제를 막습니다. 그래서 모든 레지스터가 같은 박자에 동시에 갱신되는 동기 회로가 됩니다. 한 사이클이 너무 짧으면 가장 느린 계산이 안정될 시간을 못 받아 틀리므로, 가장 긴 계산 경로가 클럭 속도의 상한을 정합니다.

</details>
:::
