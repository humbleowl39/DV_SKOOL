## 1. Why care? — "빠르다"는 한 가지 뜻이 아니다

성능 이야기를 할 때 "이게 더 빠르다"는 말은 자주 혼란을 부릅니다. 자동차 한 대가 목적지에 _빨리 도착_ 하는 것과, 고속도로가 _시간당 많은 차_ 를 통과시키는 것은 다른 의미의 "빠름"입니다. CPU 도 마찬가지로, 한 작업을 빨리 끝내는 것(지연)과 단위 시간에 많은 작업을 처리하는 것(처리량)은 다릅니다. 이 둘을 섞으면 "주파수를 올렸는데 왜 안 빨라지지?", "코어를 늘렸는데 왜 그대로지?" 같은 질문에서 길을 잃습니다.

이 모듈은 성능을 재는 _기본 어휘_ 를 세웁니다. 어려운 수식은 [M13](../13_performance_laws_dsa/)으로 미루고, 여기서는 "무엇을 재고 있는가"를 분명히 하는 데 집중합니다.

## 2. Intuition — 한 대의 차 vs 고속도로

:::tip[💡 한 줄 비유]
**지연(latency)** ≈ **차 한 대가 출발해서 도착하는 데 걸리는 시간**.<br>
**처리량(throughput)** ≈ **그 고속도로가 1시간에 통과시키는 차의 수**.<br>
차선을 늘리면(병렬화) 한 대의 도착 시간(지연)은 그대로지만 시간당 통과량(처리량)은 늘 수 있다. 거꾸로 제한속도를 올리면(주파수) 한 대도 빨라진다. "빠르게"가 어느 쪽을 뜻하는지부터 정해야 한다.
:::

### 두 가지 "빠름"

- **지연(latency)**: 한 작업을 _시작해서 끝낼_ 때까지의 시간. "이 명령 하나가 끝나는 데 몇 ns?" — 짧을수록 좋다.
- **처리량(throughput)**: _단위 시간당 끝낸 작업 수_. "1초에 몇 개의 명령을 완료?" — 클수록 좋다.

둘은 독립적입니다. [M03](../03_life_of_an_instruction/)에서 본 명령의 네 단계를 _겹쳐서_ 처리하면(파이프라인), 명령 하나의 지연은 그대로여도 처리량은 크게 오릅니다 — 조립 라인이 제품 하나 만드는 시간은 안 줄여도 시간당 생산량은 늘리는 것과 같습니다.

```d2
direction: right

LAT: "**Latency**\n작업 하나의\n시작→끝 시간\n(짧을수록 좋음)"
THR: "**Throughput**\n단위 시간당\n끝낸 작업 수\n(많을수록 좋음)"
NOTE: "겹쳐 처리(파이프라인)하면\nlatency 그대로,\nthroughput ↑"

LAT -> NOTE
THR -> NOTE
```

## 3. 작은 예 — 사이클과 주파수로 시간 재기

[M01](../01_what_is_computing/)에서 클럭은 박자, 사이클은 한 박자라고 했습니다. **주파수(frequency)** 는 1초에 박자가 몇 번 울리는지입니다 — 3 GHz 면 1초에 30억 사이클. 한 사이클의 시간(cycle time)은 그 역수입니다.

```
주파수 3 GHz  →  1 사이클 = 1 / 3,000,000,000 초 ≈ 0.33 ns
```

이제 "어떤 작업이 1,500 사이클 걸린다"면, 실제 시간은:

```
시간 = 사이클 수 × 사이클당 시간 = 1500 × 0.33 ns = 500 ns
```

여기서 함정이 보입니다. 주파수를 2배로 올리면 사이클당 시간이 절반이 되어 빨라질 것 같지만, 만약 주파수를 올리느라 _사이클 수_ 가 늘어난다면(예: 한 사이클에 하던 일을 두 사이클로 쪼갬) 이득이 상쇄됩니다. 즉 **시간 = 사이클 수 × 사이클당 시간** 의 _두 항을 함께_ 봐야 합니다.

:::note[여기서 잡아야 할 두 가지]
**(1) 주파수만으로 빠름을 말할 수 없다.** 같은 일을 더 많은 사이클로 한다면 높은 주파수가 무색해진다.<br>
**(2) 사이클 수는 "명령이 몇 개인가 × 명령당 몇 사이클인가"로 다시 쪼개진다** — 다음 절의 CPI.
:::

## 4. 일반화 — 성능의 세 축 (Iron Law 미리보기)

### 4.1 CPI 와 IPC — 명령당 사이클

명령 하나가 평균 몇 사이클을 잡아먹는지를 **CPI(Cycles Per Instruction)** 라고 합니다. 그 역수, 즉 한 사이클에 평균 몇 명령을 끝내는지가 **IPC(Instructions Per Cycle)** 입니다. 이상적 파이프라인은 매 사이클 명령 하나를 완성하므로 CPI=1(IPC=1)에 가깝고, cache miss 나 분기 실패로 멈추면(stall) CPI 가 1보다 커집니다.

### 4.2 성능을 가르는 세 축

한 프로그램의 실행 시간은 세 가지가 곱해진 것으로 분해됩니다.

```
실행 시간 = (명령 수) × (명령당 사이클, CPI) × (사이클당 시간, 1/주파수)
            └ IC ┘      └─── CPI ───┘        └─── Cycle Time ───┘
```

```d2
direction: right

IC: "**명령 수 (IC)**\n알고리즘·컴파일러·ISA 가 결정"
CPI: "**CPI**\n파이프라인·캐시·분기예측\n(마이크로아키텍처)이 결정"
CT: "**사이클당 시간**\n주파수 = 회로·공정이 결정"
TIME: "**실행 시간**\n= IC × CPI × CycleTime"

IC -> TIME
CPI -> TIME
CT -> TIME
```

이 세 축이 강력한 이유는, _어떤 성능 기법이든 이 셋 중 하나를 건드린다_ 는 것입니다. 그리고 한 축을 개선하면 다른 축이 나빠지는 맞교환이 흔합니다 — 예를 들어 파이프라인을 깊게 해 주파수(사이클당 시간)를 높이면 분기 페널티로 CPI 가 늘 수 있습니다([M06](../06_pipeline_hazard/)). 그래서 새 기법을 만나면 항상 **"이건 IC·CPI·주파수 중 무엇을 개선하고, 무엇을 희생하나?"** 를 먼저 물어야 합니다. 이 사고 틀이 [M13](../13_performance_laws_dsa/)의 **Iron Law** 로, 그리고 병렬화의 한계를 다루는 Amdahl·Roofline 으로 정식화됩니다.

## 5. 흔한 오해

:::danger[❓ 오해 — '클럭 주파수(GHz)가 높으면 무조건 빠르다']
**실제**: 실행 시간 = 명령 수 × CPI × (1/주파수)입니다. 주파수는 세 항 중 _하나_ 일 뿐입니다. 주파수를 올려도 명령 수나 CPI 가 그만큼 늘면(예: 파이프라인을 깊게 해 분기 페널티 증가) 전체 시간은 안 줄거나 오히려 늘 수 있습니다. 서로 다른 설계의 GHz 만 비교하는 것은 의미가 적습니다.<br>
**왜 헷갈리는가**: 광고·스펙에서 GHz 가 가장 눈에 띄어, 성능의 단일 척도로 착각해서.
:::

## 6. 핵심 정리 (Key Takeaways)

- **지연(latency)** = 작업 하나의 시작→끝 시간, **처리량(throughput)** = 단위 시간당 완료 수. 둘은 독립적.
- 겹쳐 처리(파이프라인)하면 지연은 그대로여도 처리량이 오른다.
- **실행 시간 = 사이클 수 × 사이클당 시간** — 주파수만으로 빠름을 말할 수 없다.
- **CPI**(명령당 사이클), **IPC**(사이클당 명령) — 이상적이면 CPI≈1, stall 이 늘리면 CPI↑.
- **성능 3축 = 명령 수(IC) × CPI × 주파수**. 모든 기법은 이 셋 중 하나를 건드리며 맞교환이 따른다 — [M13](../13_performance_laws_dsa/)의 Iron Law.

### 6.1 자가 점검

:::tip[🤔 Q1 — latency vs throughput (Bloom: Analyze)]
파이프라인으로 명령들을 조립 라인처럼 겹쳐 처리했더니 1초당 완료 명령 수가 늘었다. 그런데 명령 하나가 시작해서 끝나는 시간은 그대로다. 무엇이 개선됐고 무엇은 그대로인가?
<details>
<summary>정답</summary>

개선된 것은 **처리량(throughput)** — 단위 시간당 완료하는 명령 수입니다. 그대로인 것은 한 명령의 **지연(latency)** — 그 명령이 시작해서 끝날 때까지의 시간입니다. 파이프라인은 여러 명령의 단계를 겹쳐 _매 사이클 하나씩 완성_ 되게 만들지만, 개별 명령은 여전히 모든 단계를 다 거쳐야 하므로 그 하나의 통과 시간은 줄지 않습니다. 조립 라인이 제품 하나의 제작 시간은 안 줄여도 시간당 생산량을 늘리는 것과 같은 원리입니다.

</details>
:::
:::tip[🤔 Q2 — 성능 3축 (Bloom: Evaluate)]
"이 CPU 는 4 GHz 라 3 GHz 짜리보다 무조건 빠르다"는 주장을 평가하라.
<details>
<summary>정답</summary>

불완전한 주장입니다. 실행 시간 = 명령 수(IC) × CPI × (1/주파수)이고, 주파수는 세 축 중 하나일 뿐입니다. 4 GHz CPU 가 같은 프로그램을 더 많은 명령 수로 처리하거나(다른 ISA·컴파일), CPI 가 더 크면(파이프라인이 깊어 분기 페널티가 크거나 캐시 적중률이 낮으면) 3 GHz CPU 보다 느릴 수 있습니다. 실제로 주파수를 올리려고 파이프라인을 깊게 하면 CPI 가 나빠지는 맞교환이 흔합니다. 따라서 주파수만으로 두 설계의 속도를 단정할 수 없고, 세 축을 함께 봐야 합니다(자세히는 [M13](../13_performance_laws_dsa/) Iron Law).

</details>
:::
