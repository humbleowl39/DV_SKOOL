## Q1. (Understand)

레지스터와 메모리의 차이로 옳지 _않은_ 것은?

- [ ] A. 레지스터는 메모리보다 훨씬 빠르다
- [ ] B. 레지스터는 개수가 적고(예: 32개), 메모리는 매우 많다(GB)
- [ ] C. 레지스터는 메모리 주소 공간의 0~31번지에 해당한다
- [ ] D. 계산은 레지스터에서 일어나고, 메모리는 주로 저장 용도다

<details>
<summary>정답 / 해설</summary>

**C** 가 틀린 설명입니다. 레지스터는 메모리와 _별개_ 의, CPU 안에 있는 빠르고 작은 저장소로, 메모리 주소 공간에 속하지 않습니다(이름/번호 R1, x5 등으로 직접 지정). A·B·D 는 모두 옳은 설명입니다. 둘은 위치(CPU 안 vs 밖)도 속도(수십~수백 배)도 다릅니다.

</details>
## Q2. (Apply)

메모리 300번지 값과 304번지 값을 더해 308번지에 저장하려 한다. 레지스터에서만 계산할 수 있을 때 필요한 명령을 순서대로 적어라.

<details>
<summary>정답 / 해설</summary>

1. `LOAD mem[300] → R1` (첫 값을 레지스터로)
2. `LOAD mem[304] → R2` (둘째 값을 레지스터로)
3. `ADD R1 + R2 → R3` (레지스터끼리 덧셈)
4. `STORE R3 → mem[308]` (결과를 메모리로)

덧셈은 메모리를 직접 다루지 못하므로 두 값을 _먼저 load_ 해 레지스터에 올리고, 결과를 _따로 store_ 해야 합니다. 이것이 load/store 아키텍처입니다.

</details>
## Q3. (Understand)

"계산은 레지스터에서만, 메모리는 load/store 로만"이라는 규칙(load/store 아키텍처)의 이점은?

<details>
<summary>정답 / 해설</summary>

명령이 단순해집니다. 산술 명령은 "레지스터 → 레지스터" 한 가지 형태만 처리하면 되고, 메모리 접근은 LOAD/STORE 명령만 책임지므로 각 명령의 동작과 길이가 규칙적입니다. 이 단순함과 규칙성 덕분에 명령을 _조립 라인처럼 겹쳐_ 빠르게 처리하는 파이프라인([M06](../../06_pipeline_hazard/))이 깔끔하게 가능해집니다. 또 메모리 왕복(load/store)은 느리므로, 자주 쓰는 값을 레지스터에 붙잡아 두는 최적화의 여지도 생깁니다.

</details>
