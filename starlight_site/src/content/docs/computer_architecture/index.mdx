## 왜 컴퓨터 구조를 배우는가

검증 엔지니어가 다루는 모든 RTL은 결국 von Neumann 병목과 싸우기 위해 태어난 구조 위에 얹혀 있습니다. 캐시가 왜 존재하는지, 파이프라인이 왜 stall 하는지, 메모리 컨트롤러가 왜 요청 순서를 바꾸는지를 모르면, 검증 중에 마주치는 "느린데 버그는 아닌" 현상과 "빠른데 실제로는 잘못된" 현상을 구분하지 못합니다. 컴퓨터 구조는 이 둘을 가르는 언어이며, DUT가 왜 그렇게 동작하는지를 설명해 주는 1차 원리입니다.

이 지식이 없으면 다음 순간에 막힙니다. DRAM 컨트롤러가 row hit 을 노려 요청을 재정렬한 결과를 scoreboard 가 mismatch 로 오인하고, OoO 코어의 retirement 순서와 execution 순서를 혼동해 잘못된 기대값을 만들며, 가속기 성능이 기대에 못 미칠 때 그것이 compute bound 인지 memory bound 인지 판단하지 못합니다. 이 코스는 Hennessy–Patterson 의 정전(正典)을 따라, ISA 라는 하드웨어/소프트웨어 계약부터 이종 SoC 의 도메인 특화 가속기까지를 인과의 사슬로 엮어 설명합니다.

## 학습 목표

코스를 마치면 다음을 할 수 있습니다:

- **Explain (이해)** ISA 가 왜 하드웨어/소프트웨어의 계약인지, RISC 가 CISC 대비 어떤 파이프라이닝 이점을 가지는지 설명
- **Trace (분석)** 5-stage 파이프라인에서 한 명령이 IF→WB 로 흐를 때 RAW/control/structural 해저드가 어디서 발생하고 forwarding 이 어떻게 해소하는지 추적
- **Differentiate (분석)** in-order stall 과 Tomasulo 기반 OoO 실행, 그리고 정적/동적 분기 예측기를 정확도·비용 기준으로 구분
- **Apply (적용)** 캐시 주소 분해(tag/index/offset)와 3C miss 분류, DRAM row hit/miss 비용을 실제 접근 패턴에 적용
- **Evaluate (평가)** Iron Law·Amdahl·Roofline 으로 성능 병목이 IC/CPI/주파수 중 어디인지, compute bound 인지 memory bound 인지 평가
- **Design (생성)** 이종 SoC 에서 워크로드별로 CPU·DSA·메모리 기술을 trade-off 기반으로 배치하는 구조를 설계

## 사전 지식

다음을 알고 있으면 본문이 매끄럽게 읽힙니다:

- **디지털 논리 기본**: 동기 회로, 레지스터, 클럭, 조합/순차 논리
- **이진수와 비트 연산**: 주소 비트 분해, 2의 보수, 시프트
- **C/어셈블리 감각**: load/store, 레지스터, 분기 — 명령이 무엇을 하는지에 대한 직관
- **기초 확률/비율**: hit rate, 평균 접근 시간(AMAT) 같은 가중 평균

부족한 부분이 있으면 *Computer Organization and Design* (Patterson & Hennessy) 1–4장을 먼저 보세요.

## 코스 운영 방식

각 모듈은 다음 순서로 구성됩니다:

1. **학습 목표** — 모듈 완료 후 할 수 있는 것 (Bloom's Taxonomy 동사)
2. **사전 지식** — 본 모듈에 필요한 선행 학습 항목
3. **본문** — 동기 시나리오 → 직관(비유 + 다이어그램) → 작은 예 → 일반화 → 디테일
4. **흔한 오해 & 디버그 체크리스트** — 검증 현장에서 마주치는 함정
5. **핵심 정리** — 압축된 takeaway + 자가 점검
6. **퀴즈** — 이해도 점검 (별도 페이지)

## 학습 팁

:::tip[효율적 학습]
- **Iron Law 를 나침반으로**: 모든 마이크로아키텍처 기법은 IC / CPI / 주파수 중 하나를 건드린다 — 새 기법을 만나면 "어느 축을 개선하고 어느 축을 희생하나"를 먼저 물어라.
- **파이프라인은 손으로**: IF/ID/EX/MEM/WB 타이밍 표를 직접 그리며 해저드가 어느 칸에서 충돌하는지 칠해 본다.
- **캐시 주소는 비트로**: tag/index/offset 분해는 외우지 말고 매번 비트 수를 손으로 계산.
- **DV 연결 고리 의식**: 각 개념이 검증에서 어떤 mismatch/timing 함정으로 나타나는지 본문의 디버그 체크리스트를 챙긴다.
:::

:::caution[흔한 함정]
컴퓨터 구조에서 가장 자주 생기는 오개념은 **"OoO 코어는 결과도 순서 없이 내보낸다"**(retirement 는 in-order), **"단일 변수 성능 100% = 시스템 빠름"**(Amdahl 의 직렬 천장), **"row hit/miss 재정렬은 버그"**(정상 최적화)입니다. M03·M05·M04 에서 각각 바로잡습니다.
:::
