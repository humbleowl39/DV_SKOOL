## 1. Why care? — 데이터가 "어디 있느냐"가 동작과 속도를 가른다

코어나 메모리 시스템을 검증할 때, "이 값이 지금 레지스터에 있는가, 메모리에 있는가"는 단순한 위치 문제가 아닙니다. 값이 어디 있느냐에 따라 _접근 속도_ (레지스터는 거의 즉시, 메모리는 수십~수백 배 느림)도, _그 값을 다루는 명령_ (레지스터끼리는 바로 연산, 메모리는 먼저 가져와야 연산)도 달라집니다. 이 구분을 모르면 "왜 이 명령은 메모리를 직접 더하지 않고 LOAD 부터 하지?", "왜 같은 변수가 어떤 때는 빠르고 어떤 때는 느리지?"를 설명하지 못합니다.

이 모듈은 메모리와 레지스터라는 _두 종류의 저장소_ 와, 그 사이를 오가는 규칙을 세웁니다. 이것이 이후 모든 명령·파이프라인·캐시 이야기의 무대입니다.

## 2. Intuition — 사물함의 줄과 책상 위 몇 칸

:::tip[💡 한 줄 비유]
**메모리** ≈ **번호가 붙은 거대한 사물함의 줄**. **레지스터** ≈ **지금 작업 중인 책상 위의 몇 칸**.<br>
사물함(메모리)은 수십억 칸이라 무엇이든 넣을 수 있지만 멀어서 꺼내 오는 데 시간이 걸린다. 책상(레지스터)은 몇 칸뿐이지만 손이 바로 닿아 즉시 쓴다. 그래서 "사물함에서 필요한 것을 책상으로 꺼내(load) → 책상에서 작업 → 결과를 사물함에 도로 넣는다(store)".
:::

### 주소 — 사물함마다 붙은 번호

메모리의 각 칸에는 **주소(address)** 라는 고유 번호가 붙어 있습니다(0번, 1번, 2번 …). "메모리 100번지의 값을 읽어라"는 곧 "100번 사물함을 열어 안의 내용을 꺼내라"입니다. 주소도 [M01](../01_what_is_computing/)에서 본 것처럼 결국 이진수 비트로 표현됩니다 — 32비트 주소면 2³² ≈ 43억 개의 칸을 가리킬 수 있습니다.

```d2
direction: right

MEM: "**Memory (사물함 줄)**" {
  a0: "addr 0: 0x00"
  a1: "addr 1: 0xFF"
  a2: "addr 2: 0x3C"
  a3: "addr 3: ..."
}
NOTE: "각 칸 = 주소(번호) + 그 안의 값"
MEM -> NOTE: ""
```

### 한 장 그림 — 두 저장소의 역할 분담

```d2
direction: right

REG: "**레지스터**\n책상 위 몇 칸(예: 32개)\n매우 빠름 (~1 cycle)\n여기서만 계산"
MEM: "**메모리**\n사물함 수십억 칸\n느림 (수십~수백 cycle)\n저장 용도"

REG -> MEM: "store (결과를 넣음)"
MEM -> REG: "load (필요한 값을 꺼냄)"
```

## 3. 작은 예 — 메모리의 두 값을 더해 도로 저장하기

"메모리 100번지와 104번지의 값을 더해 108번지에 저장하라"는 일을 생각해 봅니다. 레지스터에서만 계산할 수 있으므로, 순서는 이렇게 됩니다.

```
1. LOAD  100번지 → 레지스터 R1   // 사물함에서 책상으로
2. LOAD  104번지 → 레지스터 R2
3. ADD   R1 + R2 → 레지스터 R3   // 책상 위에서 계산
4. STORE R3 → 108번지            // 책상에서 사물함으로
```

```d2
direction: down

L1: "**LOAD** mem[100] → R1"
L2: "**LOAD** mem[104] → R2"
ADD: "**ADD** R1 + R2 → R3\n(레지스터끼리만 계산)"
ST: "**STORE** R3 → mem[108]"

L1 -> L2 -> ADD -> ST
```

여기서 핵심은 **3번 ADD 가 메모리를 직접 더하지 않는다** 는 점입니다. 더하려면 먼저 두 값을 레지스터로 가져와야(load) 하고, 결과도 일단 레지스터에 둔 뒤 따로 저장(store)합니다.

:::note[여기서 잡아야 할 두 가지]
**(1) 계산은 레지스터 사이에서만.** 메모리 값은 직접 연산 대상이 못 되고, 반드시 load 로 레지스터에 올린 뒤 쓴다.<br>
**(2) 메모리 왕복은 비싸다.** load/store 는 먼 사물함을 다녀오는 일이라 느리다 — 그래서 자주 쓰는 값을 레지스터(와 나중에 배울 캐시)에 붙잡아 두는 것이 성능의 핵심이 된다.
:::

## 4. 일반화 — 왜 이렇게 역할을 나눴는가

### 4.1 레지스터 vs 메모리 — 속도와 개수의 맞교환

| 구분 | 레지스터 | 메모리(DRAM) |
|---|---|---|
| 개수 | 적음 (예: 32개) | 매우 많음 (GB 단위) |
| 속도 | 매우 빠름 (~1 cycle) | 느림 (~100+ cycle) |
| 주소 | 이름/번호로 직접 지정 | 주소(번지)로 지정 |
| 용도 | 지금 계산 중인 값 | 프로그램·데이터 전체 저장 |

왜 둘로 나눌까요? "빠르고 큰" 저장소를 만들 수 없기 때문입니다(이유는 [M10 캐시](../10_why_cache/)에서 자세히). 빠른 저장소는 비싸고 작게만 만들 수 있어 _지금 쓰는 몇 개_ 만 레지스터로 두고, 나머지 방대한 데이터는 느리지만 큰 메모리에 둡니다. 이 맞교환이 [M01](../01_what_is_computing/)에서 본 "가까울수록 빠르고 작다"의 첫 사례입니다.

### 4.2 load/store 아키텍처 — 규칙을 단순하게

"계산은 레지스터에서만, 메모리는 load/store 로만 드나든다"는 규칙을 **load/store 아키텍처** 라고 합니다. 이렇게 정하면 명령이 단순해집니다 — 산술 명령은 "레지스터 → 레지스터" 한 가지 형태만 처리하면 되고, 메모리 접근은 LOAD/STORE 명령만 책임집니다. 이 단순함이 나중에 명령을 _조립 라인처럼 겹쳐_ 빠르게 처리하는 파이프라인([M06](../06_pipeline_hazard/))을 가능하게 합니다.

(모든 컴퓨터가 이런 것은 아닙니다. 메모리를 직접 연산하는 복잡한 명령을 가진 설계도 있지만, 현대 RISC 계열은 load/store 규칙을 따릅니다 — 그 이유와 RISC vs CISC 비교는 [M05 ISA](../05_isa_riscv/)에서 다룹니다.)

## 5. 흔한 오해

:::danger[❓ 오해 — '레지스터는 메모리의 일부다 (메모리 0~31번지 같은 것)']
**실제**: 레지스터는 메모리와 _별개_ 의, CPU 안에 있는 훨씬 빠르고 작은 저장소입니다. 메모리 주소 공간에 속하지 않고, 이름/번호(예: R1, x5)로 직접 지정합니다. 둘은 위치도(CPU 안 vs 밖) 속도도(수십~수백 배) 다릅니다.<br>
**왜 헷갈리는가**: 둘 다 "값을 저장한다"는 공통점만 보고 같은 것의 일부로 묶어서.
:::

## 6. 핵심 정리 (Key Takeaways)

- **메모리** = 번호(주소) 붙은 거대한 사물함의 줄. 크지만 느리다.
- **레지스터** = CPU 안의 빠르고 작은 저장칸(예: 32개). 계산은 여기서만 일어난다.
- **load** = 메모리 → 레지스터로 값을 꺼냄, **store** = 레지스터 → 메모리로 결과를 넣음.
- **load/store 아키텍처**: 산술은 레지스터끼리만, 메모리는 LOAD/STORE 로만 — 명령이 단순해져 파이프라이닝에 유리.
- 메모리 왕복은 비싸므로 자주 쓰는 값을 가까이(레지스터·캐시) 두는 것이 성능의 핵심.

### 6.1 자가 점검

:::tip[🤔 Q1 — load/store (Bloom: Apply)]
메모리 200번지와 204번지 값을 곱해 208번지에 저장하려 한다. 레지스터만 계산할 수 있을 때 필요한 명령들을 순서대로 적어라.
<details>
<summary>정답</summary>

1. `LOAD mem[200] → R1` (첫 값을 레지스터로)
2. `LOAD mem[204] → R2` (둘째 값을 레지스터로)
3. `MUL R1 × R2 → R3` (레지스터끼리 곱셈)
4. `STORE R3 → mem[208]` (결과를 메모리로)

핵심은 곱셈이 메모리를 직접 다루지 못하므로 두 값을 _먼저 load_ 하고, 결과를 _따로 store_ 해야 한다는 것입니다.

</details>
:::
:::tip[🤔 Q2 — 레지스터 vs 메모리 (Bloom: Understand)]
"빠르고 큰 저장소 하나로 통일하면 안 되나?"라는 질문에 답하라.
<details>
<summary>정답</summary>

물리적으로 "빠르면서 동시에 큰" 저장소는 만들기 어렵고 매우 비쌉니다. 빠른 저장소(레지스터·SRAM)는 작게만 경제적으로 만들 수 있고, 큰 저장소(DRAM)는 느립니다. 그래서 _지금 당장 쓰는 몇 개_ 의 값만 빠른 레지스터에 두고, 방대한 나머지는 느리지만 큰 메모리에 둡니다. 이 "속도 vs 용량"의 맞교환이 레지스터·캐시·메모리로 이어지는 계층 구조의 근본 이유이며, 자세히는 [M10 캐시](../10_why_cache/)에서 다룹니다.

</details>
:::
