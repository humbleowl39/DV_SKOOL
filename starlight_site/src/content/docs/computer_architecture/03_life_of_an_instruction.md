## 1. Why care? — 검증의 "정답"은 명령 하나하나의 동작에서 나온다

코어를 검증할 때 우리는 "이 명령이 실행되면 어떤 레지스터·메모리가 어떻게 바뀌어야 하는가"라는 기대값을 만듭니다. 그 기대값은 결국 _명령 한 줄이 무엇을 하는지_ 를 정확히 아는 데서 나옵니다. `ADD x3, x1, x2` 가 무엇을 더해 어디에 넣는지, 그 명령이 끝나면 다음에 어느 명령이 실행되는지를 모르면, DUT 출력이 맞는지 틀린지 판단할 수 없습니다.

이 모듈은 명령 한 줄을 _읽는 법_ 과, 그 명령이 CPU 안에서 거치는 _단계_ 를 다집니다. 어려운 것은 없습니다 — "명령을 가져와서(fetch), 무슨 일인지 해석하고(decode), 실행하고(execute), 결과를 저장한다(write-back)"는 네 박자입니다.

---

## 2. Intuition — 요리사가 레시피 카드를 한 장씩 처리하기

:::tip[💡 한 줄 비유]
**명령 실행** ≈ **요리사가 레시피 카드를 한 장씩 꺼내 따라 하기**.<br>
카드 더미(메모리의 프로그램)에서 다음 카드를 꺼내고(fetch), 카드에 뭐라 적혔는지 읽고(decode), 재료를 섞어 요리하고(execute), 완성품을 접시에 담는다(write-back). 그리고 _다음 카드_ 로 넘어간다. "지금 몇 번 카드 차례인가"를 가리키는 손가락이 PC(program counter)다.
:::

### 명령 한 줄 읽는 법 — "동사 + 목적지 + 재료"

처음 보는 어셈블리도 이 틀로 읽으면 쉽습니다.

```
ADD   x3,   x1,   x2
 │     │     └──┴── 재료(source): x1 과 x2 의 값
 │     └── 목적지(destination): 결과를 넣을 곳 x3
 └── 동사(연산): 더해라
```

즉 `ADD x3, x1, x2` = "`x1` 과 `x2` 를 더해 `x3` 에 넣어라". 마찬가지로 `LOAD x1, mem[100]` = "100번지 값을 꺼내 `x1` 에 넣어라", `STORE x3, mem[108]` = "`x3` 값을 108번지에 저장하라". (여기서 `x1`,`x2`,`x3` 은 [M02](../02_memory_and_registers/)에서 본 레지스터 이름입니다.)

### 한 장 그림 — 한 명령의 네 박자

```d2
direction: right

PC: "**PC**\n다음 명령 주소"
F: "**Fetch**\nPC 가 가리키는\n명령을 메모리에서 가져옴"
D: "**Decode**\n무슨 명령인지 해석\n+ 레지스터 값 읽기"
E: "**Execute**\nALU 로 실제 연산\n(예: 덧셈)"
W: "**Write-back**\n결과를 레지스터에 저장"

PC -> F -> D -> E -> W
W -> PC: "PC 증가 → 다음 명령" { style.stroke-dash: 4 }
```

## 3. 작은 예 — `ADD x3, x1, x2` 의 일생

`x1=5`, `x2=7` 인 상태에서 `ADD x3, x1, x2` 가 실행되는 과정을 따라가 봅니다.

| 단계 | 무슨 일이 일어나나 |
|---|---|
| **Fetch** | PC 가 가리키는 주소에서 명령 비트 `ADD x3,x1,x2` 를 메모리에서 가져온다 |
| **Decode** | "이건 ADD 명령, 목적지 x3, 재료 x1·x2" 라고 해석하고, 레지스터에서 x1(=5), x2(=7) 값을 읽는다 |
| **Execute** | ALU([M01](../01_what_is_computing/)의 게이트로 만든 덧셈 회로)가 5 + 7 = 12 를 계산한다 |
| **Write-back** | 결과 12 를 레지스터 x3 에 저장한다 |
| (그 후) | PC 가 다음 명령 주소로 넘어가 또 fetch 부터 시작 |

```d2
direction: right

F: "**Fetch**\nADD x3,x1,x2"
D: "**Decode**\nx1=5, x2=7 읽기"
E: "**Execute**\n5 + 7 = 12"
W: "**Write-back**\nx3 = 12"

F -> D -> E -> W
```

:::note[여기서 잡아야 할 두 가지]
**(1) 한 명령은 여러 단계를 거친다.** 가져오고(fetch), 해석하고(decode), 계산하고(execute), 저장한다(write-back) — 이 네 박자가 모든 명령의 공통 골격이다.<br>
**(2) PC 가 "다음 차례"를 정한다.** 보통은 하나씩 증가하지만, 분기(branch) 명령은 PC 를 다른 주소로 바꿔 실행 흐름을 점프시킨다.
:::

## 4. 일반화 — PC, 그리고 한 번에 하나 vs 겹쳐서

### 4.1 PC(program counter) — 실행 순서의 손가락

**PC(program counter)** 는 "다음에 실행할 명령의 주소"를 담은 특별한 레지스터입니다. 명령 하나가 끝날 때마다 PC 는 보통 _다음 명령_ 을 가리키도록 증가합니다(순차 실행). 그런데 if 문이나 반복문은 실행 흐름을 _다른 곳_ 으로 보내야 하는데, 이때 **분기(branch) 명령** 이 PC 를 원하는 주소로 바꿉니다. 즉 프로그램의 모든 흐름 제어는 "PC 를 어떻게 바꾸느냐"로 귀결됩니다.

```d2
direction: down

SEQ: "**순차 실행**\nPC += 1 → 다음 명령\n(대부분의 경우)"
BR: "**분기 (branch)**\n조건이 맞으면\nPC = 점프 목적지"

SEQ -> BR: "if / loop 를 만나면"
```

분기는 "다음에 어느 명령을 가져올지"를 바꾸기 때문에, 미리 명령을 당겨 처리하는 빠른 CPU 에서는 골치 아픈 문제가 됩니다 — 점프할지 말지 결정되기 _전_ 에 이미 다음 명령을 가져와 버릴 수 있으니까요. 이 문제(분기 해저드)와 그 해결(분기 예측)은 [M06](../06_pipeline_hazard/)·[M09](../09_branch_prediction/)에서 본격적으로 다룹니다.

### 4.2 한 번에 하나 vs 겹쳐서 — 파이프라인의 씨앗

지금까지는 "한 명령이 네 단계를 다 끝낸 뒤 다음 명령을 시작"하는 그림이었습니다. 그런데 잘 보면, 한 명령이 Execute 하는 동안 Fetch 회로는 _놀고_ 있습니다. 그렇다면 _다음 명령의 Fetch_ 를 미리 시작하면 어떨까요? 이렇게 여러 명령의 단계를 조립 라인처럼 _겹쳐서_ 처리하면 같은 시간에 더 많은 명령을 끝낼 수 있습니다. 이것이 **파이프라인** 의 핵심 아이디어이고, [M06](../06_pipeline_hazard/)의 주제입니다. 지금은 "명령 하나에 네 박자가 있고, 그 박자들을 겹칠 여지가 있다"까지만 기억하면 됩니다.

## 5. 흔한 오해

:::danger[❓ 오해 — '명령은 그냥 한 번에 실행된다 (단계가 따로 없다)']
**실제**: 명령 하나도 내부적으로 _여러 단계_ (가져오기·해석·실행·저장)를 거칩니다. 이 단계 구분이 중요한 이유는, 현대 CPU 가 이 단계들을 _여러 명령에 걸쳐 겹쳐_ 처리해 속도를 높이기 때문입니다(파이프라인). 단계가 없다고 보면 파이프라인·해저드 같은 이후 개념이 이해되지 않습니다.<br>
**왜 헷갈리는가**: 명령이 너무 빨리 끝나 보여서 내부 단계를 의식하지 못해서.
:::

## 6. 핵심 정리 (Key Takeaways)

- 명령은 **"동사(연산) + 목적지 + 재료"** 로 읽는다 — `ADD x3, x1, x2` = "x1+x2 를 x3 에".
- 명령 하나의 일생: **Fetch → Decode → Execute → Write-back** 네 박자.
- **PC(program counter)** 가 다음 실행할 명령의 주소를 가리키며, 보통 증가하고 분기 명령이 바꾼다.
- 모든 흐름 제어(if/loop)는 결국 "PC 를 어떻게 바꾸느냐"의 문제다.
- 단계들을 여러 명령에 걸쳐 _겹치면_ 속도가 오른다 — 파이프라인([M06](../06_pipeline_hazard/))의 씨앗.

### 6.1 자가 점검

:::tip[🤔 Q1 — 명령 읽기 (Bloom: Apply)]
`SUB x5, x6, x7` 은 무슨 일을 하는가? "동사+목적지+재료" 구조로 설명하라.
<details>
<summary>정답</summary>

동사 = SUB(빼라), 목적지 = `x5`(결과를 넣을 레지스터), 재료 = `x6` 과 `x7`. 즉 "`x6` 에서 `x7` 을 빼서(`x6 - x7`) 그 결과를 `x5` 에 넣어라"입니다. 산술 명령이므로 재료와 목적지는 모두 레지스터이고([M02](../02_memory_and_registers/)의 load/store 아키텍처), 이 명령도 fetch→decode(x6,x7 읽기)→execute(뺄셈)→write-back(x5 에 저장)의 네 박자를 거칩니다.

</details>
:::
:::tip[🤔 Q2 — PC (Bloom: Understand)]
PC 가 하는 일은 무엇이며, 분기 명령은 PC 와 어떻게 관련되는가?
<details>
<summary>정답</summary>

PC(program counter)는 "다음에 실행할 명령의 주소"를 담은 레지스터입니다. 명령 하나가 끝날 때마다 보통 다음 명령을 가리키도록 증가해 순차 실행을 만듭니다. 분기(branch) 명령은 이 PC 를 _다른 주소로 바꿔_ 실행 흐름을 점프시킵니다 — if 문이 조건에 따라 다른 코드로 가거나, 반복문이 처음으로 되돌아가는 것이 모두 PC 변경으로 구현됩니다. 즉 프로그램의 흐름 제어는 전부 "PC 를 어떻게 갱신하느냐"로 귀결되며, 이 때문에 분기는 명령을 미리 당겨오는 빠른 CPU 에서 까다로운 문제가 됩니다.

</details>
:::
