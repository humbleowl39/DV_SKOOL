## 왜 RAS를 배우는가

LLM 학습 클러스터나 HPC(High-Performance Computing, 고성능 컴퓨팅) 서버는 수천 개의 가속기와 분산 메모리(HBM, DDR5)를 동기화해 돌립니다. 이 규모에서는 단 하나의 SoC(System on Chip, 한 칩에 통합된 시스템) 안에서 로컬 SRAM(칩 내부 고속 메모리) 캐시나 메모리 인터페이스에 정정 불가능한 에러(uncorrectable error) 하나가 발생하는 것만으로도 전체 학습 잡이 무너지고, 체크포인트 복구에 막대한 compute 시간과 비용이 날아갑니다. 게다가 sub-3nm 공정과 2.5D/3D 패키징(CoWoS)으로 실리콘 밀도가 극단적으로 높아진 데다, LLM 워크로드가 가속기를 거의 100% 가동률로 밀어붙이면서 열 스트레스와 전압 droop이 노화(electromigration, thermal cycling)를 가속해 transient/permanent fault 빈도가 치솟습니다. RAS(Reliability, Availability, Serviceability)는 바로 이 문제를 silicon 레벨에서 막는 설계 철학이자 메커니즘 묶음입니다.

검증 엔지니어가 RAS를 모르면 정작 중요한 순간에 막힙니다. ECC가 1-bit 에러를 정정하고 2-bit 에러를 검출하는 경계를 시뮬레이션에서 어떻게 자극해야 하는지 모르고, poison bit가 버스를 타고 흐르다 소비 지점에서 exception을 일으키는 deferred error 모델을 검증 시나리오로 옮기지 못하며, RAS-node의 `ERR<n>STATUS` 레지스터에 에러가 정확히 기록되고 인터럽트가 올라가는지 확인하는 fault injection 테스트를 설계하지 못합니다. RAS를 배우는 것은 "에러가 없는 칩"이 아니라 "에러를 정직하게 검출·격리·보고하는 칩"을 어떻게 검증하는지를 익히는 과정입니다.

## 🎯 학습 목표

코스를 마치면 다음을 할 수 있습니다:

- **Explain (이해)** RAS의 세 기둥(Reliability/Availability/Serviceability)이 각각 어떤 SoC HW 메커니즘으로 구현되는지 설명
- **Differentiate (분석)** corrected error(CE) / uncorrectable error(UE) / deferred error(poison)의 처리 흐름과 시스템 영향을 구분
- **Trace (분석)** SEC-DED ECC가 1-bit를 정정하고 2-bit를 검출하는 경계, poison bit가 생성→전파→소비되는 경로를 추적
- **Implement (적용)** RAS-node의 error record 레지스터 모델과 fault injection을 시퀀스 레벨로 구현
- **Evaluate (평가)** 어떤 메모리/경로에 ECC를 쓰고 어디에 parity로 충분한지, 에러 주입을 RTL force가 아닌 시퀀스 레벨로 해야 하는 이유를 trade-off 기반으로 판단

## 📋 사전 지식

다음을 알고 있으면 본문이 매끄럽게 읽힙니다:

- **디지털 회로 기본**: 메모리(SRAM/DRAM), 버스 트랜잭션, 인터럽트
- **메모리-맵 IO**: 레지스터 offset, access policy(RO/RW/W1C)
- **UVM 검증 기본**: sequence, sequencer, scoreboard, RAL — [UVM 코스](../uvm/) 참고
- **AMBA 트랜잭션 개념**: poison/error response가 버스를 타는 맥락 — [AMBA 코스](../amba_protocols/) 참고

부족한 부분이 있다면 Arm® RAS System Architecture 사양을 먼저 훑어보세요.

## 💡 학습 팁

:::tip[효율적 학습]
- **세 기둥부터**: R/A/S 각각이 어떤 HW 메커니즘으로 구현되는지를 M01에서 확실히 매핑하고 넘어가세요.
- **에러 분류 손에 익히기**: CE / UE / deferred(poison)의 차이는 RAS 전체를 관통하는 축입니다. 흐름을 직접 그려보세요.
- **검증은 시퀀스 레벨로**: fault injection은 RTL을 건드리지 말고 inject 레지스터/시퀀스로 — M03에서 이유를 다룹니다.
:::

:::caution[흔한 오해 경계]
- **ECC = 모든 에러를 고친다(❌)**: SEC-DED는 1-bit만 정정, 2-bit는 검출만. 그 이상은 UE로 보고됩니다.
- **poison = 즉시 panic(❌)**: poison은 데이터를 태그해 전파시키고, 실제 소비 시점에 정밀 exception을 일으킵니다.
- **fault injection = RTL force(❌)**: 재사용 가능한 검증은 inject 레지스터/시퀀스 레벨로 합니다.
:::
