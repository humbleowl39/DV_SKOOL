## 1. Why care? — "오프로드"는 한 가지가 아니다

### 1.1 시나리오 — 같은 DPU, 다른 워크로드, 다른 우선순위

두 고객이 같은 DPU 제품을 검토합니다. 한쪽은 AI 클러스터 운영사로, GPU와 원격 스토리지 사이의 데이터 이동 지연을 줄이는 것이 최우선입니다. 다른 한쪽은 금융 서비스로, 테넌트 간 보안 격리와 암호화 처리가 최우선입니다. 같은 칩이지만 두 고객이 활성화하는 오프로드 기능은 전혀 다릅니다.

이 차이를 설명하려면 "오프로드"를 하나의 덩어리로 보면 안 됩니다. DPU가 호스트에서 가져오는 작업은 성격이 다른 네 도메인 — **네트워킹, 스토리지, 보안, 가상화** — 으로 나뉘고, 각 도메인이 서로 다른 가속기·소프트웨어를 필요로 합니다. 이 모듈은 그 네 도메인을 하나씩 풀어, "어떤 워크로드에 어떤 오프로드가 필요한가"를 판단할 수 있게 합니다.

이 분류를 모르면 다음 모듈(DV 연관)에서 RDMA·TOE·Ethernet이 _어느 도메인_ 의 어떤 기능을 구현하는지 연결하지 못합니다.

## 2. Intuition — "회사의 네 부서", 와 한 장 그림

:::tip[💡 한 줄 비유]
**오프로드 도메인** ≈ **인프라 비서가 운영하는 네 부서**.<br>
우편물 부서(Networking), 창고 부서(Storage), 보안 부서(Security), 손님 응대·격리 부서(Virtualization). 부서마다 전용 도구(가속기)와 직원(코어 위 소프트웨어)이 다릅니다. 회사(워크로드)마다 어느 부서가 바쁜지가 다릅니다.
:::
### 한 장 그림 — DPU가 호스트에서 가져오는 네 도메인

```d2
direction: down

HOST: "Host CPU\n(애플리케이션)"
DPU: "**DPU**" {
  NET: "**Networking**\n가상 스위칭·라우팅\n오버레이·트래픽 정책"
  STO: "**Storage**\nNVMe-oF·가상 스토리지\n디바이스·암호화"
  SEC: "**Security**\nIPsec/TLS·접근 제어\n텔레메트리"
  VIRT: "**Virtualization**\n관리/테넌트 분리\n베어메탈 서비스"
}
NETWORK: "Datacenter Network\n+ Remote Storage"

HOST -> DPU: "인프라 작업 위임 (PCIe)"
DPU -> NETWORK: "고속 인터페이스"
```

### 왜 이 디자인인가 — Design rationale

네 도메인으로 나누는 이유는 각 작업의 _성격_ 이 다르기 때문입니다. 네트워킹과 스토리지는 데이터가 끊임없이 흐르는 **데이터 패스** 작업이라 전용 가속기(패킷 엔진, DMA, 압축)가 유리하고, 보안은 데이터 패스에 얹힌 변환(암호화)이며, 가상화는 주로 _누가 무엇에 접근하는가_ 를 다루는 **제어 패스**·격리 작업입니다. DPU는 이 성격 차이에 맞춰 데이터 패스 처리 엔진과 프로그래머블 코어를 함께 두어, 도메인마다 적절한 위치에서 처리합니다.

## 3. 작은 예 — 한 NVMe-oF 읽기 요청이 Storage 도메인에서 처리되는 과정

가상 머신이 "원격 SSD에서 블록을 읽어 달라"고 요청할 때, Storage 오프로드가 이를 어떻게 처리하는지 단계별로 봅니다.

### 단계별 다이어그램

```d2
direction: down

VM: "VM\n로컬 NVMe처럼 보이는\n가상 스토리지 디바이스에\n블록 읽기 요청"
DPU_STO: "DPU Storage offload\n① 가상 디바이스로 요청 수신\n② NVMe-oF로 원격 타깃에 전송\n③ (필요시) 복호화\n④ 데이터를 VM 메모리로 DMA"
REMOTE: "Remote NVMe target\n(네트워크 너머 SSD)"

VM -> DPU_STO: "block read (로컬처럼 보임)"
DPU_STO -> REMOTE: "NVMe-oF (네트워크 경유)"
REMOTE -> DPU_STO: "데이터 반환"
DPU_STO -> VM: "DMA로 VM 메모리에 전달"
```

### 단계별 의미

| Step | 누가 | 무엇을 | 어느 관점 |
|---|---|---|---|
| ① | DPU 가상 스토리지 디바이스 | VM에게 로컬 NVMe처럼 보이는 인터페이스 제공 | 가상화 + 스토리지 |
| ② | DPU 데이터 패스 | NVMe-oF로 원격 타깃에 요청 전달 | 스토리지 + 네트워킹 |
| ③ | DPU 암호화 가속기 | (필요 시) 데이터 복호화 | 보안 |
| ④ | DPU DMA 엔진 | 데이터를 VM 메모리로 전달 (호스트 CPU 개입 최소) | 스토리지 + 가상화 |

:::note[여기서 잡아야 할 것]
하나의 요청이 _여러 도메인에 걸쳐_ 처리됩니다. 도메인은 깔끔히 분리된 칸막이가 아니라, 한 데이터 흐름이 통과하는 _기능 계층_ 에 가깝습니다. NVMe-oF 하나만 봐도 스토리지·네트워킹·보안·가상화가 모두 관여합니다.
:::

## 4. 일반화 — 네 도메인의 대표 기능

spec §5(주요 활용 사례)를 도메인 축으로 재정리하면 다음과 같습니다.

| 도메인 | 대표 오프로드 기능 | 호스트에서 가져오는 것 |
|---|---|---|
| **Networking** | 가상 스위칭, 라우팅, 오버레이(예: VXLAN), 트래픽 정책 | vSwitch·패킷 처리 CPU 부하 |
| **Storage** | NVMe-oF, 가상 스토리지 디바이스, 스토리지 암호화 | 원격 스토리지 I/O 스택 |
| **Security** | IPsec/TLS, 접근 제어, 텔레메트리 | 암호화·정책 적용 부하 |
| **Virtualization** | 관리 기능과 테넌트 워크로드 분리, 베어메탈 서비스 | 하이퍼바이저 관리/격리 |

이 네 도메인이 spec이 정의한 주요 활용 사례(네트워크 오프로드, 스토리지 오프로드, 보안 처리, 클라우드 인프라 격리, 베어메탈 서비스)를 모두 덮습니다. 특히 **베어메탈 서비스** 는 가상화 도메인의 극단적 형태로, 호스트에 하이퍼바이저를 두지 않고도 DPU가 관리형 네트워크·스토리지를 제공해 물리 서버를 클라우드처럼 운영하게 합니다.

### 4.1 데이터 패스 vs 제어 패스

```d2
direction: right

DP: "**Data Path** 오프로드" {
  d: "패킷 분류·전달, DMA\nNVMe-oF, 암호화\n→ 데이터가 흐르는 경로\n→ 전용 가속기 유리"
}
CP: "**Control Path** 오프로드" {
  c: "정책 결정, 관리,\n테넌트 격리\n→ '누가 무엇을' 결정\n→ 프로그래머블 코어"
}
```

오프로드를 평가할 때 중요한 구분은 데이터 패스인지 제어 패스인지입니다. 데이터 패스 오프로드는 흐르는 데이터를 처리하므로 전용 가속기로 가속하기 좋고, 제어 패스 오프로드는 정책·관리·격리처럼 _결정_ 을 다루므로 프로그래머블 코어 위의 소프트웨어가 담당합니다. spec이 "데이터 패스와 제어 패스 중 어느 범위를 프로그래밍할 수 있는가"를 제품 평가 항목으로 꼽는 이유가 여기 있습니다 — 제품마다 오프로드하는 범위가 다릅니다.

## 5. 디테일 — 도메인별로 더 들어가기

### 5.1 Networking — 가장 오래된 오프로드의 후예

네트워킹은 NIC 시절부터 오프로드의 출발점이었습니다. 초기 NIC의 체크섬·segmentation 같은 고정 오프로드가 SmartNIC의 프로그래머블 가상 스위칭·패킷 필터링으로 발전했고, DPU에서는 라우팅·오버레이·트래픽 정책까지 데이터 패스 엔진과 코어가 함께 처리합니다. 검증 관점에서 Ethernet·TOE(TCP Offload Engine) 같은 IP가 이 도메인의 핵심 구성요소입니다.

### 5.2 Storage — 로컬처럼 보이는 원격 스토리지

스토리지 오프로드의 핵심은 NVMe-oF와 가상 스토리지 디바이스입니다. DPU는 VM/호스트에게 로컬 NVMe처럼 보이는 디바이스를 제공하면서, 실제로는 네트워크 너머의 원격 SSD에 NVMe-oF로 접근합니다. 여기에 스토리지 암호화가 더해져, 데이터가 디스크/네트워크에 머무는 동안 보호됩니다. 이 도메인은 §3에서 본 데이터 패스 처리 엔진(DMA·큐 관리)과 전용 가속기(압축·암호화·스토리지)에 크게 의존합니다.

### 5.3 Security — 데이터 패스에 얹힌 변환과 정책

보안 오프로드는 IPsec/TLS 같은 암호화, 접근 제어, 텔레메트리를 포함합니다. 암호화는 흐르는 데이터를 변환하므로 전용 가속기로 처리하는 데이터 패스 작업에 가깝고, 접근 제어·텔레메트리는 정책·관측이므로 제어 패스 성격을 띱니다. 호스트와 분리된 위치에서 보안을 처리한다는 점이 격리(M01의 세 번째 동기)와 직접 맞닿습니다.

### 5.4 Virtualization — 격리의 핵심

가상화 오프로드는 관리 기능과 테넌트 워크로드를 분리하는 것이 핵심입니다. 전통적으로 호스트 하이퍼바이저가 하던 관리·격리 일부를 DPU가 가져가면, 테넌트가 호스트 CPU와 더 강하게 분리됩니다. 그 극단이 베어메탈 서비스로, 호스트에 하이퍼바이저 없이도 DPU가 관리형 네트워크·스토리지를 제공합니다. 클라우드 사업자가 네트워크·스토리지·가상화를 호스트에서 분리한 구조(M01 역사의 2010년대 후반, AWS Nitro·BlueField)가 바로 이 도메인의 대표 사례입니다.

### 5.5 워크로드별 우선순위

| 워크로드 | 우선 도메인 | 이유 |
|---|---|---|
| AI 클러스터 | Networking + Storage | GPU·스토리지 간 데이터 이동 지연 최소화 |
| 금융 서비스 | Security + Virtualization | 테넌트 격리, 암호화, 낮은 지연 |
| 게임 서비스 | Networking + Security | 낮은 지연, 트래픽 정책 |
| 클라우드 베어메탈 | Virtualization + Storage | 하이퍼바이저 없는 관리형 인프라 |

spec이 지적하듯 게임·금융·AI 클러스터는 요구사항이 다르고, 우선순위가 다른 만큼 적합한 DPU의 가속기 구성과 소프트웨어 스택도 달라집니다. 따라서 도메인 분류는 "이 워크로드에 어떤 오프로드가 필요한가"를 판단하는 기준이 됩니다.

## 6. 흔한 오해 와 DV 디버그 체크리스트

### 흔한 오해

:::danger[❓ 오해 1 — '오프로드는 다 같은 종류다']
**실제**: 네 도메인(네트워킹·스토리지·보안·가상화)은 성격이 다릅니다. 데이터 패스 작업(패킷·NVMe-oF·암호화)과 제어 패스 작업(정책·격리)은 처리 위치(가속기 vs 코어)도 다릅니다.<br>
**왜 헷갈리는가**: 모두 "CPU 부담을 던다"는 한 문장으로 묶여서 — 그러나 _무엇을 어디서_ 처리하는지는 도메인마다 다릅니다.
:::
:::danger[❓ 오해 2 — '도메인은 서로 독립적으로 동작한다']
**실제**: 한 요청이 여러 도메인을 가로지릅니다. NVMe-oF 읽기 하나가 스토리지·네트워킹·보안·가상화를 모두 거칠 수 있습니다(§3).<br>
**왜 헷갈리는가**: 표로 나누면 칸막이처럼 보여서 — 실제로는 한 데이터 흐름이 통과하는 기능 계층입니다.
:::
:::danger[❓ 오해 3 — '모든 DPU가 네 도메인을 전부 오프로드한다']
**실제**: 제품마다 오프로드 범위가 다릅니다. 데이터 패스만 프로그래머블한 제품도, 제어 패스까지 가능한 제품도 있습니다.<br>
**왜 헷갈리는가**: "DPU"라는 이름이 전 기능을 보장하는 것처럼 들려서 — 실제로는 spec의 평가 질문(어떤 기능을, 어디까지 프로그래밍 가능한가)으로 확인해야 합니다.
:::

### DV 디버그 체크리스트 (도메인 매핑 시 마주칠 질문들)

| 질문 | 1차 의심 | 어디 보나 |
|---|---|---|
| "내 IP는 어느 도메인인가" | 데이터 패스 vs 제어 패스, 기능 분류 | §4 도메인 표, §4.1 패스 구분 |
| "NVMe-oF 검증이 왜 네트워크도 건드리나" | 한 요청이 다중 도메인 통과 | §3 단계별 다이어그램 |
| "이 DPU가 IPsec을 지원하나" | 제품별 오프로드 범위 차이 | 제품 데이터시트 (spec은 범주만 제공) |
| "베어메탈은 어느 도메인" | 가상화의 극단(하이퍼바이저 없는 관리형) | §5.4 |

## 7. 핵심 정리 (Key Takeaways)

- **오프로드 = 네 도메인**: 네트워킹·스토리지·보안·가상화. 각 도메인이 서로 다른 가속기·소프트웨어를 요구한다.
- **대표 기능**: 가상 스위칭(N), NVMe-oF(S), IPsec/TLS(보안), 관리/테넌트 분리·베어메탈(가상화).
- **데이터 패스 vs 제어 패스**: 흐르는 데이터(가속기) vs 결정·격리(프로그래머블 코어) — 오프로드 범위 평가의 핵심 축.
- **도메인은 칸막이가 아니다**: 한 요청(예: NVMe-oF 읽기)이 여러 도메인을 가로지른다.
- **워크로드가 우선순위를 정한다**: AI/금융/게임/베어메탈마다 바쁜 도메인이 다르다 — 도메인 분류로 적합성을 판단한다.

:::caution[실무 주의점]
- 제품이 _어느 도메인의 어떤 기능_ 을, _데이터 패스/제어 패스 중 어디까지_ 오프로드하는지는 제품마다 다릅니다 — spec은 범주만 정의합니다.
:::
### 7.1 자가 점검

:::tip[🤔 Q1 — 도메인 분류 (Bloom: Classify)]
"VM에게 로컬 NVMe처럼 보이는 디바이스 제공"은 어느 도메인의 기능인가?
<details>
<summary>정답</summary>

주로 **Storage** 오프로드(가상 스토리지 디바이스)이며, 동시에 **Virtualization** 의 성격도 가집니다 — VM에게 디바이스를 _가상화_ 해 보여 주기 때문입니다. 실제 데이터는 NVMe-oF로 네트워크를 거치므로 Networking, 암호화가 붙으면 Security까지 관여합니다. 이는 도메인이 한 데이터 흐름을 가로지른다는 §3의 예시와 같습니다.

</details>
:::
:::tip[🤔 Q2 — 데이터/제어 패스 (Bloom: Differentiate)]
IPsec 암호화와 테넌트 격리는 각각 데이터 패스/제어 패스 중 어디에 가까운가?
<details>
<summary>정답</summary>

**IPsec 암호화** 는 흐르는 데이터를 변환하므로 _데이터 패스_ 에 가깝고 전용 가속기로 처리하기 좋습니다. **테넌트 격리** 는 "누가 무엇에 접근하는가"를 결정하는 _제어 패스_ 작업으로 프로그래머블 코어 위 소프트웨어가 담당합니다. 제품이 어느 패스까지 프로그래밍 가능한지는 평가 항목입니다.

</details>
:::
