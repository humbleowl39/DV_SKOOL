## 1. Why care? — CPU가 애플리케이션이 아니라 _인프라_ 를 돌리느라 바쁘다

### 1.1 시나리오 — AI 클러스터에서 사라진 CPU 코어

64코어 서버 한 대를 AI 학습 클러스터에 넣었습니다. 기대는 64코어 전부가 모델 연산을 돕는 것이었지만, 실측해 보니 상당수의 코어가 다른 일을 하고 있습니다. GPU 사이로 텐서를 옮기기 위한 네트워크 스택, 가상 스위치, 원격 스토리지에서 데이터셋을 끌어오는 NVMe-oF 처리, 테넌트 격리를 위한 암호화 — 이 인프라 작업들이 호스트 CPU 위에서 돌고 있기 때문입니다.

이것이 흔히 말하는 데이터센터 세금(datacenter tax)입니다. 서버가 늘어나고 데이터가 서버 사이를 더 자주 오갈수록, 애플리케이션을 위해 산 CPU가 정작 데이터 이동과 인프라 제어에 점점 더 많은 시간을 쓰게 됩니다.

:::note[메커니즘 — CPU 코어가 "사라지는" 구체 경로: interrupt + copy + context-switch]
"세금" 이 추상적 비유로 들리지만, 실제로는 패킷 한 개를 처리할 때마다 호스트 CPU 가 치르는 _세 가지 구체적 비용_ 이 누적된 것입니다.

1. **Per-packet interrupt** — 패킷이 도착할 때마다 NIC 이 CPU 에 인터럽트를 겁니다. CPU 는 하던 일을 멈추고 인터럽트 핸들러로 진입했다가 돌아와야 하는데, 이 진입/복귀 자체가 수백 ns 의 고정 비용입니다. 100 Gbps 에서는 패킷 도착 간격이 한 자릿수~수십 ns 라, 인터럽트 빈도가 CPU 가 감당하는 한계를 넘어섭니다.
2. **Data copy** — 네트워크 스택은 보통 NIC buffer → 커널 socket buffer → user buffer 로 데이터를 _복사_ 합니다. 이 memcpy 가 패킷 바이트 수에 비례해 CPU cycle 과 메모리 대역폭을 먹습니다 — 100 Gbps 양방향이면 초당 수십 GB 의 복사가 됩니다.
3. **Context switch** — 인터럽트 처리, 커널/유저 전환, vSwitch·암호화 같은 인프라 스레드 스케줄링이 끊임없이 일어나면, CPU 는 _작업을 바꾸느라_ cache 가 무효화되고 파이프라인이 비워집니다. 이 전환 한 번이 수백 ns~µs 이고, 빈번하면 유효 연산 시간이 잠식됩니다.

이 셋(interrupt 진입·복귀 + byte 비례 copy + 잦은 전환)이 매 패킷마다 쌓이면, "애플리케이션을 돌리라고 산" 코어들이 실제로는 _데이터를 옮기고 전환하는 데_ 시간 대부분을 쓰게 됩니다 — 이것이 코어가 "사라지는" 메커니즘입니다. DPU·TOE·RDMA 의 가치가 모두 _이 세 비용을 호스트 CPU 에서 제거_ 하는 데서 나온다는 점에서 동형입니다 (TOE 의 [Module 01](../../toe/01_tcp_ip_and_toe_concept/), RDMA 의 [Module 01](../../rdma/01_rdma_motivation/) 과 같은 동기).
:::

DPU는 이 세금을 별도의 프로세서로 옮겨 다음 세 가지를 노립니다. 첫째, **CPU 자원 확보** — 호스트 CPU를 애플리케이션 처리에 더 많이 할당합니다. 둘째, **성능 향상** — 패킷 처리와 암호화 같은 반복 작업을 전용 하드웨어로 가속합니다. 셋째, **격리 강화** — 인프라 서비스를 호스트 OS 및 테넌트 워크로드와 분리합니다.

이 모듈을 건너뛰면 이후 모듈에서 다루는 오프로드 도메인과 DV 연관이 "왜 굳이 별도 칩으로?"라는 질문에 막힙니다. DPU의 모든 설계 결정은 이 세 가지 동기에서 출발하기 때문입니다.

## 2. Intuition — "인프라 전담 비서", 와 한 장 그림

:::tip[💡 한 줄 비유]
**DPU** ≈ **사장(호스트 CPU)에게 붙은 인프라 전담 비서**.<br>
사장은 본업(애플리케이션)에 집중하고, 우편물 분류(네트워킹)·창고 관리(스토리지)·보안 검색(보안)·손님 응대 분리(가상화)는 비서가 전담합니다. 비서는 사장과 다른 사무실(격리)에서 일하므로, 사장이 바빠도 인프라는 멈추지 않습니다.
:::
### 한 장 그림 — 인프라 작업이 호스트 CPU에서 DPU로 이동

```d2
direction: right

HOST: "Host CPU\n(애플리케이션 / 테넌트 VM)" {
  style.fill: "#e8f0fe"
}
DPU: "**DPU**\n프로그래머블 코어 + 가속기" {
  style.fill: "#fef3e8"
  NET: "Networking\noffload"
  STO: "Storage\noffload"
  SEC: "Security\noffload"
  VIRT: "Virtualization\noffload"
}
NETWORK: "Datacenter\nNetwork (Ethernet)"

HOST -> DPU: "PCIe\n(host interface)"
DPU -> NETWORK: "고속 네트워크\n인터페이스"
HOST -> HOST: "본업: 애플리케이션 처리" { style.stroke-dash: 3 }
```

### 왜 이 디자인인가 — Design rationale

DPU의 구조는 세 동기(자원·성능·격리)를 동시에 만족시키려는 결과입니다. 인프라 작업을 _프로그래머블 코어_ 가 받아 호스트와 분리(격리)하고, 반복적이고 무거운 데이터 패스 작업은 _전용 가속기_ 가 받아 가속(성능)하며, 그 결과 호스트 CPU는 비워져(자원 확보) 애플리케이션에 더 쓰입니다. 즉 DPU는 "고속 NIC + 프로그래머블 연산 코어 + 메모리 + 데이터 패스 가속기"를 하나의 SoC 또는 PCIe 카드에 통합한 형태로 정의됩니다.

## 3. 작은 예 — 한 패킷 처리 작업이 CPU에서 DPU로 옮겨가는 과정

가상 스위칭(virtual switching)을 예로, 같은 작업이 전통 서버와 DPU 서버에서 각각 어디서 처리되는지 단계별로 비교합니다.

### 단계별 다이어그램

```d2
direction: down

TRAD: "전통 서버" {
  C1: "패킷 도착 (NIC)"
  C2: "호스트 CPU 인터럽트"
  C3: "커널/vSwitch 소프트웨어\n분류·전달·정책 (CPU 코어 소비)"
  C4: "VM 전달"
  C1 -> C2 -> C3 -> C4
}
DPUSRV: "DPU 서버" {
  D1: "패킷 도착 (DPU NIC)"
  D2: "DPU 데이터 패스 엔진\n분류·전달·정책 (가속)"
  D3: "VM 전달 (호스트 CPU 개입 최소)"
  D1 -> D2 -> D3
}
```

### 단계별 의미

| Step | 전통 서버 | DPU 서버 | 차이 |
|---|---|---|---|
| 패킷 분류 | 호스트 CPU의 vSwitch 소프트웨어 | DPU 데이터 패스 처리 엔진 | CPU 코어 소비 → 가속기로 이동 |
| 정책 적용 | 호스트 커널 | DPU 프로그래머블 코어 | 인프라 로직이 호스트와 분리(격리) |
| VM 전달 | 호스트 CPU DMA | DPU가 DMA·큐 관리 | 호스트 CPU 개입 최소화 |

### 구조 요약 (전형적인 DPU 구성)

DPU는 보통 다음 요소를 포함합니다. 아래는 spec이 정리한 전형적 구성으로, 어느 작업을 어디서 처리하는지를 보여 줍니다.

```
- 고속 네트워크 인터페이스 : Ethernet 또는 InfiniBand 포트
- 프로그래머블 연산 코어/메모리 : 인프라 제어 소프트웨어 실행
- 호스트 인터페이스 : PCIe를 통한 서버 연결
- 데이터 패스 처리 엔진 : 패킷 분류, 전달, DMA, 큐 관리
- 전용 가속기 : 암호화, 압축, 스토리지, RDMA 등 제품별 기능
```

:::note[여기서 잡아야 할 것]
DPU는 단순히 패킷을 더 빨리 옮기는 장치가 아닙니다. **분류·정책 같은 _제어 로직_ 까지** 호스트에서 가져와 자체 코어에서 돌린다는 점이 핵심이며, 이것이 SmartNIC와의 결정적 차이입니다.
:::

:::note[메커니즘 — 왜 "제어=코어, 데이터패스=가속기" 로 나누나]
DPU 안에 _프로그래머블 코어_ 와 _고정 가속기_ 가 둘 다 있는 이유는, 인프라 작업이 성격이 다른 두 부류로 갈리기 때문입니다.

- **제어 로직 (→ 프로그래머블 코어/SW)**: 연결 수립, 정책 결정, 예외 처리, 새 프로토콜 대응 같은 작업은 **드물게 일어나고, 분기(if/else)가 많고, 자주 바뀝니다.** 이런 일은 고정 회로로 굳히면 유연성을 잃고, 어차피 빈도가 낮아 가속 이득도 작습니다. 그래서 _코드로 표현되는_ 프로그래머블 코어가 맡는 것이 합리적입니다 — 새 정책이 생기면 펌웨어만 바꾸면 됩니다.
- **데이터 패스 변환 (→ 고정 datapath 가속기)**: checksum, 암호화, 압축, 패킷 분류·전달 같은 작업은 **모든 byte/패킷마다 빈번하게, 정해진 규칙대로 반복** 됩니다. 분기가 적고 형태가 정형적이라 _고정 파이프라인 회로_ 로 만들면 line-rate 로 흘려보내면서 cycle·전력을 최소화할 수 있습니다.

핵심 판단 기준은 **"빈도 × 정형성"** 입니다 — 자주 + 정형적이면 가속기로 굳히고, 드물게 + 분기 많으면 코어 SW 로 둡니다. 이 분담은 TOE 의 "data path = HW, control path = SW" 원칙과 정확히 같은 논리이며, DPU 는 그것을 네트워킹뿐 아니라 스토리지·보안 도메인 전체로 확장한 것입니다.
:::

## 4. 일반화 — 계산·저장·통신 관점과 진화의 흐름

### 4.1 계산 / 저장 / 통신 관점

컴퓨터 시스템은 크게 계산(compute), 저장(storage), 통신(networking)으로 나누어 볼 수 있습니다. CPU와 GPU가 계산을 수행하고, 메모리와 SSD가 데이터를 저장하며, NIC와 스위치 같은 네트워크 장치가 서버와 장치 사이의 통신을 담당합니다.

CPU 성능만 높여서는 전체 시스템 성능이 같은 비율로 향상되지 않습니다. 데이터가 많아지고 서버와 가속기의 수가 늘어날수록 저장 장치와 네트워크를 관리하는 작업도 함께 증가하기 때문입니다. 특히 AI와 클라우드 워크로드에서는 여러 서버·GPU·스토리지를 연결하는 데이터 이동 경로가 병목이 됩니다. DPU는 바로 이 통신 장치가 고속화·프로그래머블해지는 과정에서 등장했으며, NIC 대역폭을 높이는 데 그치지 않고 CPU가 담당하던 데이터 이동과 인프라 제어 작업 일부를 가져와 가속합니다.

### 4.2 GPU와의 평행 관계

```d2
direction: right

CPU: "Host CPU\n(애플리케이션)"
GPU: "**GPU**\n그래픽·병렬 계산 분리"
DPU: "**DPU**\n네트워크·스토리지·보안·가상화 분리"

CPU -> GPU: "계산 워크로드 offload"
CPU -> DPU: "인프라 워크로드 offload"
```

GPU는 그래픽과 병렬 계산을 CPU에서 분리해 전문적으로 처리합니다. DPU도 비슷한 방식으로 네트워크·스토리지·보안·가상화 같은 인프라 작업을 CPU에서 분리합니다. 두 장치의 워크로드와 설계 목적은 다르지만, **CPU가 애플리케이션 처리에 집중하도록 만든다는 점은 같습니다.** 이 평행 관계가 "왜 또 별도의 칩인가"에 대한 가장 직관적인 답입니다 — GPU가 계산에서 해냈던 일을 DPU가 인프라에서 한다고 보면 됩니다.

:::note[메커니즘 — 격리(isolation)가 하드웨어로 보장되는 방식]
세 동기 중 _격리_ 가 "호스트와 분리" 라는 말로만 들리지만, 실제로는 몇 겹의 하드웨어 경계로 강제됩니다 (구체 구현 깊이는 제품마다 다름 — 확인 필요).

- **별도 OS/연산 도메인**: DPU 의 인프라 제어 SW 는 호스트 CPU 가 아니라 _DPU 자체 코어 위의 별도 OS_ 에서 돕니다. 그래서 테넌트 VM 이 호스트에서 무슨 짓을 하든, 인프라 제어 코드는 _다른 프로세서·다른 OS_ 에 있어 직접 건드릴 수 없습니다 — 침해 시 영향 범위가 분리됩니다.
- **별도 메모리 도메인**: DPU 는 자기 메모리를 갖고, 인프라 상태(연결 테이블·키·정책)를 _호스트가 주소로 접근할 수 없는_ 자기 메모리 공간에 둡니다. 제어 평면 데이터가 테넌트 메모리 공간 밖에 사는 셈입니다.
- **PCIe 경계 + IOMMU/주소 변환**: 호스트와 DPU 는 PCIe 로만 연결되고, 그 사이의 DMA 접근은 **IOMMU(또는 device-side 주소 변환)** 가 통제합니다. IOMMU 는 "어떤 장치/도메인이 어떤 물리주소에 접근 가능한가" 를 매핑·강제하므로, 테넌트가 인프라 제어 경로의 메모리에 임의로 DMA 하는 것을 _주소 변환 단계에서_ 차단합니다. 즉 격리는 "약속" 이 아니라 _주소 도메인 분리_ 라는 하드웨어 메커니즘으로 보장됩니다.

이 세 경계 덕분에 "테넌트 워크로드 ↔ 인프라 제어" 가 _같은 칩 안에서도_ 서로 닿지 못하며, 이것이 multi-tenant 클라우드에서 DPU 가 신뢰 경계(trust boundary)로 쓰이는 이유입니다. (검증 대상으로서의 격리는 [Module 03](../03_dpu_in_datacenter_and_dv/) 에서.)
:::

### 4.3 NIC / SmartNIC / DPU 비교

| 구분 | 주요 역할 | 특징 |
|---|---|---|
| **NIC** | 서버의 네트워크 연결 | 체크섬, segmentation 등 제한된 고정 기능 오프로드 제공 |
| **SmartNIC** | 프로그래머블 네트워크 처리 | 패킷 처리 로직을 프로그래밍 가능, 네트워크·보안 일부 오프로드 |
| **DPU** | 데이터센터 인프라 처리 | 프로그래머블 코어·메모리·가속기 포함, 네트워크 외 스토리지·보안까지 독립 처리 |

DPU는 네트워크 처리 중심의 SmartNIC보다 넓은 범위의 인프라 작업을 담당합니다. 제품을 비교할 때는 명칭이 아니라 지원 워크로드·프로그래머빌리티·가속기 구성·호스트 격리 방식을 함께 봐야 합니다.

## 5. 디테일 — 역사, 효율, 그리고 명칭의 혼란

### 5.1 진화의 역사

DPU는 NIC가 점차 더 많은 인프라 작업을 맡게 된 흐름에서 등장했습니다.

| 시기 | 변화 |
|---|---|
| 2000년대 | NIC가 체크섬, segmentation 같은 고정 기능 오프로드를 제공하기 시작 |
| 2010년대 | 네트워크 프로세서·FPGA 기반 SmartNIC가 가상 스위칭, 패킷 필터링 등 프로그래머블 처리 담당 |
| 2010년대 후반 | 클라우드 사업자가 네트워크·스토리지·가상화 기능을 호스트에서 분리하는 구조 도입 (AWS Nitro, Mellanox BlueField가 대표 사례) |
| 2020년 전후 | DPU라는 용어가 널리 사용되며 독립 인프라 프로세서 범주가 알려짐 |
| 2020년대 | 여러 벤더가 DPU 또는 유사 인프라 프로세서를 제공 |

핵심은 DPU가 어느 날 갑자기 발명된 것이 아니라, **고정 오프로드 → 프로그래머블 오프로드 → 독립 인프라 프로세서** 라는 연속적 진화의 결과라는 점입니다.

### 5.2 데이터센터 효율과 에너지

기존 서버에서는 호스트 CPU가 애플리케이션과 함께 네트워크·SSD·가속기를 관리하는 소프트웨어도 실행합니다. DPU를 적용하면 인프라 기능을 별도 프로세서와 가속기로 옮길 수 있습니다. 그 결과 호스트 CPU는 애플리케이션에 더 많은 자원을 쓸 수 있고, 서버 한 대에서 처리하는 유효 워크로드를 늘릴 수 있습니다. 데이터센터 전체로는 같은 워크로드에 필요한 서버 수를 줄일 가능성이 생기므로 장비 비용·전력·냉각·공간 절감에 기여합니다.

다만 그 효과는 워크로드, 네트워크 구조, 오프로드 가능한 기능, 소프트웨어 통합 수준에 따라 달라집니다. 정량적 수치는 제품 데이터시트와 벤치마크로 별도 확인해야 합니다 (spec 명시).

### 5.3 명칭의 혼란 — DPU / IPU / SmartNIC

DPU는 하나의 고정된 하드웨어 규격이 아닙니다. 벤더에 따라 DPU, IPU(Infrastructure Processing Unit) 등 서로 다른 이름을 쓰며, 구현 방식과 제공 기능은 제품 목적에 따라 달라집니다. 따라서 두 제품을 비교할 때 이름은 거의 의미가 없고, 실제로 어떤 기능을 오프로드하는지, 데이터 패스와 제어 패스 중 어디까지 프로그래밍 가능한지를 따져야 합니다.

## 6. 흔한 오해 와 DV 디버그 체크리스트

### 흔한 오해

:::danger[❓ 오해 1 — 'DPU는 그냥 더 빠른 NIC다']
**실제**: DPU는 대역폭만 높인 NIC가 아닙니다. 패킷 분류·정책 같은 _제어 로직_ 과 스토리지·보안 작업까지 호스트에서 가져와 _자체 프로그래머블 코어_ 에서 처리하는 독립 프로세서입니다.<br>
**왜 헷갈리는가**: 외형이 PCIe 네트워크 카드라 NIC처럼 보이기 때문 — 그러나 내부에 코어·메모리·가속기가 있다는 점이 다릅니다.
:::
:::danger[❓ 오해 2 — 'DPU / IPU / SmartNIC는 표준이라 기능이 같다']
**실제**: 표준화된 단일 규격이 아닙니다. 명칭은 벤더마다 다르고 오프로드 범위도 제품마다 다릅니다.<br>
**왜 헷갈리는가**: 같은 범주처럼 불려서 — 비교는 명칭이 아니라 오프로드 기능·프로그래머빌리티로 해야 합니다.
:::
:::danger[❓ 오해 3 — 'DPU만 넣으면 무조건 CPU가 비고 전력이 준다']
**실제**: 효과는 워크로드·네트워크 구조·오프로드 가능 기능·소프트웨어 통합 수준에 따라 달라집니다. 정량 효과는 데이터시트/벤치마크로 확인해야 합니다.<br>
**왜 헷갈리는가**: "오프로드 = 항상 이득"이라는 단순화 때문 — 통합되지 않은 기능은 오프로드되지 않습니다.
:::

## 7. 핵심 정리 (Key Takeaways)

- **DPU 등장 동기는 세 가지**: CPU 자원 확보, 성능 향상(전용 가속), 격리 강화. 모든 설계 결정이 여기서 나온다.
- **데이터센터 tax**: 서버·데이터가 늘수록 인프라 처리가 CPU를 잠식 — DPU가 이를 별도 프로세서로 옮긴다.
- **NIC → SmartNIC → DPU**: 고정 오프로드 → 프로그래머블 오프로드 → 독립 인프라 프로세서로 이어진 연속적 진화.
- **GPU와의 평행**: GPU가 계산을 분리했듯 DPU는 인프라를 분리한다 — 목적은 둘 다 CPU를 애플리케이션에 집중시키는 것.
- **명칭은 비교 기준이 아니다**: DPU/IPU/SmartNIC는 벤더마다 다르며, 오프로드 범위·프로그래머빌리티로 비교해야 한다.

:::caution[실무 주의점]
- DPU 효과의 _정량값_ 은 spec이 보장하지 않습니다 — 데이터시트/벤치마크로 확인하세요.
- "프로그래머블"의 범위(데이터 패스만 / 제어 패스까지)는 제품마다 다릅니다.
:::
### 7.1 자가 점검

:::tip[🤔 Q1 — DPU vs SmartNIC (Bloom: Differentiate)]
같은 패킷 분류 작업을 SmartNIC와 DPU가 한다면, 무엇이 다른가?
<details>
<summary>정답</summary>

SmartNIC는 _네트워크 처리 중심_ 으로 패킷 처리 로직을 프로그래밍하고 네트워크·보안 일부를 오프로드합니다. DPU는 같은 네트워크 처리를 하면서도 프로그래머블 코어·메모리·가속기를 갖춰 _스토리지·보안 작업까지 독립적으로_ 처리하고, 인프라 서비스를 호스트와 분리(격리)합니다. 즉 차이는 "오프로드 _범위_"와 "독립 처리/격리 능력"에 있습니다.

</details>
:::
:::tip[🤔 Q2 — 세 동기 (Bloom: Explain)]
DPU가 인프라를 호스트에서 분리하면 얻는 세 효과를 설명하라.
<details>
<summary>정답</summary>

(1) **CPU 자원 확보** — 호스트 CPU를 애플리케이션에 더 할당. (2) **성능 향상** — 패킷 처리·암호화 같은 반복 작업을 전용 하드웨어로 가속. (3) **격리 강화** — 인프라 서비스를 호스트 OS·테넌트 워크로드와 분리. 데이터센터 전체로는 서버 수·전력·냉각·공간 절감으로 이어질 수 있으나, 정량 효과는 워크로드와 통합 수준에 따라 다릅니다.

</details>
:::
