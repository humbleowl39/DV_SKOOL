## Q1. (Remember)

DPU의 네 오프로드 도메인이 아닌 것은?

- [ ] A. Networking
- [ ] B. Storage
- [ ] C. Security
- [ ] D. Rendering (그래픽 렌더링)

<details>
<summary>정답 / 해설</summary>

**D**. DPU 오프로드 도메인은 네트워킹·스토리지·보안·가상화 네 가지입니다(spec §5를 도메인 축으로 정리). 그래픽 렌더링(D)은 GPU의 영역이며 DPU 인프라 오프로드에 속하지 않습니다.

</details>

## Q2. (Understand)

NVMe-oF는 어느 도메인의 대표 기능이며, 무엇을 호스트에서 가져오는가?

<details>
<summary>정답 / 해설</summary>

**Storage** 도메인의 대표 기능으로, 원격 스토리지 I/O 스택을 호스트에서 DPU로 가져옵니다. DPU는 VM/호스트에게 로컬 NVMe처럼 보이는 가상 스토리지 디바이스를 제공하면서 실제로는 NVMe-oF로 네트워크 너머 원격 SSD에 접근합니다(본문 §3, §5.2). 따라서 호스트 CPU는 원격 스토리지 처리 부담을 덜게 됩니다.

</details>

## Q3. (Apply)

"테넌트 격리 정책 결정"과 "IPsec 패킷 암호화"를 데이터 패스/제어 패스로 각각 분류하라.

- [ ] A. 둘 다 데이터 패스
- [ ] B. 둘 다 제어 패스
- [ ] C. 격리 정책 = 제어 패스, IPsec 암호화 = 데이터 패스
- [ ] D. 격리 정책 = 데이터 패스, IPsec 암호화 = 제어 패스

<details>
<summary>정답 / 해설</summary>

**C**. 격리 정책 결정은 "누가 무엇에 접근하는가"를 다루는 _제어 패스_ 작업으로 프로그래머블 코어 소프트웨어가 담당하고, IPsec 암호화는 흐르는 데이터를 변환하는 _데이터 패스_ 작업으로 전용 가속기로 처리하기 좋습니다(본문 §4.1, §5.3). 데이터 패스 = 흐르는 데이터, 제어 패스 = 결정/관리/격리로 구분합니다.

</details>

## Q4. (Apply)

AI 클러스터 운영사가 DPU를 도입할 때 가장 활성화할 가능성이 높은 도메인 조합은?

- [ ] A. Networking + Storage
- [ ] B. Security + Virtualization
- [ ] C. Rendering + Compute
- [ ] D. 아무 도메인도 필요 없음

<details>
<summary>정답 / 해설</summary>

**A**. AI 클러스터는 GPU와 원격 스토리지 사이의 데이터 이동 지연 최소화가 최우선이므로 **Networking + Storage** 도메인을 주로 활성화합니다(본문 §5.5). 금융은 Security+Virtualization(B)에 더 비중을 둡니다. 워크로드마다 바쁜 도메인이 다르다는 것이 도메인 분류의 핵심 효용입니다.

</details>

## Q5. (Analyze)

"오프로드 도메인은 서로 독립적인 칸막이"라는 인식이 왜 틀렸는지, NVMe-oF 읽기 예로 분석하라.

<details>
<summary>정답 / 해설</summary>

한 NVMe-oF 읽기 요청 하나가 여러 도메인을 가로지릅니다(본문 §3): VM에게 로컬 디바이스로 보이는 인터페이스(가상화+스토리지) → NVMe-oF로 원격 타깃에 전송(스토리지+네트워킹) → 필요 시 복호화(보안) → VM 메모리로 DMA(스토리지+가상화). 도메인은 분리된 칸막이가 아니라 _한 데이터 흐름이 통과하는 기능 계층_ 이므로, 한 기능을 검증하려면 여러 도메인을 함께 봐야 합니다.

</details>

## Q6. (Evaluate)

"모든 DPU 제품은 네 도메인을 전부 오프로드한다"는 가정을 검증 계획에 쓰면 어떤 위험이 있는지 평가하라.

<details>
<summary>정답 / 해설</summary>

위험합니다. 제품마다 오프로드 범위와 프로그래머빌리티가 다르며, 데이터 패스만 프로그래머블한 제품도 있습니다(spec §6, 본문 §5). 모든 도메인을 가정하면 (1) 실제로 없는 기능을 검증하려다 범위가 부풀고, (2) 반대로 제품이 실제 지원하는 기능 중 일부를 놓칠 수 있습니다. 따라서 검증 범위는 spec의 제품 평가 질문(어떤 기능을, 어디까지 프로그래밍 가능, 어떻게 격리)으로 실제 제품 기능에 맞춰 스코핑해야 합니다.

</details>
