## 1. Why care? — 우리가 검증하는 IP는 _DPU 안에서_ 동작한다

### 1.1 시나리오 — "이 RDMA 엔진은 왜 격리·호스트 인터페이스 요구사항이 붙지?"

RDMA 엔진 IP를 검증하는데, 검증 요구사항에 단순한 RDMA 프로토콜 준수만이 아니라 "호스트와의 격리", "PCIe 호스트 인터페이스 동작", "다른 가속기와의 데이터 패스 공유" 같은 항목이 함께 들어옵니다. 프로토콜만 보던 입장에서는 이 요구사항들이 어디서 나왔는지 의아합니다.

답은 그 IP가 _DPU라는 상위 시스템의 한 블록_ 으로 통합되기 때문입니다. DPU는 고속 네트워크 인터페이스, 프로그래머블 코어, 호스트 인터페이스(PCIe), 데이터 패스 처리 엔진, 그리고 RDMA·암호화·압축 같은 전용 가속기를 하나로 묶습니다(M01 §3). RDMA 엔진은 그중 "전용 가속기" 하나이고, 따라서 자신만의 프로토콜뿐 아니라 _DPU 시스템 계약_ (호스트 인터페이스, 격리, 데이터 패스 연결)도 지켜야 합니다.

이 모듈은 개별 IP를 묶는 상위 지도를 그려, 검증 요구사항이 어디서 오는지 이해하게 합니다. 이 그림이 없으면 IP를 "고립된 프로토콜 블록"으로만 보다가, 시스템 통합에서 등장하는 요구사항에 매번 놀라게 됩니다.

## 2. Intuition — "오케스트라", 와 한 장 그림

:::tip[💡 한 줄 비유]
**DPU** ≈ **오케스트라**, 개별 IP(RDMA·TOE·Ethernet) ≈ **연주자**.<br>
각 연주자는 자기 악기(프로토콜)를 잘 다뤄야 하지만, 동시에 지휘자(DPU 데이터 패스/제어)의 박자와 무대 규칙(호스트 인터페이스·격리)도 따라야 합니다. 솔로 연습(IP 단독 검증)만으로는 합주(시스템 통합)에서 드러나는 문제를 잡지 못합니다.
:::
### 한 장 그림 — 개별 IP가 DPU 데이터 패스로 묶이는 구조

```d2
direction: right

HOST: "Host CPU / VM" {
  style.fill: "#e8f0fe"
}
DPU: "**DPU SoC**" {
  style.fill: "#fef3e8"
  HIF: "Host Interface\n(PCIe)"
  CORE: "프로그래머블 코어\n(제어·정책)"
  DP: "**Data Path Engine**\n분류·전달·DMA·큐" {
    style.fill: "#eaf7ea"
  }
  ETH: "Ethernet IP"
  TOE: "TOE\n(TCP offload)"
  RDMA: "RDMA 엔진"
  CRYPTO: "암호화/압축\n가속기"
}
NET: "Datacenter Network\n+ Remote Storage"

HOST -> HIF: "PCIe"
HIF -> DP
CORE -> DP: "정책/제어"
DP -> ETH
DP -> TOE
DP -> RDMA
DP -> CRYPTO
ETH -> NET
RDMA -> NET
```

### 왜 이 디자인인가 — Design rationale

개별 가속기들이 _공통 데이터 패스 엔진_ 을 통해 호스트 인터페이스와 네트워크에 연결되는 이유는, 데이터가 한 IP에서 끝나지 않고 여러 블록을 거치기 때문입니다. 예컨대 원격 메모리 접근은 RDMA가 의미를 처리하고, 그 아래 전송은 TOE/Ethernet이 담당하며, 보안이 필요하면 암호화 가속기를 통과합니다. 이 흐름을 매끄럽게 잇기 위해 DPU는 중앙의 데이터 패스 처리 엔진(분류·전달·DMA·큐 관리)을 두고, 프로그래머블 코어가 그 위에서 정책을 결정합니다. 검증이 IP 단독을 넘어 "블록 간 연결과 시스템 계약"을 봐야 하는 근거가 여기 있습니다.

## 3. 작은 예 — 원격 메모리 읽기 한 번이 여러 IP를 통과하는 과정

호스트가 "원격 서버의 메모리에서 데이터를 가져와라"라고 요청할 때, DPU 안에서 IP들이 어떻게 협력하는지 추적합니다. (각 IP의 내부 동작은 단순화한 _개념 흐름_ 입니다 — 추론.)

### 단계별 다이어그램

```d2
direction: down

H: "Host: PCIe로 원격 read 요청\n(가상 주소/큐)"
DP1: "DPU Data Path:\n요청 분류 → RDMA 엔진으로 라우팅"
R: "RDMA 엔진:\n원격 메모리 read 의미 처리\n(memory semantic)"
T: "TOE / Ethernet:\n전송 계층·프레이밍 처리\n네트워크로 송수신"
DP2: "DPU Data Path:\n수신 데이터 → (필요시) 복호화\n→ DMA로 호스트 메모리에 전달"
H2: "Host: 데이터 수신\n(CPU 개입 최소)"

H -> DP1 -> R -> T
T -> DP2 -> H2
```

### 단계별 의미

| Step | 어느 IP / 블록 | 무엇을 | M02 도메인 |
|---|---|---|---|
| ① 요청 수신 | Host Interface (PCIe) | 호스트의 원격 read 요청 수신 | 가상화/네트워킹 |
| ② 라우팅 | Data Path Engine | 요청을 RDMA 엔진으로 분류·전달 | 네트워킹 |
| ③ 의미 처리 | RDMA 엔진 | 원격 메모리 접근 의미 처리 | 네트워킹(전용 가속기) |
| ④ 전송 | TOE / Ethernet | 전송 계층·프레이밍, 네트워크 송수신 | 네트워킹 |
| ⑤ 변환·전달 | 암호화 가속기 + DMA | (필요 시) 복호화 후 호스트 메모리로 DMA | 보안 + 스토리지/가상화 |

:::note[여기서 잡아야 할 것]
한 요청이 **여러 IP를 직렬로 통과** 합니다. 따라서 검증은 (1) 각 IP의 단독 프로토콜 준수, (2) IP 간 인터페이스(데이터 패스 연결), (3) 호스트 인터페이스·격리라는 시스템 계약 — 세 층위를 모두 봐야 합니다. RDMA 코스/TOE 코스가 (1)을, 이 DPU 코스가 (2)(3)의 그림을 제공합니다.
:::

## 4. 일반화 — DPU 시스템 계약과 검증 층위

### 4.1 IP 단독을 넘어서는 시스템 계약

개별 IP를 DPU에 통합할 때 추가로 지켜야 하는 _시스템 계약_ 은 spec의 전형적 구조(M01 §3)와 활용 사례에서 도출됩니다.

| 계약 항목 | 무엇을 보장해야 하나 | 출처 근거 |
|---|---|---|
| Host Interface | PCIe를 통한 호스트 연결이 규약대로 동작 | spec §3 (호스트 인터페이스) |
| Data Path 연결 | 분류·전달·DMA·큐가 IP 간 데이터를 올바르게 잇는다 | spec §3 (데이터 패스 처리 엔진) |
| 격리 (Isolation) | 인프라 서비스가 호스트 OS·테넌트와 분리 | spec §1 (격리 강화), §5 (인프라 격리) |
| 가속기 협력 | RDMA·암호화·압축 등이 데이터 패스 위에서 협력 | spec §3 (전용 가속기) |

### 4.2 검증의 세 층위

```d2
direction: down

L1: "Layer 1 — IP 단독 (Protocol)\nRDMA / TOE / Ethernet 각자\n프로토콜 준수 검증" {
  style.fill: "#eaf7ea"
}
L2: "Layer 2 — IP 간 통합 (Data Path)\n데이터 패스 엔진을 통한\n블록 간 연결·전달 검증"
L3: "Layer 3 — 시스템 계약\n호스트 인터페이스·격리·\n다중 가속기 협력 검증" {
  style.fill: "#fdeaea"
}
L1 -> L2 -> L3: "통합 범위 확대"
```

검증은 IP 프로토콜에서 시작해 데이터 패스 통합, 시스템 계약 순으로 넓어집니다. 흔한 함정은 Layer 1만 통과하고 "검증 완료"로 보는 것입니다 — Layer 2/3에서야 드러나는 통합 결함(데이터 패스 라우팅 오류, 격리 위반)이 실제 escape의 큰 부분이기 때문입니다. M01에서 본 _격리_ 동기는 단순한 설계 목표가 아니라 검증해야 할 대상이 됩니다.

### 4.3 워크로드별 구성 평가

| 워크로드 | 핵심 IP/도메인 | 격리 요구 | 평가 포인트 |
|---|---|---|---|
| AI 클러스터 | RDMA + Ethernet (Networking/Storage) | 중 | GPU·스토리지 간 지연, 데이터 패스 처리량 |
| 금융 | 암호화 + 가상화 (Security/Virtualization) | 상 | 테넌트 격리, 암호화 지연 |
| 베어메탈 클라우드 | 가상화 + 스토리지 | 상 | 하이퍼바이저 없는 관리형 인프라 격리 |

spec이 강조하듯 워크로드마다 처리량·지연·CPU 사용량·전력의 우선순위가 다르고, 적합한 가속기 구성과 소프트웨어 스택도 달라집니다. 검증 계획도 이 우선순위를 반영해 어느 층위·어느 IP를 더 깊게 볼지 정해야 합니다.

## 5. 디테일 — 통합·격리·관측

### 5.1 데이터 패스 통합에서 보는 것

데이터 패스 엔진은 분류·전달·DMA·큐 관리를 담당하므로, IP 통합 검증은 이 엔진이 요청을 _올바른 가속기로 라우팅_ 하는지, _DMA가 호스트 메모리에 정확히 전달_ 하는지, 큐가 backpressure 하에서도 데이터를 잃지 않는지를 본다는 점이 핵심입니다. 이는 개별 프로토콜 검증과는 다른 시스템 레벨 시나리오(여러 IP를 동시에 자극)를 요구합니다.

:::note[메커니즘 — data path engine이 "어느 가속기로 보낼지" 정하는 dispatch]
여러 가속기(RDMA·TOE·암호화·압축)가 _하나의 공통 데이터 패스_ 를 공유하는데, 어떻게 각 요청이 제 갈 곳을 찾을까요? 핵심은 **descriptor 또는 패킷 헤더의 _type 필드_ 를 보고 분기(dispatch)** 하는 것입니다.

호스트가 요청을 큐에 넣을 때(또는 패킷이 도착할 때), 그 descriptor/header 에는 "이건 RDMA 동작" / "이건 일반 TCP 전송" / "암호화 필요" 같은 **종류를 나타내는 필드** 가 들어 있습니다. 데이터 패스 엔진은 이 type 필드를 lookup 해 **해당 가속기로 가는 내부 경로(포트/큐)로 라우팅** 합니다 — §M02 의 match-action table 과 같은 "key → 목적지" 패턴이며, 여기서 key 는 type 필드, action 은 "어느 가속기로 전달" 입니다. 한 요청이 여러 가속기를 거쳐야 하면(예: 원격 read → RDMA → 복호화), 엔진이 _단계마다_ 다음 목적지로 다시 dispatch 합니다.

이 dispatch 메커니즘이 "여러 IP 가 한 데이터 패스를 공유" 를 가능하게 하는 핵심입니다. 검증 관점에서는 _type 필드 조합마다_ 올바른 가속기로 갔는지, 잘못된/예약된 type 값에 어떻게 반응하는지, 여러 type 이 섞여 들어올 때 라우팅이 뒤섞이지 않는지가 시스템 레벨 시나리오의 핵심입니다.
:::

:::note[메커니즘 — 한 가속기의 stall이 상류로 전파되는 backpressure]
"큐가 backpressure 하에서 데이터를 안 잃는지" 가 _어떻게_ 보장되는지의 원리입니다. 여러 IP 가 데이터 패스로 직렬 연결돼 있으므로, 어느 한 가속기가 느려지면(예: 암호화 엔진이 일시적으로 바빠 입력을 못 받음) 그 정체가 상류로 _번져야_ 데이터 손실 없이 멈출 수 있습니다.

이를 가능하게 하는 것이 **ready/credit 기반 flow control** 입니다. 인접 블록 사이의 인터페이스는 보통 "보내는 쪽 valid + 받는 쪽 ready" 핸드셰이크(또는 받는 쪽이 미리 발급한 credit)를 씁니다. 받는 가속기가 buffer 가 차서 **ready 를 내리면(또는 credit 을 소진하면)**, 바로 앞 블록은 _데이터를 흘려보내지 못하고 멈춥니다_ — 멋대로 보내서 떨어뜨리는 게 아니라, "받을 수 있다" 는 신호가 올 때까지 잡아둡니다. 그러면 그 앞 블록의 buffer 가 차고, 다시 그 ready 가 내려가 _또 한 단계 상류_ 가 멈춥니다. 이렇게 **stall 이 ready/credit 신호를 타고 가속기 → 데이터 패스 엔진 → 호스트 인터페이스까지 hop 단위로 역전파** 되어, 결국 호스트가 새 요청 주입을 늦춥니다.

핵심은 _drop 대신 정지_ 라는 점입니다 — 모든 경계가 "받을 준비가 됐을 때만 전달" 하므로 한 곳의 정체가 데이터 손실로 이어지지 않고 시스템 전체가 같이 느려질 뿐입니다 (PFC 의 hop-by-hop PAUSE 나 TOE 의 AXI-S tready 전파와 같은 원리). 검증에서는 특정 가속기를 일부러 stall 시켜 backpressure 가 상류까지 정확히 전파되고 drop 이 0 인지 확인하는 것이 핵심 시나리오입니다.
:::

### 5.2 격리 검증 — DPU의 핵심 가치 중 하나

DPU의 세 동기 중 _격리_ 는 검증에서 특히 중요합니다. 인프라 서비스가 호스트 OS·테넌트와 분리되어야 한다는 것은, 테넌트 워크로드가 인프라 제어 경로에 접근하거나 다른 테넌트의 데이터에 도달하지 못해야 함을 의미합니다. 베어메탈 서비스처럼 호스트에 하이퍼바이저가 없는 구성에서는 이 격리를 DPU가 단독으로 보장하므로, 격리 위반은 곧 심각한 보안 결함입니다.

### 5.3 관측·텔레메트리

spec은 보안 처리 항목에 텔레메트리를 포함합니다. DPU는 호스트와 분리된 위치에서 트래픽·상태를 관측할 수 있어, 검증 시에도 이 관측 경로가 정확한 데이터를 제공하는지(카운터, 이벤트 로깅)가 점검 대상이 됩니다.

### 5.4 명칭이 아니라 기능으로 검증 범위 정하기

제품마다 오프로드 범위와 프로그래머빌리티가 다르므로(M01 §5.3), 검증 범위도 "DPU니까 다 한다"가 아니라 _이 제품이 실제로 오프로드하는 기능_ 을 기준으로 정해야 합니다. spec의 제품 평가 질문(어떤 기능을, 데이터/제어 패스 중 어디까지, 호스트와 어떻게 분리, 워크로드에서 처리량·지연·전력이 어떻게 변하는가, 기존 도구와 통합 가능한가)이 그대로 검증 계획의 스코핑 체크리스트가 됩니다.

## 6. 흔한 오해 와 DV 디버그 체크리스트

### 흔한 오해

:::danger[❓ 오해 1 — 'IP 프로토콜만 통과하면 DPU 검증 끝']
**실제**: IP 단독(Layer 1)은 시작일 뿐입니다. 데이터 패스 통합(Layer 2)과 호스트 인터페이스·격리(Layer 3)에서 드러나는 통합 결함이 실제 escape의 큰 부분입니다.<br>
**왜 헷갈리는가**: 프로토콜 PASS가 "완료"처럼 느껴져서 — 시스템 계약은 별도 층위입니다.
:::
:::danger[❓ 오해 2 — '격리는 보안팀 일이지 DV 일이 아니다']
**실제**: 격리는 DPU의 세 핵심 동기 중 하나이며 _검증 대상_ 입니다. 특히 베어메탈처럼 하이퍼바이저 없는 구성에서는 DPU가 격리를 단독 보장하므로 격리 위반은 검증이 잡아야 할 심각 결함입니다.<br>
**왜 헷갈리는가**: 격리를 정책/운영 이슈로만 봐서 — DPU에서는 하드웨어 동작의 일부입니다.
:::
:::danger[❓ 오해 3 — '모든 DPU 검증 범위는 동일하다']
**실제**: 제품마다 오프로드 기능·프로그래머빌리티가 다르므로 검증 범위도 제품의 _실제 기능_ 에 맞춰 스코핑해야 합니다.<br>
**왜 헷갈리는가**: "DPU"라는 이름이 전 기능을 암시해서 — spec의 제품 평가 질문으로 범위를 좁혀야 합니다.
:::

### DV 디버그 체크리스트 (시스템 통합에서 마주칠 첫 실패들)

| 증상 | 1차 의심 | 어디 보나 |
|---|---|---|
| IP 단독은 PASS, 통합 시나리오에서 데이터 누락 | 데이터 패스 라우팅/DMA/큐 backpressure | §5.1, 데이터 패스 엔진 연결 |
| 테넌트가 다른 데이터에 접근 가능 | 격리 위반 (Layer 3) | §5.2 격리 검증 |
| "이 요구사항 어디서 왔나" (호스트 인터페이스 등) | IP가 DPU 시스템 계약을 상속 | §4.1 시스템 계약 표 |
| 검증 범위가 끝없이 늘어남 | 제품 실제 기능으로 스코핑 안 함 | §5.4, spec §6 평가 질문 |
| 텔레메트리 카운터 불일치 | 관측 경로 자체 결함 | §5.3 |

## 7. 핵심 정리 (Key Takeaways)

- **개별 IP는 DPU의 한 블록**: RDMA·TOE·Ethernet·암호화는 공통 데이터 패스 엔진을 통해 호스트·네트워크에 연결된다.
- **한 요청이 여러 IP를 직렬로 통과**: 원격 read 하나가 PCIe → Data Path → RDMA → TOE/Ethernet → 암호화/DMA를 거친다.
- **검증은 세 층위**: IP 단독(프로토콜) → IP 간 통합(데이터 패스) → 시스템 계약(호스트 인터페이스·격리). Layer 1만으로는 부족.
- **격리는 검증 대상**: DPU의 핵심 동기이자 하드웨어 동작 — 특히 베어메탈에서 격리 위반은 심각 결함.
- **명칭이 아니라 기능으로 스코핑**: 제품마다 오프로드 범위가 다르므로 spec의 평가 질문으로 검증 범위를 정한다.

:::caution[실무 주의점]
- 통합 검증은 _여러 IP를 동시에 자극_ 하는 시스템 레벨 시나리오를 요구합니다 — 단독 프로토콜 시퀀스로는 데이터 패스 결함을 못 잡습니다.
- 본 모듈의 IP 내부 흐름(§3)은 개념 설명을 위한 단순화(추론)이며, 실제 IP 동작은 해당 프로토콜 코스/스펙을 따릅니다.
:::
### 7.1 자가 점검

:::tip[🤔 Q1 — IP 추적 (Bloom: Analyze)]
"호스트의 원격 메모리 read"가 DPU 안에서 통과하는 블록을 순서대로 나열하라.
<details>
<summary>정답</summary>

Host Interface(PCIe) → Data Path Engine(분류·라우팅) → RDMA 엔진(메모리 접근 의미) → TOE/Ethernet(전송·프레이밍, 네트워크 송수신) → (수신) Data Path/암호화 가속기(필요 시 복호화) → DMA로 호스트 메모리 전달. 한 요청이 여러 IP를 직렬로 통과하므로 검증은 IP 단독·IP 간 통합·시스템 계약 세 층위를 모두 봐야 합니다.

</details>
:::
:::tip[🤔 Q2 — 워크로드 평가 (Bloom: Evaluate)]
금융 서비스용 DPU 검증에서 AI 클러스터용보다 더 비중을 둘 항목은?
<details>
<summary>정답</summary>

**보안·가상화 도메인과 격리(Layer 3)** 에 더 비중을 둡니다. 금융은 테넌트 격리와 암호화 지연이 우선순위이므로, 격리 위반 시나리오·암호화 가속기 동작·접근 제어를 깊게 봐야 합니다. AI 클러스터는 RDMA/Ethernet 데이터 패스 처리량·지연이 우선이라 검증 초점이 다릅니다. 워크로드의 우선순위가 검증 계획의 깊이를 결정합니다.

</details>
