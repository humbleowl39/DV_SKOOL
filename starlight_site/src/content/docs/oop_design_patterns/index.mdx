## 왜 OOP와 디자인 패턴을 배우는가

UVM은 그 자체가 처음부터 끝까지 객체지향 프레임워크입니다. `uvm_component`를 상속하고, virtual function을 오버라이드하고, factory로 타입을 교체하고, TLM port로 컴포넌트를 느슨하게 연결하는 이 모든 동작은 OOP의 네 기둥(캡슐화·추상화·상속·다형성)과 SOLID 원칙, 그리고 GoF 디자인 패턴이 SystemVerilog 문법으로 옷을 갈아입은 것에 지나지 않습니다. 따라서 OOP의 토대를 모른 채 UVM을 외우면, "왜 driver는 `uvm_driver`를 상속하는가", "왜 monitor는 누가 듣는지 모른 채 broadcast하는가", "왜 factory override가 테스트 코드를 건드리지 않고도 동작을 바꾸는가" 같은 질문에 답할 수 없습니다. 코드는 굴러가지만 설계 의사결정은 보이지 않는 상태가 됩니다.

이 코스는 그 토대를 세웁니다. 먼저 객체지향이 절차적 코드의 어떤 문제를 풀려고 등장했는지에서 출발해 네 기둥과 IS-A/HAS-A/CAN-DO 관계를 정리하고, 그 위에 SOLID 다섯 원칙을 쌓습니다. 이어서 같은 문제를 반복해 풀어 온 선배 개발자들이 정제한 "레시피"인 GoF 23 패턴을 분류별로 훑고, 마지막으로 이 모든 개념이 SystemVerilog와 UVM에서 어떤 문법·관용구로 나타나는지를 한 장의 매핑으로 연결합니다. 이 코스를 마치면 UVM 코드를 읽을 때 "이건 factory method 패턴이고 OCP를 구현한 것"이라고 설계 의도를 짚어낼 수 있게 됩니다.

## 학습 목표

코스를 마치면 다음을 할 수 있습니다:

- **Explain (이해)** 객체지향이 절차적 코드의 어떤 한계를 해결하는지와 네 기둥(캡슐화·추상화·상속·다형성)의 역할을 설명
- **Differentiate (분석)** IS-A / HAS-A / CAN-DO 관계와 association/aggregation/composition의 수명주기 차이를 구분
- **Apply (적용)** SOLID 다섯 원칙을 클래스 설계와 코드 리뷰에 적용해 결합도를 낮추는 리팩터링을 수행
- **Analyze (분석)** GoF 23 패턴을 creational/structural/behavioral로 분류하고 각 패턴이 해결하는 문제를 트레이스
- **Implement (적용)** OOP/패턴 개념을 SystemVerilog/UVM 문법(상속·virtual·factory·config_db·TLM)으로 구현
- **Evaluate (평가)** 패턴을 도입할지 말지를 "더 단순한 방법이 없는가" 기준으로 판단하고 과용을 경계

## 사전 지식

다음 항목을 알고 있으면 본문이 매끄럽게 읽힙니다:

- **프로그래밍 일반 경험**: 함수, 변수, 자료구조를 다뤄 본 경험 (언어 무관)
- **SystemVerilog class 문법 기초**: `class`, `extends`, `function`/`task` (4장 매핑에서 활용)
- **UVM 기초 개념**(권장): 이 코스는 [UVM 코스](../uvm/)의 선수 과목입니다 — 먼저 보면 4장 매핑이 훨씬 선명해집니다

부족하면 *Head First Object-Oriented Analysis and Design* (O'Reilly) 또는 refactoring.guru의 Design Patterns 섹션을 함께 보세요.

## 코스 운영 방식

각 모듈은 다음 순서로 구성됩니다:

1. **학습 목표** — 모듈 완료 후 할 수 있는 것 (Bloom's Taxonomy 동사)
2. **사전 지식** — 본 모듈에 필요한 선행 학습 항목
3. **본문** — Why care? → Intuition(비유 + 다이어그램) → 작은 예 → 일반화 → 디테일 → 흔한 오해 → 핵심 정리
4. **흔한 오해 + 디버그 체크리스트** — 실무에서 마주칠 함정
5. **핵심 정리** — 압축된 takeaway + 자가 점검
6. **퀴즈** — 이해도 점검 (별도 페이지)

:::tip[효율적 학습]
- **토대 먼저**: 01–02를 건너뛰고 패턴부터 보면 "왜 이 모양인지"가 보이지 않습니다. 4기둥과 SOLID가 패턴의 재료입니다.
- **UVM과 왕복**: 4장의 매핑 표를 [UVM 코스](../uvm/)와 번갈아 보면 추상 개념이 구체 코드로 묶입니다.
- **과용 경계**: 패턴은 만능이 아닙니다. "더 단순한 방법이 없는가"를 항상 먼저 물으세요.
- **요리 비유 활용**: 4기둥=도구, SOLID=도구 쓰는 법, 패턴=레시피. 이 위계를 머릿속에 두면 전체 그림이 잡힙니다.
:::

:::caution[패턴 과용 주의]
디자인 패턴은 자주 발생하는 문제의 검증된 해법이지, 모든 코드에 끼워 넣어야 할 의무가 아닙니다. 단순한 문제에 패턴을 강제하면 오히려 코드가 복잡해집니다. Module 03에서 이 함정을 별도로 다룹니다.
:::
