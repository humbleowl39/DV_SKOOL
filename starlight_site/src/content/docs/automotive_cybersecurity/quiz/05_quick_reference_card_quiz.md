## Q1. (Remember)

차량 보안 4-layer 표준 스택을 순서대로 나열하라.

<details>
<summary>정답 / 해설</summary>

1. **HSM** (Root of Trust, 키 보호)
2. **SecOC** (메시지 인증 + Freshness)
3. **Secure Gateway** (도메인 격리, 라우팅 정책)
4. **IDS** (이상 탐지) + Cloud 검증 (선택적 5층).

이 순서가 중요한 이유는 각 계층이 아래 계층을 전제로 작동하기 때문입니다. HSM이 없으면 SecOC가 사용하는 키가 소프트웨어로 노출되어 MAC을 계산해도 의미가 없고, SecOC가 없으면 Gateway가 필터링하는 메시지의 진위를 보장할 수 없으며, Gateway가 없으면 IDS가 탐지하더라도 도메인 간 격리가 없어 피해 확산을 막기 어렵습니다. 1→4 순서는 동시에 "신뢰의 기원(Root of Trust)에서 감시(IDS)까지"의 계층 관계를 나타내며, 어느 계층 하나라도 빠지면 그 위의 계층이 보장하는 속성이 흔들립니다.

</details>
## Q2. (Apply)

면접에서 "CAN 이 왜 위험한가?" 라는 질문을 받았다. 30초 내에 답하기 위한 4-요점 템플릿은?

<details>
<summary>정답 / 해설</summary>

1. CAN(1983) 은 인증/암호화가 없다 (broadcast).
2. OBD-II 로 외부 접근이 합법적으로 열려 있다.
3. 어떤 노드든 임의 ECU 를 가장해 메시지 송신 가능 (spoof + replay + flood).
4. 그래서 SecOC + Gateway + IDS 가 필요 — UN R155 가 OEM 에 사실상 의무화.

이 4요점이 면접 답변으로 적합한 이유는 "문제의 원인 → 구체적 위협 → 필요한 해결책 → 규제 압력"이라는 인과 흐름을 30초 안에 전달하기 때문입니다. 요점 1은 왜 CAN이 태생적으로 취약한지 설계 역사를 근거로 제시하고, 요점 2는 그 취약점이 현실에서 어떻게 노출되는지 보여줍니다. 요점 3은 단순 취약점 언급을 넘어 구체적 공격 유형을 나열해 기술적 깊이를 보여주며, 요점 4는 "그래서 어떻게 해야 하는가"와 "규제가 이미 이것을 요구한다"는 실무 맥락으로 마무리합니다. 이 흐름은 면접관이 "그럼 어떻게 막나요?"라는 추가 질문을 하기 전에 선제적으로 답을 포함하는 구조입니다.

</details>
## Q3. (Apply)

리뷰에서 자기 ECU 에 빠진 보안 계층을 빠르게 식별하는 체크리스트는?

<details>
<summary>정답 / 해설</summary>

- [ ] HSM 또는 동등한 secure element 가 있는가?
- [ ] Secure Boot 가 마지막 stage 까지 chain of trust 를 유지하는가?
- [ ] CAN/Ethernet 메시지에 SecOC 또는 동등 인증이 있는가?
- [ ] 도메인 사이에 Gateway 화이트리스트/rate limit 이 있는가?
- [ ] IDS 또는 anomaly 로그가 V-SOC 로 전달되는가?
- [ ] OTA 패키지가 서명 검증 + 롤백 보호되는가?

이 체크리스트가 유효한 이유는 앞서 배운 4-layer 보안 스택을 "설계 리뷰에서 즉시 사용할 수 있는 질문 형태"로 변환했기 때문입니다. 각 체크 항목은 특정 계층의 존재 여부를 묻고, 하나라도 "아니오"가 나오면 그 계층이 빠진 것입니다. 예를 들어 SecOC 항목이 빠지면 Tesla 사례와 같이 CAN 인증이 없는 상태가 되고, OTA 서명 검증 항목이 빠지면 공급망 공격에 취약해집니다. 리뷰어로서 이 목록을 순서대로 묻는 것만으로도 구조적 보안 공백을 5분 안에 식별할 수 있습니다.

</details>
## Q4. (Evaluate)

UN R155(CSMS) 와 ISO/SAE 21434 의 관계를 한 줄로 평가하라.

<details>
<summary>정답 / 해설</summary>

**R155 = "WHAT" (규제 요건)**, **21434 = "HOW" (엔지니어링 방법)**. R155 가 OEM 에 CSMS 를 요구하면 그 CSMS 의 구체적 활동(TARA, 검증 등) 은 21434 에 따라 수행되는 것이 사실상의 표준 경로. 둘은 보완 관계 — 어느 하나만으로는 부족.

이 관계를 "보완적"이라고 평가하는 근거는 두 문서의 성격 차이에 있습니다. UN R155는 규제 당국이 OEM을 심사하는 기준으로, "CSMS를 운영하라", "위험 식별과 대응 사이클을 갖춰라"라는 결과 요건을 명시합니다. 그러나 어떻게 TARA를 수행할지, 어떤 공학적 활동이 "검증"에 해당하는지는 정의하지 않습니다. ISO/SAE 21434는 이 빈칸을 채웁니다. TARA의 자산 식별, 위협 시나리오 작성, 위험 등급 산정 방법론과 검증 활동을 구체적으로 정의합니다. 따라서 21434 없이 R155만 따르면 "무엇을 해야 하는지는 알지만 어떻게 해야 하는지 모르는" 상태가 되고, R155 없이 21434만 따르면 규제 요건 충족 여부를 확인할 기준이 없어집니다.

</details>
## Q5. (Evaluate)

이 코스를 마친 학습자가 "차량 보안 엔지니어로 즉시 실전 투입 가능" 하기 위해 추가로 학습해야 할 4영역은?

<details>
<summary>정답 / 해설</summary>

1. **AUTOSAR Classic / Adaptive 실 코드** — SecOC, CryIf, KeyM 모듈의 실제 사용.
2. **AUTOSAR · UDS · DoIP 진단 보안** — Security Access(0x27), Authentication(0x29).
3. **Penetration testing 도구** — CANalyzer / SocketCAN / 리버스 엔지니어링 (Ghidra, IDA).
4. **TARA 실습** — 자기 시스템에 직접 적용 + 외부 reviewer 와의 challenge 사이클 경험.

이 네 영역이 추가로 필요한 이유는 이 코스가 "무엇을 왜 해야 하는가"의 개념을 다루지만 "실제로 어떻게 구현하고 공격하고 평가하는가"는 다루지 않기 때문입니다. AUTOSAR 실 코드 경험 없이는 SecOC 모듈 설정 파라미터를 잘못 구성하는 흔한 실수를 피하기 어렵고, 진단 보안을 모르면 OBD-II나 DoIP를 통한 공격 경로를 제대로 평가할 수 없습니다. 침투 테스트 도구 없이는 내가 설계한 방어가 실제로 작동하는지 검증할 수단이 없으며, TARA를 교재로만 배우는 것과 실제 시스템에 적용해 외부 리뷰어의 반박을 받아보는 것 사이에는 넘기 어려운 실력 차이가 있습니다.

</details>
