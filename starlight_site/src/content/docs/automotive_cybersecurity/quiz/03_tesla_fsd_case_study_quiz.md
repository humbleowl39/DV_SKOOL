## Q1. (Remember)

2023 Pwn2Own Tesla FSD 탈옥의 출발점이 된 두 약점은?

<details>
<summary>정답 / 해설</summary>

1. **CAN 통신의 무인증** — 내부 메시지에 SecOC 미적용.
2. **로컬 Feature Flag 의존** — FSD 활성화 여부를 차량 내 NVRAM 의 플래그로 결정 → 변조 가능.

이 두 약점이 "출발점"인 이유는 탈옥 체인의 첫 관문 역할을 했기 때문입니다. 공격자가 물리적으로 접근한 다음, CAN 인증이 없으므로 내부 버스에 임의 메시지를 주입할 수 있었고, Feature Flag가 로컬 저장소에 있었으므로 서버 검증 없이 NVRAM을 직접 조작해 FSD를 활성화할 수 있었습니다. 만약 CAN에 SecOC가 적용되어 있었다면 메시지 주입 자체가 MAC 검증으로 거부되었을 것이고, Feature Flag가 클라우드 서명 기반이었다면 로컬 조작만으로는 활성화가 불가능했을 것입니다.

</details>
## Q2. (Understand)

SCS(보안 칩) 가 있어도 탈옥이 가능했던 이유는?

<details>
<summary>정답 / 해설</summary>

SCS 가 보호한 영역은 **Secure Boot · OTA 검증 · Cloud 인증** 등 "외부 경계" 였고, **CAN 인증** 은 SCS 의 적용 범위 밖에 있었다. 즉, 보안 IP 가 있다는 사실 자체보다 **"어디에 적용되어 있는가"** 가 결정적이다.

이 사례가 "SCS가 있어도 탈옥이 가능한" 역설적 상황을 만든 이유는 보안 설계에서 흔히 발생하는 "경계 오판" 때문입니다. SCS는 외부 공격자가 인터넷을 통해 침투하거나 OTA를 위변조하는 시나리오를 막도록 설계되었고, 그 역할은 충실히 수행했습니다. 그러나 차량 내부의 CAN 버스는 "이미 신뢰할 수 있는 내부 영역"으로 간주하여 SCS의 보호 대상에서 제외했습니다. 공격자가 물리 접근으로 이 내부 경계에 들어오는 순간, SCS는 아무런 도움이 되지 않았습니다. 이것이 "보안 IP의 존재"와 "보안 커버리지"가 다른 개념임을 보여주는 핵심 교훈입니다.

</details>
## Q3. (Apply)

자기 회사의 ECU 한 종류를 골라 Tesla 사례를 적용해 위협 모델을 만들 때 던져야 할 5가지 질문은?

<details>
<summary>정답 / 해설</summary>

1. 부팅 검증은 어디까지 이어지는가? (Stage 별 chain of trust)
2. 런타임 통신(CAN/Ethernet) 도 인증되는가, 아니면 부팅만 보호되는가?
3. 정책(Feature Flag, mode switch) 이 로컬에 저장되는가, 클라우드 서명/대조가 있는가?
4. 외부 신호(GPS, V2X) 를 단독 보안 통제로 쓰고 있지는 않은가?
5. AUTOSAR 등 표준 스택을 안 쓴다면, 같은 기능을 자체 구현했는지 확인했는가?

이 다섯 질문이 유효한 이유는 Tesla 사례의 각 결함이 정확히 이 질문들에 대응하기 때문입니다. 질문 1과 2는 "부팅 보안이 런타임으로 이어지지 않은" 문제를 겨냥하고, 질문 3은 "로컬 플래그 의존"이라는 정책 결함을 검증하며, 질문 4는 외부 신호를 신뢰하는 아키텍처 취약점을 잡아냅니다. 질문 5는 특히 중요한데, AUTOSAR SecOC처럼 이미 검증된 표준 구현을 사용하지 않으면 같은 기능을 자체 구현하더라도 보안 검토가 별도로 필요하기 때문입니다. 이 다섯 질문을 체크리스트로 사용하면 Tesla 사례와 유사한 구조적 취약점을 자기 시스템에서 미리 발견할 수 있습니다.

</details>
## Q4. (Analyze)

Tesla 탈옥 체인의 단계를 ① 기술적 결함 ② 정책적 결함 ③ 아키텍처 결함으로 분류하라.

<details>
<summary>정답 / 해설</summary>

- **기술적 결함** : CAN 메시지 무인증, GPS spoofing 가능성.
- **정책적 결함** : "이 지역에서는 FSD 비활성" 같은 통제가 로컬 플래그로 결정됨, OTA 후 검증이 차량 내부에 머무름.
- **아키텍처 결함** : 외부 경계만 SCS 로 보호하고 내부 경계(CAN) 를 신뢰함. AUTOSAR 생태계 밖에서 자체 SW 스택 사용 → SecOC 누락.

이 3축을 모두 갖추지 못하면 보안 IP 만으로는 부족하다.

세 가지 결함을 분류하는 이유는 대응 방법이 각기 다르기 때문입니다. 기술적 결함은 SecOC 적용이나 패치로 해결할 수 있지만, 정책적 결함은 아키텍처 결정(클라우드 검증 도입)을 바꿔야 하고, 아키텍처 결함은 설계 단계로 돌아가 신뢰 경계를 재설정해야 합니다. Tesla 사례에서 세 가지가 동시에 존재했던 것은 "빠른 개발 사이클"과 "표준 외 자체 스택" 선택이 가져온 결과로, 어느 하나만 수정해도 탈옥 체인의 고리 하나가 끊어졌을 것입니다. 이 분류는 차후 개선 우선순위를 정할 때도 유용합니다.

</details>
## Q5. (Evaluate)

같은 사건이 SecOC + Cloud 검증 기반 OEM 에서 발생할 수 있는가? 평가하라.

<details>
<summary>정답 / 해설</summary>

**가능성 ↓ (그러나 0은 아님)**.
- SecOC 가 있으면 외부 노드의 CAN 주입이 거의 불가능 (MAC 검증 실패).
- Cloud 검증이 있으면 차량 내부 플래그만 바꿔도 서버 측 정책과 충돌해 거부 가능.

그러나:
- **합법 ECU 가 침해되면** SecOC 도 통과 가능 → IDS + 키 회수 정책 필요.
- **Cloud 통신 자체가 가용성 의존** → 통신 단절 시 정책 fallback 의 보수성 점검 필요.

결론: 다층 방어가 있으면 같은 종류의 단순 탈옥은 막히지만, 공격자는 다른 surface 로 이동한다 — 보안은 끝이 없다.

"가능성 낮아졌지만 0은 아니다"가 정답인 이유는 보안 통제가 위협을 "불가능하게" 만드는 것이 아니라 "공격 비용을 충분히 높이는 것"이 목표이기 때문입니다. SecOC와 클라우드 검증이 있으면 Tesla 사례에서 쓰인 단순 CAN 주입과 로컬 플래그 조작이라는 공격 경로는 막힙니다. 그러나 공격자가 합법적 ECU를 먼저 침해해 그 키를 획득하거나, 클라우드 통신 자체를 가로막는 새로운 전술로 이동한다면 다층 방어도 우회 가능합니다. 따라서 "SecOC가 있으니 안전하다"는 결론이 아니라, "어떤 다른 경로가 새로 생겼는가"를 지속적으로 재평가해야 한다는 것이 이 질문의 핵심입니다.

</details>
