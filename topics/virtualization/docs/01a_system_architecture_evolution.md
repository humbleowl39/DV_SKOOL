# Unit 1a: 시스템 아키텍처 진화 — HW Only에서 가상화까지

## 핵심 개념
**현대 가상화 시스템은 하루아침에 만들어진 것이 아니다. 고정 기능 HW → 프로그래머블 프로세서 → HW 가속기 통합 → OS 도입 → MMU/IOMMU → 가상화로 이어지는 진화 과정의 결과물이다. 각 단계마다 이전 단계의 한계를 극복하기 위한 HW/SW가 추가되었다.**

> 이 Unit은 DV TechForum #54 "Evolution of system architecture, HW virtualization" 발표 내용을 기반으로 정리한 것이다.

---

## 진화 7단계 전체 조감도

```
단계 1        단계 2         단계 3          단계 4
HW Only  →  Processor(FW) → FW+HW Accel → OS(ARM-M)+HW Accel
고정 기능     프로그래머블     하이브리드       MPU, DRAM, 시스템콜
             저성능          고성능           애플리케이션 특화

    │              │              │              │
    ▼              ▼              ▼              ▼

단계 5              단계 6                    단계 7
OS(ARM-A)+MMU  →  +PEs+IOMMU+LLC+Coherency → Virtualization
범용, 고성능        확장 가능                   VM 격리
IOMMU 없음 (문제)   현대 SoC                   HW 지원
```

---

## 단계 1: HW Only (고정 기능)

### 구조

```
User inputs → [ Sensor ] → [ HW IP ] → [ Actuator ] → Executed outputs
```

HW 로직(게이트, ASIC)만으로 구성된 시스템. 소프트웨어 없음.

### 특징

| 항목 | 설명 |
|------|------|
| 성능 | **극도로 높음** — 고정 기능에 최적화된 HW |
| 에너지 효율 | **최고** — 불필요한 회로 없음 |
| 유연성 | **없음** — 기능 변경 = 재설계 |
| 예시 | 고정 알고리즘 신호처리기, 초기 계산기 |

### 한계와 전환 동기

```
문제: 요구사항이 바뀌면 HW를 처음부터 다시 설계해야 함
      → 비용과 시간이 막대

동기: "프로그래밍으로 기능을 바꿀 수 있는 시스템이 필요하다"
      → 프로세서 (마이크로컨트롤러) 도입
```

---

## 단계 2: Processor + FW (프로그래머블)

### 구조

```
                 ┌─────────┐
                 │  Flash  │ ← FW 저장
                 └────┬────┘
                      │
User inputs → Sensor →┤ Processor(FW) ├→ System Interconnect → Actuator → Outputs
                      │                │
                      └────────────────┘
                             │
                          ┌──┴──┐
                          │ RAM │
                          └─────┘
```

마이크로컨트롤러 기반. Flash에 펌웨어를 저장하고, RAM에서 실행.

### 특징

| 항목 | 설명 |
|------|------|
| 성능 | **낮음** — 범용 프로세서의 한계 |
| 유연성 | **있음** — FW 업데이트로 기능 변경 가능 |
| 예시 | 전자레인지, 세탁기, 간단한 소비자 전자기기, 장난감 |

### 한계와 전환 동기

```
문제: 범용 프로세서로는 연산 집약적 작업(영상처리, 암호화 등)의 성능 부족
      → SW만으로는 실시간 요구를 충족할 수 없음

동기: "반복적이고 무거운 연산은 전용 HW에 맡기자"
      → HW 가속기 (Hardware Accelerator) 추가
```

## 단계 3: Processor(FW) + HW Accelerator (하이브리드)

### 구조

```
                 ┌─────────┐
                 │  Flash  │
                 └────┬────┘
                      │         Configuration
              ┌───────┴───────┐  Interface   ┌──────────────┐
              │ Processor(FW) │──Interrupt──→│ HW Accelerator│
              └───────┬───────┘              └──────┬───────┘
                      │                             │
                      │     Memory access            │
                      │     physical address directly │
              ┌───────┴─────────────────────────────┴──┐
              │          System Interconnect             │
              └────────────────┬────────────────────────┘
                               │
                            ┌──┴──┐
                            │ RAM │
                            └─────┘
```

프로세서가 제어/관리를 담당하고, 연산 집약적 작업은 HW 가속기가 처리.

### 핵심 포인트

- **HW Accelerator는 물리 주소(PA)로 직접 메모리 접근**
- 프로세서가 Configuration Interface로 가속기를 설정
- 가속기 작업 완료 시 Interrupt로 프로세서에 통보

### 특징

| 항목 | 설명 |
|------|------|
| 성능 | **높음** — 집약 연산을 HW 오프로드 |
| 유연성 | **제한적** — 가속기는 고정 기능, 프로세서는 프로그래머블 |
| 예시 | 데이터 로직 처리, 간단한 제어 시스템 + 액추에이터 |

### 한계와 전환 동기

```
문제 1: 확장성 — 더 복잡한 애플리케이션을 동시에 실행하려면?
문제 2: 보호 — 여러 작업이 메모리를 공유하면 충돌/보안 문제
문제 3: 관리 — 사용자 프로그램과 시스템 제어를 분리해야 함

동기: "OS를 도입하여 자원 관리 + 프로세스 격리를 하자"
      → OS + MPU(Memory Protection Unit) 도입
```

## 단계 4: Middle-end Processor(OS) + HW Accelerator + MPU

### 구조

```
                 ┌─────────┐
                 │  Flash  │
                 └────┬────┘
                      │         Configuration
    ┌─────────────────┴───────┐  Interface   ┌──────────────┐
    │  Processor (OS)         │──Interrupt──→│ HW Accelerator│
    │  ex) ARM-M              │              └──────┬───────┘
    │  ┌─────┐                │                     │
    │  │ MPU │ Memory         │     Memory access    │
    │  └─────┘ Protection     │     physical address │
    └─────────────────┬───────┘                     │
                      │                              │
              ┌───────┴──────────────────────────────┴──┐
              │          System Interconnect              │
              └────────┬──────────────────┬─────────────┘
                       │                  │
                    ┌──┴──┐            ┌──┴──┐
                    │ RAM │            │DRAM │ ← 대용량 메모리 도입
                    └─────┘            └─────┘
```

### 핵심 변화: OS + MPU 도입

```
OS 역할:
  - 시스템 자원 관리 (메모리, CPU 시간, I/O)
  - 사용자 애플리케이션과 시스템 제어 분리
  - 시스템 콜 인터페이스로 HW 접근 제어

MPU (Memory Protection Unit):
  - 메모리 영역별 접근 권한 설정 (R/W/X)
  - 사용자 프로세스가 다른 영역 침범 방지
  - MMU와 다름: 주소 변환 없음, 보호만 담당
```

### MPU vs MMU

| 항목 | MPU | MMU |
|------|-----|-----|
| 기능 | 메모리 **보호**만 | 메모리 보호 + **주소 변환** |
| 가상 주소 | 없음 (물리 주소 직접 사용) | VA → PA 변환 |
| 복잡도 | 단순 | 복잡 (Page Table, TLB) |
| 사용처 | ARM Cortex-M (RTOS) | ARM Cortex-A (Linux/범용 OS) |
| 페이지 단위 | 영역(region) 단위 | 페이지(4KB/2MB/1GB) 단위 |

### 특징

| 항목 | 설명 |
|------|------|
| 성능 | 높음 (HW 가속기 오프로드) |
| OS | RTOS 또는 경량 OS 실행 가능 |
| DRAM | 대용량 메모리 → 복잡한 애플리케이션 수용 |
| 예시 | 오디오/비디오 처리, 데이터 압축/암호화, 자동차 ECU, 로봇 |

### 한계와 전환 동기

```
문제 1: 가상 주소 없음 → 프로세스마다 물리 주소 직접 관리 (복잡, 비효율)
문제 2: HW 가속기가 PA로 직접 접근 → 보안 위험
문제 3: 범용 OS (Linux 등) 실행 어려움 → 생태계 제한

동기: "MMU를 도입하여 가상 주소 + 범용 OS + 풍부한 SW 생태계를 확보하자"
      → ARM Cortex-A급 프로세서 + MMU 도입
```

## 단계 5: High-end Processor(OS) + HW Accelerator + MMU (IOMMU 없음)

### 구조

```
                 ┌─────────┐
                 │  Flash  │
                 └────┬────┘
                      │          Configuration
    ┌─────────────────┴───────┐   Interface  ┌──────────────┐
    │  Processor (OS)         │──Interrupt──→│ HW Accelerator│
    │  ex) ARM-A              │    ctrlr     │    (PA)       │
    │  ┌───────┐ ┌─────┐     │              │  물리 주소로   │
    │  │ Cache │ │ MMU │     │              │  메모리 직접접근│
    │  └───────┘ └─────┘     │              └──────┬───────┘
    └─────────────────┬───────┘                     │
                      │        access on reserved   │
              ┌───────┴────────memory chunk──────────┴──┐
              │          System Interconnect              │
              └───┬──────────────────┬──────────────┬───┘
                  │                  │              │
               ┌──┴──┐           ┌──┴──┐        ┌──┴──┐
               │DRAM │           │DRAM │        │ RAM │
               └─────┘           └─────┘        └─────┘
```

### 핵심 변화: MMU 도입, 하지만 IOMMU 없음

```
CPU 측: MMU가 VA → PA 변환
  → 프로세스 격리, 가상 메모리, 범용 OS(Linux) 실행 가능

HW Accelerator 측: 여전히 PA(물리 주소)로 직접 접근
  → MMU의 보호를 받지 않음!
```

### IOMMU 없음의 문제 (중요)

```
문제 1: 메모리 비효율
  ┌──────────────────────────────────────────────┐
  │                    DRAM                       │
  │  ┌────────┐                   ┌────────────┐ │
  │  │ OS +   │   빈 공간 (낭비)  │ HW Accel용  │ │
  │  │ App    │                   │ 예약 영역   │ │
  │  │ 사용   │                   │ (큰 연속    │ │
  │  │        │                   │  메모리)    │ │
  │  └────────┘                   └────────────┘ │
  └──────────────────────────────────────────────┘

  HW 가속기는 PA로 접근하므로:
  → 큰 연속 물리 메모리 청크를 미리 예약해야 함
  → OS가 이 영역을 다른 용도로 사용 불가
  → 메모리 활용률 감소

문제 2: 보안 위험
  HW 가속기가 PA로 접근 = 어떤 물리 주소든 접근 가능
  → OS 커널 메모리, 다른 프로세스 메모리도 접근 가능
  → DMA 공격과 동일한 위험

  공격 시나리오:
    악의적 드라이버가 HW 가속기의 DMA 대상 주소를
    커널 메모리로 설정 → 커널 데이터 유출/변조
```

### 특징

| 항목 | 설명 |
|------|------|
| 성능 | 높음 (ARM-A + HW 가속기) |
| 범용성 | 높음 (Linux/Android 실행 가능) |
| 확장성 | **낮음** — IOMMU 없이 디바이스 관리 비효율적 |
| 예시 | 스마트폰, 태블릿, 스마트TV (초기 세대) |

### 한계와 전환 동기

```
동기: "HW 가속기/디바이스에도 주소 변환과 접근 제어가 필요하다"
      → IOMMU 도입
      "특수 연산 효율을 더 높이자"
      → GPU, DSP, NPU 등 PEs(Processing Elements) 통합
      "메모리 접근 지연을 줄이자"
      → Last Level Cache + HW Coherency
```

## 단계 6: 현대 SoC — IOMMU + PEs + LLC + HW Coherency

### 구조

```
                  ┌─────────┐
                  │  Flash  │
                  └────┬────┘
                       │
    ┌──────────────────┴────────┐  mailbox  ┌────────────────────┐
    │  Processor (OS)           │           │       PEs          │
    │  ex) ARM-A                ├───────────┤  GPU, DSP, NPU,    │
    │  ┌───────┐ ┌─────┐       │           │  DPU               │
    │  │ Cache │ │ MMU │       │           │  ┌─────┐           │
    │  └───────┘ └─────┘       │           │  │ MMU │           │
    └──────────┬────────────────┘           └───┬───────────────┘
               │                                │
               │    interrupt   ┌──────────────────────────┐
               │    ctrlr       │     HW Accelerator       │
               │                │        (VA!)             │
               │                │     ┌───────┐            │
               │                │     │ IOMMU │            │
               │                │     └───────┘            │
               │                │  per page translation    │
    ┌──────────┴────────────────┴──────────────────────────┴────┐
    │              ┌──────────────────┐                          │
    │              │ Last Level Cache │                          │
    │              └──────────────────┘                          │
    │                System Interconnect                         │
    │                + HW Coherency                              │
    └──────┬──────────────────┬──────────────────┬──────────────┘
           │                  │                  │
        ┌──┴──┐            ┌──┴──┐            ┌──┴──┐
        │DRAM │            │DRAM │            │ RAM │
        └─────┘            └─────┘            └─────┘
```

### 4대 핵심 추가 요소

#### 1. IOMMU (가장 중요한 전환점)

```
단계 5 (IOMMU 없음):             단계 6 (IOMMU 있음):

HW Accel → PA → DRAM             HW Accel → VA → IOMMU → PA → DRAM
(직접 물리 접근)                  (가상 주소로 접근)
                                           │
                                   ┌───────┴───────┐
                                   │ 페이지 단위    │
                                   │ 주소 변환      │
                                   │ + 접근 권한    │
                                   │   검사         │
                                   └───────────────┘
```

| IOMMU 이전 | IOMMU 이후 |
|-----------|-----------|
| HW 가속기가 PA 직접 접근 | VA → PA 변환 (페이지 단위) |
| 큰 연속 물리 메모리 예약 필요 | 가상으로 연속, 물리로 불연속 가능 |
| 임의 메모리 영역 접근 가능 (보안 위험) | 커널이 설정한 범위만 접근 가능 |
| 디바이스 드라이버가 물리 주소 의존 | 드라이버가 가상 주소 사용 → 이식성 향상 |

#### 2. Processing Elements (PEs)

| PE | 특화 분야 | 장점 |
|----|----------|------|
| **GPU** | 병렬 연산 (그래픽, GPGPU) | 수천 코어로 대규모 병렬 처리 |
| **DSP** | 신호 처리 (오디오, 통신) | 고정소수점 연산 최적화 |
| **NPU** | AI/ML 추론 | 행렬 연산, 저전력 추론 |
| **DPU** | 데이터 처리 (네트워크, 스토리지) | CPU 오프로드, 인프라 가속 |

#### 3. Last Level Cache (LLC)

```
CPU ──┐
      ├──→ LLC ──→ DRAM
PE  ──┤     (공유)
      │
HW  ──┘

- CPU, PE, HW 가속기가 LLC를 공유
- 자주 접근하는 데이터의 DRAM 접근 횟수 감소 → 지연시간 감소
- 컴포넌트 간 데이터 공유 효율 향상
```

#### 4. HW Coherency (하드웨어 일관성)

```
Coherency 없을 때:
  CPU가 캐시에 데이터 A = 10 보유
  GPU가 DRAM에서 A = 5를 읽음 (오래된 값)
  → 불일치! SW가 수동으로 flush/invalidate 필요

Coherency 있을 때:
  CPU가 A = 10으로 갱신하면
  HW coherency 프로토콜이 GPU 측에 자동 통보
  → GPU도 A = 10을 봄
  → SW 복잡도 대폭 감소
```

| 항목 | 설명 |
|------|------|
| 데이터 일관성 | CPU, GPU, 가속기 간 자동 캐시 동기화 |
| SW 단순화 | 수동 flush/invalidate 불필요 |
| 성능 최적화 | 불필요한 DRAM 접근 감소 |

### 이 단계의 특징

| 항목 | 설명 |
|------|------|
| 범용성 | **최고** — 어떤 워크로드든 처리 가능 |
| 확장성 | **높음** — PE/가속기 추가 용이, IOMMU로 관리 |
| 성능 | **높음** — LLC, Coherency, 특화 PE |
| 예시 | 현대 스마트폰 SoC (Snapdragon, Apple Silicon), 서버 SoC |

### 한계와 전환 동기

```
문제: 이 강력한 HW 위에서 여러 독립된 실행 환경을 동시에 운영하려면?
      - 서버: 여러 고객의 워크로드를 하나의 물리 머신에서 격리 실행
      - 자동차: 안전 기능과 인포테인먼트를 동일 SoC에서 격리 실행
      - 모바일: 보안 영역과 일반 영역 분리

동기: "HW 자원을 VM 단위로 분할/격리하자"
      → Hypervisor + 가상화 도입
```

---

## 단계 7a: 가상화 — HW 지원 없음 (SW 에뮬레이션)

### 구조

```
                      ┌──────────────────────────────────────────┐
                      │  기존 SoC + Hypervisor                    │
  ┌──────┐ ┌──────┐  │                                           │
  │ VM0  │ │ VM1  │  │  2-stage translation (CPU 메모리)          │
  │ 4GB  │ │ 2GB  │  │  인터럽트: Hypervisor가 capture 후 분배    │
  └──┬───┘ └──┬───┘  │  HW Accel: Hypervisor가 설정/관리          │
     │        │      │                                           │
     └────┬───┘      │  ┌─────────────────────────────────────┐  │
          │          │  │     DRAM                             │  │
          ▼          │  │ ┌────────┬────────┬────────┬──────┐ │  │
     Hypervisor      │  │ │VM0 STE│VM0 STE│VM1 STE│ ... │ │  │
                     │  │ └────────┴────────┴────────┴──────┘ │  │
                     │  └─────────────────────────────────────┘  │
                     └──────────────────────────────────────────┘
```

### SW 가상화의 한계

```
에뮬레이션 방식:
  - Hypervisor가 네트워크, 스토리지 등의 디바이스를 SW로 에뮬레이션
  - 각 VM에 가상 디바이스를 제공 (마치 물리 디바이스인 것처럼)
  - Hypervisor가 HW 자원을 VM에 시분할 할당

문제:
  - 모든 I/O가 Hypervisor를 경유 → 심각한 성능 오버헤드
  - HW 가속기 설정도 Hypervisor가 대행 → 지연
  - 확장성 제한 (VM 수 증가 시 오버헤드 선형 증가)
```

## 단계 7b: 가상화 — HW 지원 (현대 시스템)

### 핵심 변화: HW가 가상화를 직접 지원

```
HW 가상화 지원 3대 요소:
  1. I/O 가상화      → 디바이스를 VM별로 분할 (PF/VF)
  2. 인터럽트 가상화  → 인터럽트를 올바른 VM에 직접 전달
  3. 메모리 가상화    → IOMMU 2-stage translation
```

### I/O 가상화: PF/VF 분리

```
┌──────────────────────────────────────────────┐
│              HW IP (MVIP)                     │
│                                               │
│  ┌─────────┐   ← PF (Physical Function)      │
│  │ engine0 │     Hypervisor가 직접 설정/관리   │
│  │ (ctrlr) │     전체 디바이스 초기화 담당      │
│  └─────────┘                                  │
│                                               │
│  ┌─────┐ ┌─────┐ ┌─────┐  ← VF (Virtual      │
│  │ VF0 │ │ VF1 │ │ VF2 │    Function)         │
│  │     │ │     │ │     │    각 VM의 OS가       │
│  │VM0용│ │VM1용│ │VM2용│    직접 설정 가능      │
│  └─────┘ └─────┘ └─────┘                      │
└──────────────────────────────────────────────┘

Hypervisor 역할:
  - PF를 통해 디바이스 전체 초기화/관리
  - 각 VF를 해당 VM에 할당 (인터페이스를 VM에 노출)

VM의 Guest OS 역할:
  - 할당받은 VF를 직접 설정/사용
  - Hypervisor 개입 없이 I/O 수행 → Pass-through!
```

### 인터럽트 가상화

```
┌─────────────────────────────────────────────┐
│                                              │
│  VMx ←─── 인터럽트 ←─── Interrupt Controller │
│                              │               │
│                         Hypervisor           │
│                         (인터럽트 라우팅)     │
│                                              │
│  동작:                                       │
│    1. HW가 인터럽트 발생                      │
│    2. Interrupt controller가 포착             │
│    3. Hypervisor가 "어떤 VM의 인터럽트인지"    │
│       판단하여 해당 VM에 전달                  │
└─────────────────────────────────────────────┘
```

### 메모리 가상화: IOMMU 2-Stage Translation

```
┌──────────────────────────────────────────────────┐
│  IOMMU 2-Stage Translation                        │
│                                                    │
│  ┌──────────┐     ┌──────────┐     ┌──────────┐  │
│  │ Stage 1  │     │ Stage 2  │     │          │  │
│  │          │     │          │     │          │  │
│  │ VA → IPA │────→│ IPA → PA │────→│   DRAM   │  │
│  │          │     │          │     │          │  │
│  └──────────┘     └──────────┘     └──────────┘  │
│   VMx OS 관리       Hypervisor 관리                │
│                                                    │
│  AxUSER 신호:                                      │
│    디바이스가 AXI 트랜잭션에 AxUSER 필드를 포함     │
│    → IOMMU가 "이 접근이 어떤 SW 엔티티(VM/         │
│       Hypervisor)에 속하는지" 식별                  │
│    → 올바른 Stage 1/2 page table 선택              │
└──────────────────────────────────────────────────┘

VMx OS 입장:
  VA → IPA 를 관리 (IPA가 PA라고 생각함)

Hypervisor 입장:
  IPA → PA 를 관리 (실제 물리 메모리 배치 결정)
```

### SW 가상화 vs HW 가상화 비교

| 항목 | SW 가상화 (7a) | HW 가상화 (7b) |
|------|---------------|---------------|
| I/O | Hypervisor가 에뮬레이션 | PF/VF 분리, VM이 VF 직접 접근 |
| 인터럽트 | Hypervisor가 모든 인터럽트 처리 | HW 라우팅 + Hypervisor 분배 |
| 메모리 | 2-stage (SW 관리 비중 높음) | 2-stage (HW 지원, IOMMU) |
| HW 가속기 설정 | **Hypervisor가 설정** | **VM의 OS가 직접 설정** (VF) |
| 성능 | 오버헤드 큼 | Bare metal에 근접 |
| 확장성 | 낮음 | 높음 (VF 추가로 VM 수 확장) |

## 전체 진화 요약

```
단계  구성                     핵심 추가 요소          해결한 문제
────  ──────────────────────  ─────────────────────  ─────────────────
 1    HW Only                 (없음)                  —
 2    Processor (FW)          프로그래밍              유연성 확보
 3    Processor + HW Accel    HW 가속기              연산 성능
 4    OS(ARM-M) + HW Accel    OS, MPU, DRAM          자원 관리, 메모리 보호
 5    OS(ARM-A) + HW Accel    MMU, Cache             가상 주소, 범용 OS
 6    + PEs + IOMMU + LLC     IOMMU, LLC, Coherency  디바이스 격리, 확장성
 7a   + Virtualization (SW)   Hypervisor             VM 격리 (성능 희생)
 7b   + Virtualization (HW)   PF/VF, HW int, 2-stage VM 격리 + 성능 유지
```

## Q&A

**Q: IOMMU가 "가상화의 전제 조건"이라고 하는 이유는?**
> "가상화에서 디바이스를 VM에 pass-through 할당하려면 DMA 격리가 필수다. IOMMU 없이는: (1) 디바이스가 PA로 직접 접근하므로 VM 간 DMA 격리 불가 — VM0 디바이스가 VM1 메모리에 접근 가능, (2) 모든 I/O를 Hypervisor가 에뮬레이션해야 하여 성능 저하, (3) 각 VM에 연속 물리 메모리 예약 필요 — 비효율적. IOMMU는 '디바이스용 MMU'로서 디바이스 수준의 주소 변환 + 접근 제어를 제공하며, 이것 없이는 안전하고 효율적인 I/O 가상화가 불가능하다."

**Q: 시스템 아키텍처 진화에서 단계 6(IOMMU, PEs, LLC, HW Coherency)이 가상화에 기여하는 바는?**
> "네 요소가 각각 다른 측면을 해결한다. IOMMU — 디바이스 DMA의 2-stage translation으로 VM별 메모리 격리(핵심 전제 조건). PEs(GPU, NPU, DPU) — GPU pass-through, DPU 네트워크 오프로드 등 각 PE에도 IOMMU로 VM별 격리 적용. LLC — 여러 VM 실행 시 메모리 접근 지연 감소, Hypervisor page table도 캐싱되어 가상화 오버헤드 감소. HW Coherency — VM 간 PE 공유 시 SW 캐시 관리 불필요, 자동 일관성 유지로 SW 복잡도 감소."

**Q: AXI의 AxUSER 신호가 IOMMU 2-stage translation에서 하는 역할은?**
> "AxUSER(ARUSER/AWUSER)는 AXI 트랜잭션에 'VM 정체성(identity)'을 부여하는 메커니즘이다. 디바이스가 메모리 접근 시 AxUSER에 SW 엔티티 정보를 실어 보내면, IOMMU가 이를 읽고 해당 VM의 Stage 1 PT로 VA→IPA, Stage 2 PT로 IPA→PA 변환을 수행한다. AxUSER 없이는 IOMMU가 어떤 VM의 접근인지 구분 불가. ARM SMMU에서는 이것이 StreamID에 해당하며, Stream Table Entry(STE)를 인덱싱하여 디바이스/VM별 변환 설정을 찾는다."
