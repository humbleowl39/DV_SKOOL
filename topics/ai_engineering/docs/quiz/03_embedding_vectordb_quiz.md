## Q1. (Remember)

FAISS 의 대표 인덱스 3종류는?

??? answer "정답 / 해설"
    FAISS의 대표 인덱스 세 종류는 **IVF**, **HNSW**, **PQ** 이며, 각각 속도·메모리·정확도의 다른 트레이드오프를 선택한다.

    **IVF(Inverted File)**는 먼저 k-means로 전체 벡터를 클러스터로 나누고, 검색 시 질의와 가까운 클러스터만 스캔한다. 전수 탐색 대비 속도가 빠르고 메모리 효율도 좋지만, nprobe 파라미터(몇 개 클러스터를 볼 것인지)로 정확도와 속도를 직접 조정해야 한다. **HNSW**는 그래프 기반으로, 계층적 탐색 덕분에 가장 빠른 검색 속도를 제공하지만 그래프 구조를 저장해야 해서 메모리 사용량이 크다. **PQ(Product Quantization)**는 벡터를 여러 sub-vector로 쪼개 각각을 별도 코드북으로 양자화해 메모리를 대폭 줄이지만 정확도 손실이 있다. 실무에서는 IVF와 PQ를 합친 IVF+PQ 조합이 메모리와 속도의 균형을 잡는 표준으로 널리 쓰인다.

## Q2. (Understand)

Cosine similarity 가 dot product 가 아닌 정규화된 형태로 자주 쓰이는 이유는?

??? answer "정답 / 해설"
    dot product는 두 벡터의 방향뿐 아니라 **크기(노름)** 도 점수에 반영된다. 임베딩 벡터의 크기는 문서 길이나 어휘 빈도 같은 의미와 무관한 요소에 따라 달라질 수 있는데, 이 경우 긴 문서일수록 유사도 점수가 인위적으로 높아지는 편향이 생긴다.

    Cosine similarity는 두 벡터를 각각 정규화한 뒤 내적을 계산하므로, **방향(의미적 유사성)** 만 비교하고 크기 차이는 제거한다. 따라서 문서 길이가 달라도 의미적 유사성을 공정하게 비교할 수 있다. 다만 최신 임베딩 모델 대부분은 출력 벡터를 unit-norm으로 정규화하므로, 그 경우에는 dot product와 cosine similarity가 수학적으로 동일하다.

## Q3. (Apply)

사내 코드 100k 파일을 FAISS index 로 만들 때 chunk 단위와 chunk size 를 어떻게 정할 것인가?

??? answer "정답 / 해설"
    코드 retrieval의 품질은 chunk 설계가 결정한다. 단순히 "몇 토큰씩 자를 것인가"가 아니라 **의미 단위를 기준으로 자르는 것** 이 핵심이다.

    코드에서는 함수, 클래스, SystemVerilog module처럼 기능 단위로 자르는 것이 임의의 N-토큰 분할보다 retrieval 품질이 훨씬 좋다. 관련 코드가 한 chunk 안에 온전히 들어가기 때문이다. 크기는 256~512 토큰이 권장 범위인데, 너무 짧으면 질의에 답하기 위한 컨텍스트가 부족하고, 너무 길면 임베딩 벡터가 chunk 전체를 평균화해 버려 특정 내용을 정확히 표현하지 못한다. 함수 경계가 항상 정확히 맞지 않는 경우를 대비해 50~100 토큰의 sliding overlap을 두면 경계에서의 정보 손실을 막을 수 있다. 마지막으로 파일 경로, 함수명, 라인 번호를 메타데이터로 저장해두면 re-ranker나 사람이 결과를 검토할 때 즉시 출처를 확인할 수 있다.

## Q4. (Analyze)

10M 벡터에서 latency 를 1ms 이하로 유지해야 한다. IVF · HNSW · PQ 의 trade-off 를 분석하라.

??? answer "정답 / 해설"
    1ms 이하 latency와 10M 벡터라는 두 조건을 동시에 만족하려면 인덱스 선택이 중요하다.

    **HNSW**는 그래프 기반 탐색 덕분에 1ms 이하가 가장 쉽게 달성 가능하지만, 10M 벡터에서 그래프 구조 자체가 벡터 크기의 1.5~2배 메모리를 차지한다. **IVF**는 nprobe 값을 올리면 정확도가 오르고 낮추면 속도가 오르는 명확한 knob이 있어 튜닝이 쉬우며, 메모리 효율도 좋다. 그러나 클러스터 크기와 nprobe 조합이 맞지 않으면 1ms 보장이 어렵다. **PQ**는 벡터를 10배 이상 압축해 메모리를 획기적으로 줄이지만, 압축 과정에서 정보 손실이 발생해 recall이 낮아질 수 있다.

    실무에서 10M 스케일에서는 **IVF+PQ** (메모리 절약 우선) 또는 **IVF+HNSW** (정확도·속도 우선) 조합이 표준이며, 단일 인덱스로 모든 제약을 만족하기 어렵기 때문에 조합이 필요하다.

## Q5. (Evaluate)

자기 도메인의 retrieval 품질을 평가할 때 권장하는 절차는?

??? answer "정답 / 해설"
    자기 도메인의 retrieval 품질을 평가하려면 "모델 벤치마크 점수"가 아닌 **내 데이터·내 질의에 대한 직접 측정** 이 필요하다.

    먼저 사람이 직접 query와 relevant 문서를 라벨링한 gold set 50~200개를 만든다. 이것이 가장 신뢰할 수 있는 기준이다. 그 다음 MRR, Recall@k, nDCG 같은 검색 품질 지표로 숫자를 낸다. 지표만으로는 어떤 모델이 좋은지 알 수 없으므로, 임베딩 모델 후보를 2~3개 동시에 A/B 비교한다. 숫자 외에 **에러 분석**이 중요한데, 약어나 코드 식별자처럼 특정 패턴의 query에서 dense 검색이 실패하는 경우를 발견하면 BM25 같은 sparse 검색과 hybrid를 적용할 근거가 된다. 서비스가 운영 중이라면 사용자 클릭이나 체류 시간 같은 implicit signal을 retrieval 품질의 proxy로 상시 추적해 점수와 실제 사용자 만족이 연동되는지 검증한다.
