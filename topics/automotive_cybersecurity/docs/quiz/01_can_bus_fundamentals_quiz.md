## Q1. (Remember)

CAN Bus 는 어느 레이어 모델에서 어떤 두 계층을 정의하는가?

??? answer "정답 / 해설"
    OSI 모델의 **데이터 링크 계층(L2)** 과 **물리 계층(L1)** 을 정의한다. 상위 계층(전송, 응용) 은 별도 프로토콜(UDS, OBD-II 등) 이 담당.

    이것이 정답인 이유는 CAN 프로토콜이 처음 설계될 때부터 "버스 접근 중재와 전기 신호 전달"이라는 두 가지 역할만을 목표로 했기 때문입니다. L1(물리 계층)은 Dominant/Recessive 전압 레벨과 비트 타이밍을 정의하고, L2(데이터 링크 계층)는 프레임 구조·CRC·중재 규칙을 정의합니다. 보안 인증이나 세션 관리 같은 상위 기능이 CAN 자체에 없는 것은 설계 범위 밖이기 때문이며, 이것이 바로 SecOC와 UDS가 별도로 존재해야 하는 근본 이유입니다.

## Q2. (Understand)

CAN 의 비파괴적 비트별 중재(non-destructive bitwise arbitration) 방식이 어떻게 동작하며, 왜 "낮은 ID 가 우선"이 되는가?

??? answer "정답 / 해설"
    각 노드는 ID 를 비트 단위로 송출하며, **dominant(0)** 이 **recessive(1)** 을 덮어쓴다. 0이 더 우세하므로 ID 의 상위 비트에 0이 더 많은(즉 수치가 작은) 메시지가 충돌 시 살아남고, 진 쪽은 자동으로 전송을 멈춘다. 충돌해도 데이터가 깨지지 않는다는 점에서 "비파괴적" 이라 불린다.

    이 메커니즘이 "낮은 ID = 높은 우선순위"인 까닭은 전기적 원리에서 출발합니다. CANH·CANL 차동 버스에서 dominant 상태(0)는 드라이버가 전압 차를 만드는 것이고 recessive 상태(1)는 드라이버가 수동인 상태여서, 여러 노드가 동시에 쓸 때 0을 보낸 쪽이 항상 이깁니다. 따라서 ID 앞 비트에 0이 더 많이 나오는 — 즉 수치가 작은 — 메시지가 매번 중재에서 이기게 됩니다. "비파괴적"이라는 말은 중재에서 진 노드의 데이터가 손상되지 않고 그저 전송을 재시도한다는 뜻으로, 이더넷의 CSMA/CD 방식(충돌 후 폐기)과 구분되는 CAN의 핵심 특성입니다.

## Q3. (Apply)

회사가 OBD-II 포트를 항상 살아 있도록 두는 양산 차량에 대한 보안 검토를 의뢰받았다. 가장 먼저 확인해야 할 두 가지 위협 시나리오는?

??? answer "정답 / 해설"
    1. **Diag 인증 누락** — UDS Security Access(0x27) 가 없거나 약한 키로 보호되어 외부에서 진단 모드 진입이 가능한가?
    2. **버스 격리 부재** — OBD-II 가 Powertrain CAN 까지 직접 연결되어 있어, 외부에서 ABS/Engine 메시지를 그대로 주입할 수 있는가? Secure Gateway 를 사이에 두는지 확인.

    이 두 가지가 가장 먼저 확인해야 할 이유는 OBD-II 포트 자체가 "설계상 열려 있는 물리 접점"이기 때문입니다. 포트를 막을 수 없다면 그 뒤에 인증 관문(UDS 0x27)이 있어야 하고, 그 관문조차 뚫렸을 때 Powertrain CAN과의 격리(Gateway)가 두 번째 방어선이 됩니다. Diag 인증만 있고 격리가 없으면 인증 우회만으로 제동·조향 메시지 주입이 가능해지고, 반대로 격리만 있고 인증이 없으면 진단 기능 전체가 노출됩니다. 두 통제를 동시에 확인해야 "Defense-in-Depth" 관점의 최소 요건을 충족합니다.

## Q4. (Analyze)

CAN 의 "broadcast + 무인증" 특성이 만들어내는 4가지 구조적 약점을 분류해 설명하라.

??? answer "정답 / 해설"
    1. **Spoofing** — 누가 보낸 메시지인지 검증할 방법이 없다 → 임의 노드가 Engine ECU 를 가장 가능.
    2. **Replay** — 카운터/타임스탬프가 없으므로 정상 메시지를 녹음 후 재전송 가능.
    3. **DoS by Flooding** — 낮은 ID 메시지를 1ms 주기로 보내면 정상 메시지가 영원히 중재에서 진다.
    4. **Eavesdropping** — 모든 메시지가 broadcast 되어 같은 버스 위 모든 노드에서 청취 가능.

    이 네 가지 모두 **프로토콜 자체로는 해결 불가** — SecOC + Gateway + IDS 같은 상위 계층 통제가 필요하다.

    이 약점들이 모두 "구조적"인 이유는 CAN이 발신자 식별 정보를 프레임에 포함하지 않고 카운터·타임스탬프도 없으며 모든 노드가 동일한 물리 매체를 공유하기 때문입니다. Spoofing과 Replay는 "인증 없음"이라는 같은 뿌리에서 나오고, DoS는 중재 우선순위 설계를 역이용하며, Eavesdropping은 broadcast 매체의 불가피한 부작용입니다. 따라서 CAN 스펙을 바꾸지 않는 한 이 네 가지는 프로토콜 수준에서 봉쇄할 수 없고, 반드시 SecOC(인증/Replay 방어), Gateway(격리/필터링), IDS(Flooding 탐지)라는 상위 계층을 함께 배치해야 합니다.

## Q5. (Evaluate)

CAN-FD 가 SecOC 를 더 실용적으로 만드는 결정적 변화 두 가지는?

??? answer "정답 / 해설"
    1. **Payload 64B 까지 확장** — 8B payload 시 MAC 16B 를 동봉하면 데이터가 거의 남지 않는다. 64B 가 되면 freshness + truncated MAC 을 충분히 실을 수 있다.
    2. **Data-phase 비트레이트 분리(최대 5~8 Mbps)** — MAC 부착으로 인해 늘어난 메시지 길이의 전송 시간을 만회. 결국 SecOC overhead 를 감춘다.

    둘이 합쳐져야 SecOC 가 "버스 대역 다 잡아먹는 부담" 이 아니라 일상적으로 켤 수 있는 기능이 된다.

    Payload 확장이 결정적인 이유는 수학적으로 단순합니다. 기존 CAN의 최대 8바이트 중 MAC 4바이트와 Freshness 4바이트를 할당하면 실제 데이터에 쓸 수 있는 공간이 0바이트가 됩니다. 64바이트 payload에서는 MAC 16바이트와 Freshness 8바이트를 써도 40바이트 이상이 남으므로, 보안 헤더가 데이터를 잠식하는 문제가 사라집니다. 비트레이트 향상은 이 늘어난 프레임 길이를 실시간 제어 주기(1~10ms) 안에 전송할 수 있게 만들어, 두 변화가 함께 작용해야 비로소 SecOC가 실전에서 쓸 수 있는 기능이 됩니다.
