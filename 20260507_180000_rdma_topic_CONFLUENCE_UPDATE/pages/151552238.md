# Automatic Path Migration

<!-- id=151552238 parent=32211017 -->

Automatic Path Migration (APM) is an optional facility that enables connection recovery (similar to multipath) in the case of failures. APM is available for reliable connected, unreliable connected and XRC QP services.

- Here, the term “path” includes “port”, which means it is required to have multiple ports to support APM

- You can also find relevant comments in this link ([software example for APM](https://docs.nvidia.com/networking/display/rdmaawareprogrammingv17/programming%2Bexamples%2Busing%2Bibv%2Bverbs#src-34256583_safe-id-UHJvZ3JhbW1pbmdFeGFtcGxlc1VzaW5nSUJWVmVyYnMtQXV0b21hdGljUGF0aE1pZ3JhdGlvbihBUE0p))

To do so, IBTA spec introduces three new states for migration state (which is seperate states from connection state - remember INIT, RESET, RTR, RTS, etc? )

- Migrated / Rearm / Armed

</div>

- Migrated

  - This state describes either “initial” state or “is newly migrated”

  - If one of endpoints does not support (or does not want to support) APM, the state remains “Migrated”

  - In this state, outgoing packet should set MigReq field in BTH header.

  - FYI: You can easily see CX is sending packets with MigReq field set for every packet!

- REARM

  - Indicates the alternate path is loaded

  - The only example I can find which set QP migration state to REARM is RDMA-CM

    - In MAD message, there is field named “alternate path, failover accepted, etc..” - it seems using this field, each endpoint can set alternate path

  - Once each endpoints received REQ / REP MAD, rdma-cm stack (linux kernel) transitions QP state from MIGRATED to REARM by calling ibv_modify_qp verb.

    - Only verb consumers are allowed to transition state to REARM(spec)

- ARMED

  - It seems, this state gurantees “each side” has loaded alternate path, so that one of endpoints can trigger path migration anytime from this point.

  - **Only hardware can transition migration state from REARM to ARMED** (modify_qp verbs are prohibited from this transition)

**Question: When to transition states from ARMED to MIGRATED? (This is out-of-scope, actually)**

- Manual intervention (load balancing)

- Excessive retry occured in primary path (network failure)

Once one of endpoints decided to use alternate path instead of primary path, it should set MigReq bit on outgoing packet, so that opposite side can recognize the migration event

- Each side shall notify the host by raising Path Migrated affiliated asynchronous event for that QP

- **After migration, RNIC may refuse to accept incoming request or response packet arriving from the original path!**

Q**uestion: How can we re-enable path migration?**

- There are specific MAD packets for this purpose - LAP (Load Alternate Path) and APR (Alternate Path Response).

  - From the specification, only the endpoint that acted as a client in initial RDMA CM stage can send LAP packets.

- **However, their relevant codes are deprecated in upstream kernel** **And I cannot find any example that actually re-enables path migration**

- In hardware perspective, I think yes.

- However, relevant SW support is insufficient;

  - Upstream kernel is preventing RoCE to enable APM

      - We need to wait for patches (or, may build our own kernel IB stack? )

### Scenario: Cable Unplugging and Reconnecting

- If a cable connected to the primary port of an InfiniBand device is unplugged, the hardware will detect that it can no longer successfully send packets via this path. After exhausting retries, the QP will automatically migrate to the alternate path, ensuring that communication continues without interruption.

- When the cable is plugged back in, the system does not automatically revert to the primary path unless configured to do so. The alternate path will continue to be used until either the system is manually instructed to switch back or another failure triggers a migration.
