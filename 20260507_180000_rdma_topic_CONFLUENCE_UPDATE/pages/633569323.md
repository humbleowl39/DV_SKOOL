### Coverage 작성 policy 초안

1.  Path 분기하는 경우

    1.  If statement

        1.  If 내로 들어가기 위한 조건 cover

        2.  ***Q. If문에 들어가지 않기 위한 조건 cover?***

            1.  path에 분기가 생길 때 양쪽 path를 모두 cover해야하는지 or 한쪽만 cover해도 되는지?

    2.  Switch statement

        1.  switch 내의 각 case cover

        2.  switch에서 default case cover (invallid한 경우 assert)

    3.  삼항연산자

        1.  발생 가능한 모든 case를 구분하여 cover (collectively exhaustive)

    4.  다른 모듈에서 받은 input으로 분기가 나뉘는 경우 (e.g., AXI register input)

        1.  1안) Address space 상의 모든 경우의 수 cover

        2.  2안) 모듈이 의도한 address space 상의 경우의 수 cover

        3.  **Q. Invalid input (AXI reg write)은 어떻게 처리? (cover or assert)**

            1.  out of valid range input 모두 cover?

        4.  ***Q. 얼마나 세부적으로 cover case를 나눌 것인가?..***

            1.  발생 가능한 가장 작은 단위?

            2.  기능 단위

            3.  **알잘딱? → 괜찮을까요**

    5.  그외?

2.  Assert 넣는 기준

    1.  다른 모듈/thread에서 온 invalid input/output

    2.  혹시 모를 숨어있는 logic error 탐색용 code

    3.  TOE

        1.  input: 이 input이 들어오면 이 모듈은 망가진다

        2.  복잡한 FSM에서 이게 깨지면 로직에 문제가 발생한다

3.  comment 작성 요령

    1.  assert / cover 넣는 case별로 얼마나 자세하게 적을 건지? 너무 당연한 내용은 그냥 넘길 것인가?

        1.  assert 는 달자, cover는 필요한 경우에 달아두자

        2.  **TOE에 assert + comment 버전이 있었다 → policy가 바뀐 것인지 확인 필요**

    2.  (100% 통용되는 원칙은 만들 수 없겠지만) 대략적인 가이드라인이 있으면 도움이 될 듯함
