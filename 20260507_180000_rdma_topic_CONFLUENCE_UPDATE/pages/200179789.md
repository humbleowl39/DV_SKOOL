# Background: MPI Operations

<!-- id=200179789 parent=200179723 -->

MPI provides a comprehensive set of functions for inter-process communication and coordination:

- **Point-to-Point Operations:** Enable direct communication between pairs of processes. Key functions include:

  - `MPI_Send` and `MPI_Recv`: Basic send and receive operations.

  - `MPI_Isend` and `MPI_Irecv`: Non-blocking variants allowing computation overlap.

- **Collective Operations:** Involve all processes within a communicator, facilitating group communication patterns. Examples are:

  - `MPI_Bcast`: Broadcasts data from one process to all others.

  - `MPI_Reduce`: Aggregates data from all processes to a single process using a specified operation (e.g., sum, max).

  - `MPI_Allgather`: Gathers data from all processes and distributes the combined data to all.

In MPI's two-sided communication model, matching operations ensure that messages are correctly paired between senders and receivers:

- **Tags:** User-defined integers assigned to messages, aiding in distinguishing between different types of messages.

- **Source and Destination Ranks:** Identifiers for processes within a communicator, specifying the sender and receiver.

A receive operation matches a send operation when the communicator, source rank, destination rank, and tag align. This matching mechanism is crucial for the correct routing of messages in complex communication patterns.

## **Non-Matching Operations**

Non-matching operations refer to scenarios where receive operations are designed to accept messages without specifying certain attributes, providing flexibility:

- **Wildcards:**

  - `MPI_ANY_SOURCE`: Allows a process to receive messages from any source.

  - `MPI_ANY_TAG`: Enables reception of messages with any tag.

These wildcards are useful in situations where the receiver does not need to know the exact source or tag of incoming messages, facilitating more dynamic communication patterns.

## **Remote Memory Access (RMA) Operations**

RMA operations, introduced in MPI-2 and enhanced in MPI-3, support one-sided communication, allowing a process to directly access the memory of another process without explicit synchronization:

- **Key Functions:**

  - `MPI_Put`: Writes data from the origin process to a target process's memory.

  - `MPI_Get`: Reads data from a target process's memory to the origin process.

  - `MPI_Accumulate`: Performs a reduction operation on data in the target process's memory.

- **Windows:** Regions of memory that a process exposes for RMA operations, created using functions like `MPI_Win_create`.

- **Synchronization Mechanisms:** Ensure data consistency during RMA operations, including:

  - `MPI_Win_fence`: Provides collective synchronization.

  - `MPI_Win_lock` and `MPI_Win_unlock`: Offer passive synchronization for individual processes.

RMA operations can reduce synchronization overhead and improve performance for certain communication patterns by allowing direct memory access between processes.

## **Rendezvous and Eager Protocol**

**Eager Protocol**:

In the eager protocol, the sender transmits the entire message to the receiver without awaiting confirmation of the receiver's readiness. This approach is typically employed for small to medium-sized messages, where the overhead of immediate transmission is minimal. The receiver must have sufficient buffering capacity to store incoming messages until they are processed.

**Rendezvous Protocol**:

For larger messages, the rendezvous protocol is utilized to prevent overwhelming the receiver's buffer. The sender initiates communication by sending a small control message, often referred to as a "request-to-send" (RTS), to inform the receiver of its intent to send data. Upon receiving the RTS, the receiver responds with a "clear-to-send" (CTS) message when it is prepared to accept the data. Only after receiving the CTS does the sender transmit the actual data.

**Eager Portion in Rendezvous Protocol**:

Some implementations of the rendezvous protocol incorporate an "eager portion," where a small segment of the data is sent immediately along with the RTS message. This strategy allows the receiver to begin processing part of the data before the entire message is received, potentially improving performance. The remaining data is transmitted after the receiver sends the CTS, ensuring that the receiver is ready to handle the full message.
