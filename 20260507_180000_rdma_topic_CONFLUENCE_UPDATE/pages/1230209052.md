# HLS Timing Analysis: completer_frontend & responder_frontend

**Date:** 2026-04-05  
**Author:** Yoonhyuk Choi (assisted by Claude Code)  
**Context:** Combined Vivado implementation shows excessive CLB regslice consumption. WNS regression \> 3ns observed. Investigation compares before & after.

## 1. Executive Summary

Both `completer_frontend` and `responder_frontend` show severe HLS-level timing violations that directly cause CLB regslice blowup in combined implementation. The root cause is **FSM state explosion** in multiple threads, compounded by new ConnMux arbiter instances.

|                              |                           |                             |
|------------------------------|---------------------------|-----------------------------|
| Metric                       | completer_frontend        | responder_frontend          |
| Clock period                 | 4.0 ns                    | 4.0 ns                      |
| HLS critical path slack      | -1.47 ns                  | -1.47 ns                    |
| Worst reg-to-reg slack       | -1.36 ns                  | **-3.33 ns**                |
| Largest FSM (AND-gate terms) | and#1213 (th_dma_write)   | **and#1649 (th_psn_check)** |
| Most violated thread         | th_swq_cmd / th_dma_write | **th_ack (56 violations)**  |

## 2. Why Standalone WNS Causes Combined CLB Regslice Blowup

At \>3 ns WNS (nearly 2x the clock budget), Vivado cannot close timing on the module at all. The cascade:

1.  Vivado spreads the module's logic across more CLB columns seeking shorter routes

2.  Spread-out placement steals routing tracks from neighboring RDMA modules

3.  Neighbors fail timing → Vivado inserts register slices for retiming/replication

4.  More regslices → more CLB usage → further routing congestion

At \>3 ns scale, the standalone module is the **direct cause** — not a secondary contributor.

### 3.1 Key Changes Causing FSM Bloat

- **th_dma_write (and#1213):** New multi-phase MR pre-validation with two `#pragma hls_pipeline_init_interval 1` loops (Phase 1: push AWDs, Phase 2: pop RDs). Variable-length while-loop for SGE iteration replacing fixed for-loop. `check_local_mr_violation` ccore + error handling branches.

- **th_swq_cmd (and#511):** Added `flag.err_type != NO_ERROR` guards, `mb_swq_cmd` construction replacing `mb_mem_cmd`, added branching around MODIFY path.

- **ConnMux (x2):** `A_comp_err_mux` and `A_qp_state_err_req_mux` replaced software-polled `th_err_mux`. Hardware round-robin arbiters violate at -0.825 ns.

- **Pipeline width growth:** `completer_essentials_trim` +34 bits (global_lkey, send_cqn, interrupt_en), `tx_info_comp_trim` +16 bits (wr_id), `completer_flags.err_type` 1→3 bits. Flows through all 7 pipeline concat structs.

### 3.2 Fix Applied: Option A — Error Dispatch Mux Factoring

Two 6-way switch blocks in `th_comp` (non-large-psn error at ~line 863, large-psn error at ~line 1202) each called `send_err_completion_rdma` with different constants. HLS saw 6 call sites per block → wide per-CQE-field mux tree.

**Fix:** Collapsed each switch into a 3-bit → 8-bit lookup (`wc_flag`, `wc_status`) + single `send_err_completion_rdma` call. Reduces rank \#8 violation. Does not address ranks \#1–#7.

### 4.1 Critical Path \#1 Deep Dive: th_psn_check at -3.33 ns

The worst path in either module. 7.33 ns combinational chain in one clock cycle:

` m_read_resource_rw_port.rd.Pop() → mux#2 (186-bit data select) → psn_compare (24-bit add + slice + 7-way AND) → operator< (comparator) → and#228 → or#365 → nor#256 → nor#255 → mr_valid:and#6 (MR validity check) → stage_0:mx1 (pipeline stage mux) → if:if:if:mux → or#4 → for:and → nor → mux1h (3-way one-hot mux) → flen_shf stall register `

This path worsened because of the additions: (1) `qp_rq_ready[512]` bitmap branching, (2) `is_invalid_request` / `is_mr_invalid` error propagation conditions, (3) more conditional logic between Pop and stall decision.

### 4.2 Key Changes Causing Timing Regression

- **th_psn_check (and#1649):** New `qp_rq_ready[MAX_QP_NUM]` bitmap (512 regs + decode), `is_invalid_request` handling paths, modified MR invalidity propagation through pipeline (was dropped early in the previous version)

- **th_ack (and#1425):** FSM state explosion from read-response generation. New error-path filtering (`is_invalid_request`, `is_unrecoverable_err`) adds branches.

- **th_prot_check:** Full Send lkey/PD/access/range checking implemented (was stub in the previous version). Shared comparator optimization helps area but adds branch depth.

- **th_dma_write:** 2nd-SGE MR protection check in `scatter_send_desc` (returns 0xFF on violation). New error handling branches.

- **ConnMux (x2):** Same pattern as completer_frontend — replaced polling-based `th_err_mux` with hardware `A_resp_err_mux` + `A_qp_state_err_req_mux`.

- **Pipeline struct note:** `responder_essentials` and `responder_flags` are unchanged (unlike completer_frontend). However, `mb_qp_resp_trim_rx.wqe` changed from `NVUINTW(512)` to `NVUINTW(PAYLOAD_IN_BIT)` — doubles to 1024 bits for 200G flavors.

## 5. Recommended Fixes (Both Modules)

### 5.1 responder_frontend — Priority 1: Break th_psn_check critical path

Insert `wait()` between `m_read_resource_rw_port.rd.Pop()` and the psn_compare / branch logic. Cuts the 7.33 ns path roughly in half. Costs 1 cycle latency. Targets the -3.33 ns violation.

### 5.2 responder_frontend — Priority 2: Split th_psn_check

Split into two threads (packet classification + resource allocation) connected by Combinational channel. Reduces the 1649-state FSM into two smaller FSMs.

### 5.3 Both modules — Priority 3: Merge ConnMux instances

`A_qp_state_err_req_mux` carries only `NVUINTW(9)` (qp_num). `th_qp_state_err_write` can extract `local_qid` from the CQE already flowing through `A_comp_err_mux` / `A_resp_err_mux`. Eliminates 4 channels + 1 ConnMux per module (saves -1.357 ns violations).

### 5.4 completer_frontend — Priority 4: Extract MR validation from th_dma_write

Move multi-phase MR pre-validation (Phase 1 push, Phase 2 pop) into separate SC_MODULE or thread. Isolates pipeline states from th_dma_write's already-massive FSM (and#1213).

### 5.5 completer_frontend — Priority 5: Trim pipeline struct widths

Move `global_lkey` (24b), `send_cqn` (9b), `interrupt_en` (1b) from `completer_essentials_trim` (flows through all stages) to `completer_stateupdate_concat` only (needed only by th_dma_write). Saves 34 bits x 5 pipeline stages.

### 5.6 Both modules — Priority 6: Replace qp_sq_ready / qp_rq_ready bitmaps

512-entry 1-bit arrays with full address decode fragment placement. Use existing QP state memory read instead, or reduce to smaller hash-indexed shadow bitmap.
