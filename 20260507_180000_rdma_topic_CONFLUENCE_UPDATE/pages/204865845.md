# CORN: Cloud-Optimized RDMA Networking

<!-- id=204865845 parent=76644711 -->

CORN is another congestion control algorithm implementation from Samsung SDS

Objective:

- Build a congestion control algorithm w/o relying on PFC, in datacenter!

  - Datacenter topology is quite limited

Key features:

- Delay-based congestion detection

  - Increase of RTT means there are congestion in the network

    - **RTT requires ACK, so CORN assumes there are no cumulative ACK! (= 1 ACK per 1 packet)**

      - **<span style="color: rgb(255,86,48);">I think, this will incur performance overhead</span>**

    - When it detects the increase of RTT by comparing the previous RTT, it reduces the data rate until the RTT value before increasing RTT is reached

    - **Calculate BDP and use this as a congestion metric**

- **Window-based congestion control**

  - Just like TCP or UEC

  - To fine-tune the number of packets to be transmitted instead of adjusting the data rate

  - Each window slot corresponds to a specific PSN

  - Counts \# of inflight packets

- Fairness

  - Calculate optimal BDP - assuming there is a single flow

    - In real case, actual BDP will be lower than this.

  - CORN reduces the datarate of flows that contribute heavily to network congestion

    - In other words, flow with largest BDP will be the target

Congestion control algorithm

- Phase 1: Initial stage

  - Starting from initial window size, progressively increase the window size by multiplying Increase Factor (IF)

  - If RTT increases, it means there is a congestion so move to next phase

- Phase 2: Window size reduction

  - When amount of inflight packets exceeds BDP

    - Cut down window size using Decrease Factor (DF)

  - When packet drop occurs or network congestion is unresolved (I don’t understand this case)

    - Halve the window size

- Phase 3: When congestion is resolved

  - When RTT reaches previous RTT (that is prior to congestion), increase window size using different Increase Factor (IF3)

  
They also claim they use SACK to resolve bad goodput due to Go-back-N. For example, when receiver detects holes in the packet sequence, it sends SACK containing the lost packets' sequence numbers to sender. When sender receives SACK, it retransmits the packet associated with the lost sequence numbers.
