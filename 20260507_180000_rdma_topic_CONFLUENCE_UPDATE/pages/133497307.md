### frontend_attr_saver

- qp destroy는 write 후 read를 하는데 (write 된 것을 보장하기 위해서) create 는 그런 것을 하고 있지 않음

### info_arb

- Input port specification

  - s_cmd_port_0 : requester_frontend와 completer_frontend 겸용 (mem_cmd_switch로 mux 됨)

    - requester_frontend가 여기로 cmd 보내는 경우

      - 1\. 새로 생성된 RC (UD, GSI는 포함X) type의 WR → **WRITE**

    - completer_frontend가 여기로 cmd 보내는 경우

      - 1\. ACK coalescing 하는 중에 implicit하게 완료 처리된 WR → **DELETE**

      - 2\. 도착한 packet이 valid하고, 현재 WR에 대해 받아야 하는 마지막 packet → **DELETE**

      - 3\. 도착한 packet이 valid하고, 현재 WR에 대해 받아야 할 packet이 더 남아있음 → **MODIFY** (PSN만 수정, 나머지 그대로)

  - s_cmd_port_1 : completer_retry 전용

    - 1\. RNR 받았을 경우 → (MEM\_)**RNR**

    - 2\. RNR이 아닌 retry를 받은 상황에서, unrecoverable error 또는 이미 NAK이 나간 경우가 아님 → **GET_ALL**

  - s_cmd_port_2 : frontend_attr_saver 전용

    - 1\. qp attribute save 요청 들어옴 → **INIT** (해당 QP 대상으로)

    - 2\. qp reset 요청 들어옴 → **FLUSH**

  - s_cmd_port_3, 4 : completer_frontend read 전용

    - port 3 : completer_frontend에서 QP num + cur tail ptr로 WR 찾아서 최초로 읽어올 때 → **READ INIT**

    - port 4 : th_comp에서 ACK coalescing하는 과정 중에 다음 WR도 읽어와야 할 때 → **READ**

  - info_arb 안에서 port 0, 1, 2가 공통으로 th_cmd_arb를 거쳐서 같은 path로 처리됨 / port 3, 4는 completer frontend에 대한 read-only port들이라서 독립적으로 처리됨

- input port, opcode → SWQ operation output으로의 translation

  - requester frontend write, completer frontend modify / delete (input port 0)

    - QP num, head (write) or tail (modify, delete) ptr로 indexing → **SWQ_WRITE/MODIFY/DELETE**로 해당하는 entry를 수정

    - SWQ에서 나오는 return 없음

  - completer retry RNR / GET_ALL (input port 1)

    - RNR : retry timer에 rnr restart만 trigger 하고 swq에 아무것도 보내지 않음

    - GET_ALL (RNR 아닌 NAK일때) : th_retry로 QP num을 넘김 + head ~ tail 전체 WR에 대해 timer reset

      - th_retry : head~tail 위치로 read_left 정한 다음 그 개수만큼 swq 읽어서 retry_trans로 보냄

  - completer frontend input (input port 3, 4)

    - **SWQ_READ**로 읽어온 다음 completer_frontend에 읽어온 값을 return

    - 항상 completer frontend가 관리하고 있는 tail 위치를 읽어옴 (=가장 오래된 in-flight WR)

- 개선점 : head/tail boundary 밖에서의 이상한 operation 이 발생하면 error cmpt 를 넣는 것이 좋을 듯

### inflight_request_controller

- 각 변수의 역할

  - w_num_total_insert : 들어온 모든 req 개수의 총합

  - w_num_info_delete : SWQ에서 지금까지 entry를 delete 한 횟수

  - w_num_req_delete : requester_frontend에서 WR을 자체적으로 drop한 횟수 (invalid QP 등등)

  - MAX_INFLIGHT : SWQ의 WR 저장 용량

- retry handling을 위해 전체 in-flight WR의 개수를 항상 SWQ의 max entry 개수 이하로 유지해야 함 → 본 - module이 현재 SWQ의 남은 entry 수를 계산 → 상태에 따라 들어오는 WR을 block (SWQ가 꽉 찬 경우, 빈 공간이 생길때까지) / bypass (otherwise)
