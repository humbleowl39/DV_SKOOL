# [completer_frontend] 1K MTU RDMA Read >4KB throughput 저하 분석 및 수정

<!-- id=1276379157 parent=1228931074 -->

## 개요

Commit `9f37f44dd` ("completer_frontend th_dma_write split")에서 `completer_frontend` 모듈의 MR read 구조를 변경한 후, 1K MTU에서 payload \>4KB인 RDMA Read의 throughput이 거의 0에 수렴하는 성능 저하가 발생함. 4K MTU에서는 1MB payload까지 line-rate 정상 동작 확인됨.

본 문서는 root cause 분석, HLS report를 통한 검증, 그리고 수정 방안을 정리함.

## Root Cause

### 핵심 수치: th_dma_write II vs T_arrival

200G line-rate / 250 MHz / 1K MTU 조건에서:

- **T_arrival** (packet 간 도착 간격) ≈ **11 cycles** (1090 bytes × 8 / 200 Gbps / 4 ns)
- **th_dma_write throughput (broken)** = **12 cycles** (Catapult HLS report 확인)
- **th_dma_write throughput (original)** = **11 cycles**

Broken 버전에서 th_dma_write의 II가 T_arrival을 1 cycle 초과함. 이 1 cycle의 deficit는 pipeline 구조상 **복구가 불가능**하며, 지속적으로 누적되어 input buffer overflow → flow control oscillation → throughput 붕괴로 이어짐.

### +1 cycle의 원인: pipelined for-loop FSM state

Broken 버전의 `th_dma_write`에서 transition MR을 pop하기 위해 pipelined for-loop을 사용:

이 for-loop은 HLS에 의해 **dedicated FSM state** (`while:else:else#2:if:else:for`)로 합성됨. Single-SGE (mr_transition_count == 0)인 경우에도 해당 state를 반드시 거쳐야 하므로 **매 packet마다 1 cycle 추가**.

반면 original 버전의 Phase 1/Phase 2 loop은 `if (!all_same_lkey)` block 안에 있어서, single-SGE path에서는 **해당 FSM state를 완전히 bypass** (0 cycle). 이것이 original = 11 cycles, broken = 12 cycles의 차이.

### Zero-margin pipeline의 불안정성

th_dma_write의 minimum II = 11 cycles인데, T_arrival도 ≈ 11 cycles로 margin이 0임. 이 조건에서:

- 어떤 perturbation (downstream backpressure, FIFO handshake latency 등)이 발생하면 deficit가 영구적으로 남음
- th_dma_write의 minimum II가 11이므로 어떤 packet도 11 cycles 미만으로 처리 불가 → **복구 메커니즘 없음**
- Deficit가 누적되면 input buffer (comp_header_fifo, depth=64) overflow → transport layer flow control 발동 → stop-and-go oscillation

4K MTU에서는 T_arrival ≈ 42 cycles로 margin이 ~30 cycles여서 어떤 perturbation도 흡수 가능, 따라서 성능 문제 없음.

≤4KB (≤4 packets at 1K MTU)에서는 stateupdate FIFO (depth=4)가 포화되지 않아 pipeline이 steady-state에 진입하지 않고 initial slack으로 처리되므로 문제 없음.

## 원래 Commit이 실패한 이유

### 의도: Multi-SGE path 최적화

Original 버전의 `th_dma_write`는 multi-SGE lkey transition 시:

1.  Phase 1: `m_mr_rw_port_scatter`로 AWD push (validation용)
2.  Phase 2: `m_mr_rw_port_scatter`에서 RD pop (validation)
3.  Scatter 중: `m_mr_rw_port_scatter.read()`로 MR 재조회 (5-7 cycle memory latency)

→ 동일 transition MR을 **2회 memory access**. Commit은 이를 `th_state_update`에서 pre-fetch하여 cached array로 전달하는 구조로 변경, scatter loop에서 memory access 제거 (0 cycle).

### 부작용: Single-SGE path 성능 저하

Multi-SGE 최적화를 위해 도입된 for-loop이 **single-SGE path에도 1 cycle overhead**를 부과함.

- Original: `if (!all_same_lkey) { Phase1; Phase2; }` → single-SGE시 if block 전체 bypass, 0 cycles
- Broken: `for (i = 0; i < MAX_SGE-1; i++) { if (i == count) break; Pop(); }` → single-SGE시에도 for-loop FSM state 진입, 1 cycle

### 추가 문제: Orphaned MR response (correctness bug)

`th_state_update`에서 AWD push를 SGE while-loop **이전**에 수행. Loop 도중 `is_mr_invalid`가 true로 전환되면:

- AWD는 이미 push됨 → MR response가 `axis_data_fifo_0`에 도달
- `th_dma_write`는 `flag.is_mr_invalid = true`를 보고 MR pop을 skip
- → orphaned response가 FIFO에 잔류, 이후 모든 packet의 MR 데이터가 1개씩 shift되어 **wrong MR data 사용**

이 bug는 multi-SGE + mixed valid/invalid MR 조건에서 발생하며, single-SGE perftest에서는 trigger되지 않음.

## 수정 내용

### 1. th_state_update: AWD push를 SGE loop 이후로 이동 (Option 1)

- SGE while-loop에서는 transition lkey만 기록 (blocking I/O 제거)
- Loop 종료 후, `!flag.is_mr_invalid` 조건 하에 모든 AWD push 실행
- → Orphan bug 구조적 제거 + SGE loop FSM state 감소 (4 → 3)
- → th_state_update throughput: 11 → 9 cycles (refill slack 증가)

### 2. th_dma_write: for-loop을 conditional로 교체

- `MAX_SGE = 2` 조건에서 transition은 최대 1회
- Pipelined for-loop (항상 1 FSM state) → `if (mr_transition_count > 0)` conditional (single-SGE시 0 FSM state)
- → th_dma_write throughput: 12 → 11 cycles (line-rate budget 복귀)

### 변경 파일

- `completer_frontend.cpp`: `th_state_update` (AWD push 위치 변경) + `th_dma_write` (for-loop → conditional)
- 기타 파일 변경 없음 (`completer_frontend.h`, `completer_frontend_def.h`, `scatter_read_resp_desc`, craw header 모두 그대로)

## 결론 및 교훈

1.  **200G / 250 MHz / 1K MTU 조건에서 th_dma_write II budget은 정확히 11 cycles**. 1 cycle이라도 초과하면 unrecoverable deficit → throughput 붕괴.
2.  **Pipelined for-loop은 iteration 0에서 break해도 1 FSM state를 소비**. 반면 if-guarded block은 조건 불충족 시 0 cycle로 bypass 가능. HLS에서 hot path의 loop 구조 선택 시 이 차이가 critical.
3.  **Cross-thread channel의 AWD push는 반드시 대응 credit (stateupdate) push와 동일 조건에서 수행**. 선행 push는 orphaned response → channel desync 위험.
4.  Multi-SGE 최적화가 single-SGE (common path) 성능을 저하시키지 않도록 **conditional bypass 패턴**을 사용해야 함.
