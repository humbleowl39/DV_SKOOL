**MPI architecture**

<div>

</div>

- *Open, Portable Access Layer (OPAL)*: OPAL is the bottom layer of Open MPI's abstractions. Its abstractions are focused on individual processes (versus parallel jobs). It provides utility and glue code such as generic linked lists, string manipulation, debugging controls, and other mundane—yet necessary—functionality. OPAL also provides Open MPI's core portability between different operating systems, such as discovering IP interfaces, sharing memory between processes on the same server, processor and memory affinity, high-precision timers, etc.

- *Open MPI Run-Time Environment (ORTE) (pronounced "or-tay")*: An MPI implementation must provide not only the required message passing API, but also an accompanying run-time system to launch, monitor, and kill parallel jobs. In Open MPI's case, a parallel job is comprised of one or more processes that may span multiple operating system instances, and are bound together to act as a single, cohesive unit. In simple environments with little or no distributed computational support, ORTE uses rsh or ssh to launch the individual processes in parallel jobs. More advanced, HPC-dedicated environments typically have schedulers and resource managers for fairly sharing computational resources between many users. Such environments usually provide specialized APIs to launch and regulate processes on compute servers. ORTE supports a wide variety of such managed environments, such as (but not limited to): Torque/PBS Pro, SLURM, Oracle Grid Engine, and LSF.

- *Open MPI (OMPI)*: The MPI layer is the highest abstraction layer, and is the only one exposed to applications. The MPI API is implemented in this layer, as are all the message passing semantics defined by the MPI standard. Since portability is a primary requirement, the MPI layer supports a wide variety of network types and underlying protocols. Some networks are similar in their underlying characteristics and abstractions; some are not.

**About MPI APIs**

MPI (Message Passing Interface) is for **interprocess communication**, where different threads have different address space. Interprocess communication consists of (1) synchronization, and (2) data movement from one’s address space to another’s.

MPI basics: send and receive

- Process 0 sends data, and Process 1 receives

  - How will data be described?

  - How will the receiver recognize/screen messages?

  - What will it mean for these operations to complete?

- Data is packed with“envelope”, which consist of some necessary data. This includes “tag” and “destination/source” information. So at the receiver side, it can filter all message except the one from expected sender whose tag matches the one specified in recv call. Message with different tag will be buffered until receiver is ready. → **This is “tag” matching**

- Advantage: Any change in the receiveing process’s memory is made with the receiver’s explicit participation. Communication and synchronization are combined.

- Two methods: Eager vs Rendezvous

  - Eager: Message sent assuming destination can store

    - Reduce syncrhonization

    - Requires significant bufferings - May require active involvement of CPU to drain network at receiver’s end → May introduce additional copy

  - Rendezvous: Message not sent until destination oks → First send envelope only, then data later after destination’s acknowledgement.

    - Introduce syncrhornization delay

    - More complex programming

    - Remove copy

One-sided comunication (put/get)

- Advantage: Decouple data movement with process synchronization → No need to be delayed at the sender side

- Remote memory read / write (in favor of RDMA)

- Bypass tag matching - effectively precomputed as part of remote offset

- Significantly faster than send/recv

- Only one process needs to explicitly participate.

  - Put: like eager / Get: like rendezvous

  - Can be implemented through RDMA write and RDMA read

- Accumulate, Compare and swap, fetch and op is also done in remotely accessible memory

- Basically non-blocking → Return does not guarantee anything.

**OpenMPI is flexible, modular**

We also call PML, BTL, … as “framework”. And “the methodology of implementing framework” is called “components”, or “plug-ins”

- Code hierarchy: ompi/mca/\<framework\>/\<components\> (e.g., ompi/mca/pml/ucx)

There are many frameworks on OpenMPI, and some notable ones are:

- PML : point-to-point messaging layer

  - Such as MPI_SEND, MPI_RECV

  - There are two components to support RDMA on OpenMPI

    - UCX

      - Default in 5.0

      - Directly connected to UCX library

      - According to <https://www.infinibandta.org/wp-content/uploads/2019/10/IB-Interop-Testing-MOI-2019-10-09.pdf> , Mellanox suggested to drop openib and use ucx → **<span style="color: rgb(255,153,31);">Let’s look at UCX</span>**

      - GPU transfer built in support (CUDA / ROCm)

      - Looks like it use RDMA send/recv (not sure about RDMA read or write)

    - ob1 (borrowed from starwars)

      - Use BTL components (but deprecated in 5.0 and above

      - Previous default point-to-point library.

- Coll: collective communication

  - In a typical collective operation, the MPI algorithm issues multiple P2P communication requests. For example, MPI_Bcast() sends the same message to a set of ranks.

    - (<https://sc19.supercomputing.org/proceedings/workshops/workshop_files/ws_exampi105s2-file1.pdf>)

    - By MCA_PML_CALL, collective communications are restored to PML calls

    - There are some library called UCC ← managed in UCX project.

- OSC: one-sided communication

  - Such as MPI_PUT, MPI_GET

  - Notable component: UCX

    - Directly connected to UCX library, and eventually restored to RDMA read/write
