# Error handling in RDMA

<!-- id=152502273 parent=32211017 -->

There are several types of errors in RDMA. Each error needs corresponding fault behavior.

Currently, our HW assumes that the driver and HW works well without any bug (also for the connection peer if exists). That is, it is not considering fault behaviors from HW-SW system failure (e.g. recv WQE has some corrupted fields). For now, the error coverage only includes the cases from user’s fault (e.g. post send with wrong Lkey). Further non-user-fault error cases may be supported in the future where the verification will be done via VMG.

Below is the summary from RDMA specification. So MB already supports / or later should support the following features.

## Error State and WQE Flush

Some ibv structs have error states to properly handle abnormal situations. Given that we only regard user’s fault, the main concern will be the error states of **QP**. **SRQ** and **(S)CQ** also have the error state, but it can never be the case of user’s fault.

Once the ibv struct has fallen into an error state, its functionality fails; it will not respond to incoming WRs nor packets anymore, while all associated WQEs must be flushed. You can see some details of error state behavior for each ibv struct.

#### QP error state **(implementation partially done)**

A QP falls into an error state when **(1)** a HW detects abnormal behavior resulting in QP state transition into an error, **(2)** it attempts to access other structs (e.g. SRQ, (S)CQ) that is in error state, and **(3)** SW manually calls modify_qp cmd to changes such QP into an error state.

Once a QP falls into an error state, it ignores all incoming WQEs and packets. All pending (already posted but not completed yet) WQEs in such QP must be returned with IBV_WC_WR_FLUSH_ERR status. The only exception to this rule is the very first one (i.e. earliest-issued WQE among pending ones) if the error state transition has been occurred from HW error completion. In such case, the wc_status field of the first WQE must be preserved with the original value. Besides, completion generation must be processed in-order even if the QP has entered an error state.

#### SRQ error state **(out of our scope)**

SRQ falls into an error state when it **(1)** detects malformed RQE to be fetched and **(2)** experiences local QP conditions to prevent inbound packets to be processed normally.

*Note – We still do not assume error cases which are not from user’s fault; I think both of (1) and (2) are out of our scope, which means that we do not need to implement SRQ error state handling at all.*

Once an error state transition of a certain SRQ has been occurred, it must flush its queue to report the error status upward to the SW. However, unlike RQ’s error state transition, it does not cause any NAK generations. An error handling behavior related to SRQ always obeys the rule of **Responder Class A Fault Behavior**. There is no way to recover SRQ’s error state, where the only way is to destroy and re-create it.

Any QPs that attempt to access an error-state SRQ also fall into an error state and follow Responder Class A Fault Behavior. In this case, NAK-Remote Operational Error will be sent to the client.

#### (S)CQ error state **(out of our scope)**

The only scenario that (S)CQ falls into an error state is CQ overflow; both requester and responder can trigger this event when one or more QP attempts to write new CQE(s) to the CQ and such CQ is inaccessible or already full.

*Note – Similar as above, I think we do not need to implement (S)CQ error state handling at all.*

Since the CQ has been failed, all CQEs corresponding to outstanding WQEs cannot be delivered toward SW but to remain in unknown state. Like SRQ’s case described in above, any QPs that attempt to access an error-state (S)CQ also fall into an error state.

## Requester / Completer Side Fault Behavior (RC QP)

#### Class A : recoverable error with retry, totally transparent to SW **(already done)**

- Packet sequence error. Retry count not exceeded.

- Implied NAK sequence error. Retry count not exceeded.

- Local Ack Timeout error. Retry count not exceeded.

- RNR NAK Retry error. RNR retry count not exceeded.

- **MB already supports all of them.**

#### Class B : current WR completed in error, subsequent WRs all flushed, QP into an error state **(partially done)**

Common behaviors of the HW regarding this type of error

- This kind of error can occur in out-of-order manner; there could be some WQEs which has been issued prior to error-detected WQE but still in flight. In such case, HW must not generate error completion toward SW until all prior WQEs to be completed.

  - If all prior WQEs successfully completed, then subsequently report error completion toward SW. Flush SQ and RQ (**NOT SRQ**. SRQ-related case must be handled differently.) if needed.

  - If any error (send / recv doesn’t matter) occurs while holding aforementioned error-detected WQE, then simply discard the error status. SW will automatically treat such WQE to be flushed.

- The occurrence of this error must be totally transparent to the connection peer.

#### Class C : error that cannot be associated with certain QP **(out of our scope)**

- Local Operation Error -affiliated or unaffiliated

  - Once it occurs, then flush all subsequent send / recv WQEs and change QP into an error state

  - Current HW patch deals with this error by returning tail WQE with IBV_WC_WR_FLUSH_ERR completion, and then SW generates all subsequent flush err cmpls if exist.

- One example of this error type is SRQ-related error

  - A failure of SRQ does not make all associated QPs into an error state

  - Instead, any QP that **attempts to access already-dead SRQ** falls into this kind of error state, executing the actions written above

    - Thus, responder_frontend is responsible for detecting SRQ-related Class C Error

- **Only the SRQ-related behavior has verified with CX-CX**

#### Class E : no retry, but recoverable **(already done)**

- Ghost Acknowledge (it seems that this is about duplicate request/ACK handling)

- **This is already supported by our HW IP.**

#### Class G : CQ overflow **(out of our scope)**

- CQ overflow. CQ is inaccessible or full while an attempt is made to complete a WQE.

  - QP falls into an error state.

  - CQ falls to an **Affiliated Asynchronous error state**; any WQ that attempts to access this CQ will fall into an error state.

  - All outstanding send / recv WRs are remained with unknown state.

  - **CX-CX behavior not verified, MB does not support this yet (but should we support this?)**

## Responder Side Fault Behavior (RC QP)

#### Class A : Error from local QP or SRQ **(not started)**

- Lkey violation of RQE

  - current RQE is completed with **IBV_WC_LOC_PROT_ERR**

  - all subsequent send / recv WRs are completed with **IBV_WC_WR_FLUSH_ERR**

  - send **NAK-Remote Operational Error** to its connection peer

  - Note - RQE fetched from SRQ can also trigger this. According to CX’s behavior in the case, it *kills the recipient QP but not the SRQ. (and the current error handling routine of MB already does as desired)*

#### Class B : Error detected while processing inbound request packets **(already done)**

In this case, responder has a responsibility to notify the occurrence of error to its connection peer by NAK. After detecting this type of error, NAK with appropriate syndrome is always generated. This type of error does not affect the state nor the functionality of the local side QP.

In fact, only two cases below belongs to the **RC** Responder Side Class B Fault Behavior.

- PSN sequence error. **This is already supported by our HW IP.**

- RNR. **This is already supported by our HW IP.**

#### Class C : Error detected while processing inbound request packets **(partially done)**

In addition to the Class B Fault Behavior, Class C Error detected by the responder must be reported to the SW of local side as well. The local QP turns its state into an error state, generates a corresponding error completion, and sends unrecoverable NAK to its remote peer. The remote QP receives NAK and eventually turns into an error state as well, while flushing its WQEs if needed.

(common) once an error completion has been generated, then all subsequent send / recv WRs of such QP will be completed with **IBV_WC_WR_FLUSH_ERR** (errors w/o error cmpl generation, such as Rkey violation, must not trigger this). In Responder Side Class C Fault Behavior, this eventually happens at the both local and remote side.

*Note - All cases that generate NAK - Remote Operational Error belong to Responder Side Class A Fault Behavior, so we do not need to implement them.*

#### Class D : exactly same behavior as Class B **(already done)**

- If an incoming packet header does not match any of active QP in the responder, then silently drop it. **This is already supported by our HW IP.**

#### Class G : CQ overflow **(out of our scope)**

- CQ overflow. CQ is inaccessible or full while an attempt is made to complete a WQE.

  - QP falls into an error state.

  - CQ falls to an **Affiliated Asynchronous error state**; any WQ that attempts to access this CQ will fall into an error state.

  - All outstanding send / recv WRs are remained with unknown state.

  - **CX-CX behavior not verified, MB does not support this yet (but should we support this?)**

#### Class J : Send with invalidate error **(not started)**

- Send with invalidate error. It must obey the rules described below.

  - This error has lower priority than preceding send error(s).

    - For example, if a remote host sent “send only with invalidate” and it caused other unrecoverable fault, then the IETH Rkey violation must not be reported.

  - In case only the preceding packet(s) have been processed successfully, the inspection for this error begins. Then,

    - regardless of this error to occur or not, one recv WR is always consumed from the RQ.

    - recv WR must not be completed before the inspection finishes.

  - If Rkey contained in IETH is invalid, then this error occurs. The subsequent action must be:

    - Complete the current RQE with IBV_WC_MW_BIND_ERR, turn such QP into an error state

    - Do **NOT** send any NAK to its connection peer (besides, if the client performs any other WRs, then it will eventually end up with retry exceed error)

## RDMA specification for Reference

#### 11.6.2. Completion Return Status

**Success**

- Operation completed successfully.

**Local Length Error**

- Generated for a Work Request posted to the local Send Queue when the sum of the Data Segment lengths exceeds the message length for the channel adapter port. Generated for a Work Request posted to the local Receive Queue when the sum of the Data Segment lengths is too small to receive a valid incoming message or the length of the incoming message is greater than the maximum message size supported by the HCA port that received the message.

**Local QP Operation Error**

- An internal QP consistency error was detected while processing this Work Request.

**Local Protection Error**

- The locally posted Work Request’s Data Segment does not reference a Memory Region that is valid for the requested operation.

- On a Reserved L_Key:

  - The Verbs Consumer does not have Reserved L_Key access enabled.

**Work Request Flushed Error**

- A Work Request was in process or outstanding when the QP transitioned into the Error State.

**Memory Management Operation Error**

- The Verbs Consumer had insufficient rights to perform the operation, because:

- On a Fast-Register this includes:

  - The Verbs Consumer does not have Fast Register access enabled;

  - Requested PBL exceeds size of allocated PBL;

  - Remote access was requested, but L_Key does not have an accompanying R_Key; or

  - Memory access was attempted on an L_Key or R_Key that is in the Invalid State.

- On a Memory Window this includes:

  - The Verbs Consumer has insufficient access rights;

  - A zero length Bind Memory Window or Post Send Bind WR operation on a Type 2 Memory Window

  - Verb Consumer attempted to Bind a MW to a Zero Based Virtual Address Memory Region.

- On an Invalidate Operation this includes:

  - Memory access was attempted on an L_Key or R_Key that is in the Invalid State;

  - Memory Region could not be Invalidated, because it is a Shared Memory Region;

  - Memory Region can not be invalidated because it has bound Memory Window; or

  - Memory Region could not be Invalidated, because it was created through a Register Memory Region or Reregister Memory Region.

  - Memory Window could not be Invalidated, because it was a Type 1 Memory Window.

The following errors are reported only for Reliable QPs.

**Bad Response Error**

- An unexpected transport layer opcode was returned by the responder.

**Local Access Error**

- A protection error occurred on a local data buffer during the processing of a RDMA Write with Immediate Data operation sent from the remote node.

**Remote Invalid Request Error**

- The responder detected an invalid message on the channel. Possible causes include the operation is not supported by this receive queue, insufficient buffering to receive a new RDMA or Atomic Operation request, or the length specified in an RDMA request is greater than 231 bytes.

- In the case where the buffer size is insufficient to handle the request, the number of bytes transferred into the buffer is indeterminate. However, the CI shall not write beyond the buffer bounds.

**Remote Access Error**

- A protection error occurred on a remote data buffer to be read by an RDMA Read, written by an RDMA Write or accessed by an atomic operation. This error is reported only on RDMA operations or atomic operations.

**Remote Operation Error**

- The operation could not be completed successfully by the responder. Possible causes include a responder QP related error that prevented the responder from completing the request or a malformed WQE on the Receive Queue.

**Transport Retry Counter Exceeded**

- The local transport timeout retry counter was exceeded while trying to send this message.

**RNR Retry Counter Exceeded**

- The RNR NAK retry count was exceeded.

# HW Requester Side Error Handling

(IBTA 1.7 final 기준 spec 9.9 ~ 9.9.2.4.6)

#### Error types

- 발생한 위치에 따른 구분

  - Locally detected error - remote side의 개입 없이 local HW에서 error를 발견한 경우 ((RNR) retry exceed도 일단은 이 부류임)

  - Remotely detected error - 상대의 responder 측에서 발생했지만 거기서 온 NAK을 받아 이쪽 QP도 error state로 빠져야 하는 경우

- 이후 처리 방법에 따른 구분

  - **Class A - retry 등을 통해 recoverable한 error 종류이며 이미 MB에도 구현되어있음**

  - **Class B - 대상 WQE에 대한 err cmpl 생성, 해당 QP의 기능 정지 <span style="color: rgb(76,154,255);">+ error state로 변경, 이후 WQE들이 있다면 flush</span>**

  - **Class C - (error가 어떤 WQE에 특정될 수 없기 때문에) 대상 QP를 특정할 수 있는 경우에만 그 QP를 error state로 변경 + 이후 WQE들이 있다면 flush**

  - **Class E - duplicate response에 대한 처리이므로 이미 MB에도 구현되어있음**

  - **Class F - CQ가 망가진 경우이기 때문에 지금 당장 구현할 필요는 없을 듯**

# HW Responder Side Error Handling

(IBTA 1.7 final 기준 spec 9.9.3 ~ 9.9.3.1.10)

#### Error types

- Class A : (S)RQ 혹은 RQE에 문제가 있어서 이쪽의 QP가 error state로 빠지는 경우

  - (YH) 스펙에 의하면 SRQ도 error state에 빠질 수 있고 이 경우 처리가 좀 복잡해지는데, SRQ가 error state로 빠지는 경우는 malformed WQE 외에는 일어날 가능성이 없어보임 → 일단 지금은 생각 안해도 될 듯

  - **처리 (항상 RQE 때문에 발생한다고 가정) - NAK remote operational error 보낸 후 해당 RQE에 대한 err cmpl 생성, <span style="color: rgb(76,154,255);">그 RQE가 있던 RQ 및 QP를 error state로 바꾸고 다른 WQE들은 flush</span>**

    - **만약 해당 QP가 다른 QP와 SRQ를 공유하고 있다고 하더라도 QP에서 발생한 error는 해당 QP에 한정됨; <span style="color: rgb(76,154,255);">해당 RQE + error를 발견한 순간까지 해당 QP로 fetch가 이미 완료된 RQE들만 flush되고</span>, SRQ 자체 or 그와 연결된 다른 QP는 그대로 정상 동작해야 함**

    - **error가 QP가 아니라 SRQ 자체에서 발생할 수도 있지만, 위에서 언급한 것과 같이 이 경우는 배제해도 될 듯함**

- Class B : PSN sequence error 혹은 RNR처럼 recoverable한 종류

  - **이미 MB HW에 구현되어있음**

- Class C : 상대가 보낸 packet에 문제가 있어서 이쪽 QP가 error state로 빠지는 경우

  - **처리 - 각 case에 맞는 NAK을 보낸 후 <span style="color: rgb(76,154,255);">문제가 된 req가 send나 write with immediate였을 경우에만 RQE 1개를 소비하고 err cmpl 생성, 이와 별개로 QP는 항상 error state로 빠지며 나머지 WQE들을 flush</span>**

- Class D : Silent drop해야 하는 경우

  - **처리 - RC duplicate atomic인 경우 이미 MB HW에 구현되어있음, <span style="color: rgb(76,154,255);">이외에도 SRQ가 아닌 UD (send) length error 또한 이렇게 처리되어야 함</span>**

- Class E : UD SRQ일 때 들어온 (send) packet이 length error인 경우

  - **<span style="color: rgb(76,154,255);">처리 - 해당 RQE에 대한 err cmpl 생성, error를 발견한 시점에서 이미 SRQ로부터 해당 QP로 fetch된 RQE들만 flush, 이후 모든 QP와 SRQ는 error state로 바뀌지 않고 계속 동작해야 함</span>**

- Class G : CQ overflow

  - **requester side와 마찬가지로 당장 구현할 필요 없을 듯**

- Class J : Send with invalidate 받았지만 해당 Rkey로 invalidate 불가능할 경우

  - **<span style="color: rgb(76,154,255);">처리 - 만약 해당 send가 invalidate 처리 이전에 다른 문제가 있었다면 그것대로 진행해야 함</span>**

  - **<span style="color: rgb(76,154,255);">send 자체는 문제없지만 invalidate에만 문제가 있는 경우 → 이 경우에도 정상 동작 때와 마찬가지로 RQE 1개 소모하면서 err cmpl 생성, 해당 QP를 error state로 바꾸고 나머지 WQE들을 flush시키지만 NAK은 보내지 않음</span> (i.e. 상대편은 이 error의 발생 여부를 알 수 없음)**

# SW Side Error Handling

(IBTA 1.7 final 기준 spec 10.10 ~ 10.10.3.4)

(여기서 Channel Interface = RNIC이라고 가정함)

#### Error types

- Immediate error : Verb 또는 driver 단계에서 걸러지는 종류, HW에게 절대로 보이지 않음

- Completion error : 일단 WQE가 HW로 post된 후에 감지된 error로, 아래 2종류가 있음

  - interface check - (굉장히 설명이 애매한데) link에 data를 올리기 전에 (=packet 쏘기 전?) 잡아낸 error

  - processing error - HW가 WQE processing을 하다가 잡아낸 error

  - 사실상 이 2종류는 완전히 똑같은 것으로 생각해도 될 듯

  - **SQ error 처리 - HW로부터 대상 WQE에 대한 err cmpl이 올라옴, 해당하는 QP의 기능 정지 + error state로 변경, 이후 WQE들은 있다면 flush**

  - **RQ error 처리 - HW로부터 대상 WQE에 대한 err cmpl이 올라옴, 해당하는 QP의 기능 정지 + error state로 변경, 이후 WQE들은 있다면 flush**

- Asynchronous error : error가 났지만 error completion으로 전달될 수 없는 방식일 때

  - 따라서 Asynchronous Event Handler Verb라는 방식으로 driver → Verb 전달

  - Unaffiliated asynchronous error - error가 어떠한 QP/CQ에 특정되지 않는 경우

    - **처리 - 따로 뭔가 안 하고 Verb단에서 Local Catastrophic Asynchronous Error를 띄우는 듯 (Table 95)**

  - Affiliated asynchronous error - error의 원인을 어떤 QP/CQ로 특정할 수 있지만 error completion을 보낼 수 없는 상태인 경우

    - **처리 - 해당하는 QP의 기능 정지 + error state로 변경, 해당 QP에 연결된 CQ가 망가지지 않았다면 err cmpl + flush 올라옴**

#### YH’s opinion

- 같은 error type에 대해 Interface / Processing이 모두 존재하는 만큼 정석은 driver / HW 양쪽에서 모두 (같은 것들을 체크하더라도) error check를 해주는 것인 듯함

- asynchronous error가 HW 안에서 발생한다면 이를 SW에 알려주는 로직이 필요할 듯?
