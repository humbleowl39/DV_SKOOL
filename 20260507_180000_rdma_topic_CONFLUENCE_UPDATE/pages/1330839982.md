# About PSN-related fields of CQE (DV spec delivery)

<!-- id=1330839982 parent=32080200 -->

# 1. Field locations

From the CQE bit layout, the three PSN-related debug fields are:

|                  |       |                  |                                                                                  |
|------------------|-------|------------------|----------------------------------------------------------------------------------|
| Field            | Width | Bit offset (LSB) | Nominal meaning                                                                  |
| `pkt_psn`        | 24    | 368              | PSN of the packet that produced this completion.                                 |
| `debug_epsn`     | 24    | 400              | Device-tracked expected PSN (ePSN) at the time the CQE was generated.            |
| `debug_last_psn` | 24    | 424              | Last expected PSN of the WR this CQE belongs to (`first_psn + num_packets − 1`). |

<div>

<div>

**Field names are aspirational, not contractual.** On several CQE categories the actual value carried by a field deviates from its nominal meaning, and several categories leave one or two fields at 0 (unused, not "PSN was 0"). Always disambiguate via `cmpt_src` + `wc_status` (and `debug_wc_flag` for the fatal-event sub-case) before interpreting the PSN fields. The black-box contract for each wrapper output is given in §3.

</div>

</div>

# 2. CQE source identification

Every CQE carries a `cmpt_src` tag and a `wc_status`. Together they classify the CQE into one of the categories listed in §3. The DV team should treat the requester, responder, and completer wrappers as black boxes; the fingerprints below are sufficient to attribute every CQE to its emitting wrapper without inspecting wrapper internals.

|                                 |                                                                      |                                                                                         |
|---------------------------------|----------------------------------------------------------------------|-----------------------------------------------------------------------------------------|
| Wrapper                         | Possible `cmpt_src` values                                           | Possible `wc_status`                                                                    |
| Requester                       | `REQUESTER` (success), `ERROR` (requester-detected)                  | `IB_WC_SUCCESS`, `IB_WC_LOC_PROT_ERR`, …                                                |
| Responder                       | `RESPONDER`, `RESPONDER_OPS` (success), `ERROR` (responder-detected) | `IB_WC_SUCCESS`, `IB_WC_LOC_PROT_ERR`, `IB_WC_LOC_LEN_ERR`, `IB_EVENT_QP_ACCESS_ERR`, … |
| Completer                       | `COMPLETER` (success), `ERROR` (completer-detected)                  | `IB_WC_SUCCESS`, `IB_WC_LOC_QP_OP_ERR`, `IB_WC_REM_*`, `IB_EVENT_QP_FATAL`, …           |
| Device-fatal monitor (separate) | `ERROR` with `debug_wc_flag = 0xba`                                  | `IB_EVENT_DEVICE_FATAL`                                                                 |

An `ERROR` CQE is attributed to the wrapper that detected the error; the device-fatal sub-case is fingerprinted by `debug_wc_flag = 0xba` + `cqid = RSVD_CQ_ERR` and must be handled separately (see §3.4).

## 3.4 Device-fatal CQE (separate from the three wrappers)

The device-fatal monitor emits CQEs into the same CQ stream but is not part of the requester/responder/completer wrappers. It is identified by the following fingerprint:

|                 |                         |
|-----------------|-------------------------|
| Field           | Value                   |
| `cmpt_src`      | `ERROR`                 |
| `wc_status`     | `IB_EVENT_DEVICE_FATAL` |
| `debug_wc_flag` | `0xba`                  |
| `cqid`          | `RSVD_CQ_ERR`           |

For this category the PSN fields are repurposed:

|                  |                                                            |
|------------------|------------------------------------------------------------|
| Field            | Carries                                                    |
| `pkt_psn`        | **0**                                                      |
| `debug_epsn`     | **0**                                                      |
| `debug_last_psn` | **Internal HW occupancy meter (12-bit value). NOT a PSN.** |

DV scoreboards **must** short-circuit on this fingerprint and refuse to decode `debug_last_psn` as a PSN.

# 4. Observable behaviours and batching fingerprints

## 4.1 ACK coalescing (completer success, write/send)

When a single inbound ACK acknowledges N write/send WRs at once, the completer wrapper emits N consecutive CQEs (`cmpt_src = COMPLETER`, `wc_status = IB_WC_SUCCESS`). Within the batch:

- `pkt_psn` is identical on every CQE in the batch.
- `debug_epsn` is identical on every CQE in the batch (it is the closing ACK's BTH PSN).
- `debug_last_psn` increases per CQE (it is the per-WR last expected PSN).
- The last CQE in the batch satisfies `debug_last_psn = debug_epsn`.
- Earlier CQEs in the batch satisfy `debug_last_psn < debug_epsn` (modulo 2<sup>24</sup> wraparound).

For a non-coalesced clean completion (batch of 1), all three fields coincide: `pkt_psn = debug_epsn = debug_last_psn`.

## 4.2 Implied ACK (completer success, write/send)

When an inbound ACK arrives with a BTH PSN beyond a write/send WR's last expected PSN (skipping that WR's individual ACK), the completer emits a CQE for that WR carrying `debug_last_psn < debug_epsn`. From the DV black-box view this is indistinguishable from a coalesced-batch member, and DV scoreboards should treat both uniformly.

## 4.3 RI batching (responder success, write-w-IMM)

A single inbound `WRITE_WITH_IMM` can produce N CQEs with `cmpt_src = RESPONDER_OPS`, where N is the WR/QP-configured `num_cqe`. All CQEs in the batch share identical `pkt_psn` (and identical other identifying fields). N is bounded and known per QP, so DV can verify exact batch length.

## 4.4 Completer success — PSN field name vs. value

For all completer success CQEs, the field **`pkt_psn` does not carry the inbound ACK's BTH PSN.** It carries the device-tracked expected-PSN at the moment the closing ACK began processing. The two values coincide only on a non-batched single-WR completion. DV scoreboards comparing `pkt_psn` against a wire monitor must:

- For non-batched (single-CQE) completions: assert `pkt_psn` equals the closing ACK's BTH PSN.
- For batched completions: assert `pkt_psn` is constant across the batch; do not assert against the wire-observed BTH PSN of the triggering ACK.

# 5. DV coverage hooks

1.  **Coalesced-batch invariant** (write/send completer success): within one batch, `pkt_psn` is constant, `debug_epsn` is constant, `debug_last_psn` is strictly monotone increasing, and the final CQE satisfies `debug_last_psn = debug_epsn`.
2.  **Non-batched-completion equality** (completer success): when only a single CQE is produced for one inbound ACK / response, `pkt_psn = debug_epsn = debug_last_psn`.
3.  **Read / atomic invariant** (completer success, read or atomic `wc_opcode`): `debug_last_psn = debug_epsn`.
4.  **RI-batch invariant** (responder success, `RESPONDER_OPS`): a run of N consecutive CQEs sharing identical `pkt_psn`, with N matching the WR/QP-configured `num_cqe`.
5.  **Field-zero invariants**:
    - Requester success / requester error: `debug_epsn = 0` AND `debug_last_psn = 0`.
    - Responder success (both `RESPONDER` and `RESPONDER_OPS`): `debug_epsn = 0` AND `debug_last_psn = 0`.
    - Responder error: `debug_last_psn = 0`.
    - Completer error payload-side category: `debug_epsn = 0`.
6.  **Device-fatal short-circuit**: when the device-fatal fingerprint matches (§3.4), do not decode `debug_last_psn` as a PSN.
7.  **Wire-comparison rule for completer `pkt_psn`**: only assert against the wire BTH PSN of the closing ACK on non-batched single-WR completions.
8.  **Error-PSN comparators**: for completer primary errors, `(pkt_psn vs. debug_epsn)` is expected to land in `{LEFT_LARGE, EQUAL}` for OOO / missing-PSN faults; for responder errors, the gap `(debug_epsn − pkt_psn) mod 2`<sup>`24`</sup> is the OOO or duplicate index.
9.  **Cross-WR PSN continuity**: across consecutive successful completer CQEs on a single QP (whether batched or not), `debug_epsn` is monotone non-decreasing modulo 2<sup>24</sup>.

# 6. Maintenance

- If the RTL changes the value carried by any of the three fields for any CQE category, update the corresponding row in §3 and the affected coverage hooks in §5 in the same change.
- If a new `cmpt_src` value or a new `debug_wc_flag` sub-category is introduced, add a row in §3 with explicit field semantics, and add the corresponding invariants in §5.
- If a field is repurposed (e.g. an additional case of `debug_last_psn` carrying something other than a PSN, analogous to the device-fatal CQE), document the repurposed semantics explicitly under a new sub-section in §3 and add a short-circuit rule in §5.
