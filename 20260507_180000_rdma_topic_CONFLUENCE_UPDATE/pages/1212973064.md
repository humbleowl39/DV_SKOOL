## 2. Functional Behavior

### 2.1. RDMA response packet handling

Whenever a new RDMA ACK or Read Response packet comes in, this module:

- checks if there are any intrinsic (e.g. NAK) or induced (e.g. Opcode sequence violation) errors and

  - triggers retry if it is recoverable

  - sends out an error completion if it is unrecoverable

- may complete some pending WQE(s) with corresponding WQE deletion (always) and completion (if signaled)

  - then, it may update the oldest unfinished WQE’s state according to the incoming packet

- provides the information about dealing with the arrived packet’s payload or generated completion(s), including:

  - if the packet includes a payload, then whether to drop it or not + its destination DMA address (if not)

  - ordering rules among completion(s) and (if exist) payload segment(s)

- provides the information about adjusting the Congestion Control (CC) settings, including:

  - notification of CNP arrival

  - ACK’ed status of WQEs

### 2.2. QP control signal handling

This module is a part of a QP control stream pipeline. It bypasses the incoming QP control commands. However, if the signal was to ‘Destroy QP’, then it generates some cleanup signals toward info_arb (which stores WQE) and CC-related module.

### 2.3. MR control signal handling

This module is a part of an MR control stream pipeline. It bypasses the signal by default but additionally generates cleanup signals toward MMU, only for the case ‘Deregister MR’.

### 2.4. Retry handling

- This module keeps track of every active QP’s timer that may expire. New status update signals from info_arb may refresh the current state of the timer.

- Some incoming RDMA ACK / Read Response packets and timer expirations may issue a retransmission. All retry setup and detections occur inside this wrapper, except the initialization which must be signaled from the Requester side. Once the module decides to start a retry, it:

  - signals info_arb to fetch some pending WQEs again for retry, when the target QP is still alive and there isn’t any already-fired retry for that QP

  - sends the information about NAK to CC-related module, if the retry was triggered by the (implied) NAK

  - sends the information about Selective ACK to the Requester, if the retry was triggered by a received SACK packet

- Retry count (or RNR retry count) may exceed, when this module fails to get fresh ACKs or Read Responses in many consecutive times. In that case, it notifies info_arb and invalidates the corresponding QP.

## 3. Constraints & Exceptions

- The following descriptions are about valid I/O sequences. Any violations from those must be considered as illegal.

- Ordering legend — **\[TRIGGER\]**: initiating event. **\[THEN\]**: strict sequential order. **\[ANY-ORDER\]**: siblings may reorder. **\[ASYNC\]**: asynchronous, no ordering. **\[IF/ELSE\]**: conditional. **\[PRECONDITION\]**: must hold before entry. **\[COMPILE-TIME: X\]**: only when macro X is defined. **\[DEFAULT\]**: applies when no compile-time guard matches.

### 3.1. RDMA response packet handling

3.1.1. **\[TRIGGER\]** Initiated by `s_comp_header_stream` input

3.1.2. **\[THEN\]** QP and QP state metadata read request via `m_comp_qp_rw_port_awd` and `m_comp_qp_state_rw_port_front_awd` → response via `m_comp_qp_rw_port_rd` and `m_comp_qp_state_rw_port_front_rd`

3.1.3. **\[THEN\]** Early termination & drop1/2

3.1.3.1. **\[ANY-ORDER\]** Processing terminates if the target QP is invalid or not-yet-RTS, input packet’s source IP address mismatches with the QP’s remote one, the packet’s PSN \< ePSN (duplicate), or the packet is large-PSN SACK / non-SR SACK. Only the SACK cases generate error completion via `m_comp_err_stream` with writing QP’s state into an error via `m_comp_qp_state_rw_port_front_awd`.

3.1.3.2. **\[ANY-ORDER\]** Regardless of termination, drop1/2 must output if the packet contains any payload. Drop sequences: termination `m_comp_payload_drop1`(1) → no `m_comp_payload_drop2`, OR `m_comp_payload_drop1`(0) → `m_comp_payload_drop2`(1). Continuation: `m_comp_payload_drop1`(0) → `m_comp_payload_drop2`(0).

3.1.4. **\[THEN\]** Unless the input packet is a same-PSN NAK, RNR NAK, SACK, request WQE info via `m_cmd_port_3` -\> get data via `s_data_port_3`

3.1.4.1. **\[ANY-ORDER, COMPILE-TIME: COALESCING_BY_QP_TYPE\]** If the QP is only for write/send, then treat the WQEs as completed cumulatively up to the packet’s MSN and request subsequent one. Request this QP’s post list via `m_post_rw_port_awd` (and retrieve via `m_post_rw_port_rd`) as well for large-PSN packets.

3.1.4.2. **\[ANY-ORDER\]** Otherwise, request the oldest pending WQE.

3.1.5. **\[THEN\]** If this packet is ECN-marked and has a payload (including zero-length), then notify its target QP via `m_notify_cnp_qpn`

3.1.6. **\[THEN\]** ACK coalescing

3.1.6.1. **\[COMPILE-TIME: COALESCING_BY_QP_TYPE\]** If the QP is only for write/send, then output all completions ahead of the packet via `m_comp_cmpt_stream`.

3.1.6.2. **\[DEFAULT\]** Otherwise, repeat the loop {send out the completion for the oldest WQE via `m_comp_cmpt_stream` → request subsequent WQE via `m_cmd_port_4` → retrieve via `s_data_port_4`} until the packet’s PSN falls within currently oldest WQE (including boundary) or the oldest WQE becomes RDMA Read or Atomic. If info_arb returns an invalid WQE info, the loop breaks with an error completion via `m_comp_err_stream` with writing QP’s state into an error via `m_comp_qp_state_rw_port_front_awd`.

3.1.7. **\[THEN\]** If the packet carries a non-zero payload, exactly one drop3 is output: `m_comp_payload_drop3`(0) on successful processing, `m_comp_payload_drop3`(1) on same-PSN unrecoverable error or implied NAK during coalescing.

3.1.8. **\[THEN\]** If this packet finishes the oldest WQE (i.e. packet PSN == WQE info’s last PSN), generate a completion via `m_comp_cmpt_stream`. Or if the packet indicates unrecoverable error, generate an error completion via `m_comp_err_stream` with writing QP’s state into an error via `m_comp_qp_state_rw_port_front_awd`.

3.1.9. **\[THEN\]** Issue info deletion for all completed WQEs via `m_cmd_port_0` → if this packet has changed the progress of the oldest WQE, then issue modification via `m_cmd_port_0` → collect all deleted WQEs via `s_data_port_0`

3.1.10. **\[THEN\]** Post-processing

3.1.10.1. **\[ANY-ORDER\]** Except same-PSN unrecoverable error cases, send CC window update signal via `m_ack_info`

3.1.10.2. **\[ANY-ORDER\]** If the packet has no NAK, implied NAK, or error, and has a payload with valid MR, request MR info via `m_comp_mr_rw_port_normal_scatter_awd`

3.1.11. **\[THEN\]** Generate a direction signal via `m_comp_payload_cmd_stream` for each condition below (**exceptionally order matters here**):

3.1.11.1. **\[IF\]** Any coalesced WQE(s) → (cmpt_num = coalesced count, dest = PAYLOAD_DEST_COMP_ZERO_PAYLOAD)

3.1.11.2. **\[IF\]** Zero-length Read without any NAK, WQE is signaled → (cmpt_num = 1, dest = (MR valid)? PAYLOAD_DEST_COMP_ZERO_PAYLOAD : PAYLOAD_DEST_COMP_MR_INVALID)

3.1.11.3. **\[IF\]** Read with a payload without any NAK, MR is invalid → (len = payload’s byte length, drop = 1, dest = PAYLOAD_DEST_COMP_MR_INVALID)

3.1.11.4. **\[IF\]** Read with a payload without any NAK, MR is invalid, last oldest WQE is signaled and completed → (cmpt_num = 1, dest = PAYLOAD_DEST_COMP_MR_INVALID)

3.1.12. **\[THEN\]** If the packet is RDMA Read with a payload without any NAK, MR is valid → first retrieve MR metadata via `m_comp_mr_rw_port_normal_scatter_rd`, then repeat the loop until all of the payload has been sent: {send a direction signal via `m_comp_payload_cmd_stream` where (len = DMA length for this segment, cmpt_num = (last segment && WQE signaled)? 1 : 0, dest = PAYLOAD_DEST_COMP_NORMAL). Segments split at SGE and page boundaries. Each segment is accompanied by a VA via `m_comp_write_translate` → (switch to next SGE or page if needed)}

### 3.2. QP control signal handling

3.2.1. **\[TRIGGER\]** Initiated by `s_qp_control_cmd`

3.2.2. **\[THEN\]** QP and QP state metadata read request via `m_comp_qp_rw_port_awd` and `m_comp_qp_state_rw_port_front_awd` → response via `m_comp_qp_rw_port_rd` and `m_comp_qp_state_rw_port_front_rd`

3.2.3. **\[THEN\]** If its key is Destroy SQ, trigger MEM_FLUSH via `m_cmd_port_0` and send CC window reset signal via `m_ack_info`

3.2.4. **\[THEN\]** Output bypass via `m_qp_control_cmd_stream`

### 3.3. MR control signal handling

3.3.1. **\[TRIGGER\]** Initiated by `s_mr_ctrl_stream`

3.3.2. **\[THEN\]** If its key is Deregister MR, zero-write MR via `m_comp_mr_rw_port_normal_scatter_awd` and:

3.3.2.1. **\[IF\]** If it commands MMU flush, signal via `m_mmu_flush` → reap the response via `s_mmu_flush_resp`

3.3.2.2. **\[ELSE\]** Otherwise, specify the MR ID via `m_dereg_mr_id` → reap the response via `s_dereg_rsp`

3.3.3. **\[THEN\]** Output bypass via `m_mr_ctrl_stream`

### 3.4. Retry handling

3.4.1. **\[PRECONDITION\]** There must be `req_retry_info_stream_v` input of certain QP before use

3.4.2. **\[ASYNC\]** Variety of cause of retry allows the following to occur asynchronously:

3.4.2.1. **\[ASYNC\]** `s_set_timer` accepts four masks: TIMER_START (start if inactive), TIMER_RESTART (restart unless RNR timer is active), RNR_TIMER_RESTART (always restart with RNR timeout), TIMER_STOP (deactivate). A normal TIMER_RESTART is suppressed if the QP's timer is currently counting down an RNR timeout.

3.4.2.2. **\[ASYNC\]** RNR NAK received → send MEM_RNR command via `m_cmd_port_1`, mark the retry has already been started (to prevent subsequent NAK-driven retry)

3.4.2.3. **\[ASYNC\]** PSN NAK or implied NAK occurred but there exists other retry in progress → notify cc_handler via `m_nak_info`

3.4.2.4. **\[ASYNC\]** Local ACK timeout without count exceed OR PSN / implied NAK occurred with no other retry in progress → trigger MEM_RETRY via `m_cmd_port_1`, output both `m_nak_info` and `m_sack_info`

3.4.2.5. **\[ASYNC\]** RNR timeout without count exceed → trigger MEM_RETRY via `m_cmd_port_1`, output `m_sack_info`

3.4.2.6. **\[ASYNC\]** Local ACK or RNR timeout count exceed → notify with MEM_RET/RNR_CNT_EXC via `m_cmd_port_1`, write the QP state as an error via `m_comp_qp_state_rw_port_retry_awd`, (only for local ACK timeout exceed) output `m_nak_info`
