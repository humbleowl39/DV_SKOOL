## Zero-touch RoCE

Zero-touch RoCE is an NVIDIA term that makes RNICs operate lossy network with high-performance, *without manual setting on PFC nor ECN*. It includes several sub-features which are not publicly documented , but in this document I summarize compiled information from several sources.

Zero-touch RoCE is equivalent to “Lossy RoCE acceleration”. When RoCE works on top of lossy network (e.g., Ethernet), it is common to experience packet drops since Ethernet will drop packets in case of buffer overflow unless flow control is configured (e.g., global pause frame or PFC). And due to the nature of RoCE, the RNIC will retransmit packets using Go-back-N algorithm after certain timeout, which results in significant performance degradation.

To accelerate performance in lossy environments, Zero-touch RoCE focuses on minimizing packet drops (**without PFC!**). This includes:

- Developing a congestion handling mechanism which is better adjusted to a lossy environment

- Moving more of the congestion handling mechanism to the hardware and to the dedicated microcode

- Moderating traffic bursts by tuning of transmission window and slow restart of transmission

Zero touch RoCE contains the following facilities, used to enable the above algorithms. [Ref](https://docs.nvidia.com/networking/display/winof2v237/ethernet+network#src-132464292_EthernetNetwork-Facilities)

- **SlowStart:** Start a re-transmission with low bandwidth and gradually increase it

  - Related to artifical CNP, as described here: <a href="https://forums.developer.nvidia.com/t/pfc-cc-doesnt-work-enabling-pfc-disables-dcqcn/292815/4" data-card-appearance="inline">https://forums.developer.nvidia.com/t/pfc-cc-doesnt-work-enabling-pfc-disables-dcqcn/292815/4</a>

  - Official answer: “We monitor the link and if we observe that there were dropped packets we assume it was due to congestion. We will then have an internal mechanism to reduce the rate of the outgoing traffic and by doing so we will start slow to allow the network to recover and increase the rate gradually.”

- **AdpRetrans:** Adjust re-transmission parameters according to network behavior

  - Try to retransmit faster to reduce the time till retransmission, and the internal timer will not exceed user defined timer and will try to track the network RTT

  - <a href="https://forums.developer.nvidia.com/t/how-to-change-ack-timeout-value-in-cx5/309846" data-card-appearance="inline">https://forums.developer.nvidia.com/t/how-to-change-ack-timeout-value-in-cx5/309846</a>

- **TxWindow:**

  <div>

  Automatic tuning of the transmission window size

  </div>

  - Official answer: “Per QP window based control to limit the number of packets in-the-air, thus limit the max number of Go-Back-N retransmissions.”

(Above answers are from: <a href="https://forums.developer.nvidia.com/t/connectx-6-dx-card-sending-cnp-even-when-there-is-no-ecn-marked-roce-traffic-from-switches/270587/7" data-card-appearance="inline">https://forums.developer.nvidia.com/t/connectx-6-dx-card-sending-cnp-even-when-there-is-no-ecn-marked-roce-traffic-from-switches/270587/7</a> )

All features can be enabled/disabled independently, by configuring registers (e.g., running Mlx5Cmd).

Each of them is supported by:

- **ConnectX-4 and above:** Adaptive retransmission and SlowStart

- **ConnectX-5 and above:** tx-window

- **ConnectX-6 Dx and above:**

  - Proprietary selective repeat

    - Selective repeat, Adaptive routing (requires switch support)

  - ZTR-CC

    - Measurement of RTT included (Handling RTT packets are supported on these devices)

    - <a href="https://forums.developer.nvidia.com/t/scaling-zero-touch-roce-technology-with-round-trip-time-congestion-control/198048" data-card-appearance="inline">https://forums.developer.nvidia.com/t/scaling-zero-touch-roce-technology-with-round-trip-time-congestion-control/198048</a>

If above features are enabled, ConnectX behaves differently from IBTA spec, including:

- CNP is generated even there is no ECN-marked packets, in case when receiver NIC observe packet drop. The sender will use this CNP signal to reduce the rate of transmitter.

  - <a href="https://forums.developer.nvidia.com/t/connectx-6-dx-card-sending-cnp-even-when-there-is-no-ecn-marked-roce-traffic-from-switches/270587" data-card-appearance="inline">https://forums.developer.nvidia.com/t/connectx-6-dx-card-sending-cnp-even-when-there-is-no-ecn-marked-roce-traffic-from-switches/270587</a>

- All RDMA write is converted to RDMA write only (due to selective repeat?)

  - <a href="https://forums.developer.nvidia.com/t/why-do-lossy-roce-accelerations-change-rdma-write-first-middle-last-to-rdma-write-only/205952" data-card-appearance="inline">https://forums.developer.nvidia.com/t/why-do-lossy-roce-accelerations-change-rdma-write-first-middle-last-to-rdma-write-only/205952</a>

- Adaptive retransmission does not follow IBTA specification

  - <https://www.cs.jhu.edu/~zhuolong/papers/sigcomm23lumina.pdf>

  - With additional timer, it retransmits much faster than user defined (through retry_cnt and timeout fields in QP attribute)

According to <a href="https://docs.nvidia.com/doca/archive/2-9-2/doca+telemetry+service+guide/index.html" data-card-appearance="inline">https://docs.nvidia.com/doca/archive/2-9-2/doca+telemetry+service+guide/index.html</a> , we can find relevant counters, including

## ZTR-RTTCC

<a href="https://forums.developer.nvidia.com/t/scaling-zero-touch-roce-technology-with-round-trip-time-congestion-control/198048/2" data-card-appearance="inline">https://forums.developer.nvidia.com/t/scaling-zero-touch-roce-technology-with-round-trip-time-congestion-control/198048/2</a>

RTTCC is now a default congestion control algorithm for AI/HPC workload, while maintaining DCQCN good performance for storage workloads. It improves the scalability of ZTR, by actively monitoring RTT to detect network congestion.

<div>

</div>

To measure the network RTT, it uses timing packets - which is basically QP1-MADs (It seems there are sync mechanism through RDMA-CM). They can be handled by ConnectX6-DX+ devices in HW and above. Timing packets are peridoically sent to the receiver, and immediately returned to sender to measure network latency. With ZTR-RTTCC, data center nodes do not wait to be notified of packet loss; instead, they actively identify congestion *prior to* packet loss and react accordingly, notifying initiators to adjust transmission rates.
